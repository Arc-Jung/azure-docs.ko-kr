# <a name="azure-stack-capacity-planning"></a>Azure Stack 용량 계획
Azure Stack 솔루션을 평가할 때 Azure Stack 클라우드에의 한 전체 용량에 직접 영향을 주는 하드웨어 구성을 선택할 수 있습니다. 이 CPU, 메모리 밀도, 저장소 구성 및 전반적인 솔루션 확장 또는 서버 수가 클래식 선택 항목입니다. 기존의 가상화 솔루션과 달리 이러한 구성 요소가 사용 가능한 용량을 확인할 수의 간단한 산술 연산을 적용 되지 않습니다. 첫 번째 이유는 Azure Stack 솔루션 자체 내에서 인프라 또는 관리 구성 요소를 호스트 하도록 설계 되는 경우 복원 력; 지원 하기 위해 예약 되어 솔루션의 용량 중 두 번째 이유는 테 넌 트 워크 로드의 중단을 최소화 하는 방법에 대 한 솔루션의 소프트웨어 업데이트 합니다.

> [!IMPORTANT]
> 제공 된 용량 계획 정보 및 함께 제공 되는 스프레드시트 구성 결정 하 고 Azure Stack에서 계획을 내릴 수 있도록 가이드로만 제공 됩니다. 고유한 조사와 분석에 대 한 대체 되지 않으므로 합니다. 

## <a name="compute-and-storage-capacity-planning"></a>계산 및 저장소 용량 계획
Azure Stack 솔루션 계산, 네트워킹 및 저장소의 하이퍼 수렴 형 클러스터로 빌드됩니다. 이 효과적으로 사용을 허용 하거나 클러스터의 모든 하드웨어 용량 공유 라고 배율 단위는 Azure Stack에 대 한 가용성과 확장성을 제공 하는 방식에서입니다. 모든 인프라 소프트웨어 가상 머신 (Vm)의 집합 내에서 호스팅되고 테 넌 트 Vm으로 동일한 물리적 서버를 공유 합니다. 모든 Vm 크기 조정 단위의 Windows Server 클러스터링 기술과 개별 Hyper-v 인스턴스에 의해 관리 됩니다. 이 방법은 획득 및 Azure Stack 솔루션의 관리를 간소화 하 고 이동 및 모든 서비스 (테 넌 트 및 인프라)의 확장성에 대 한 전체 배율 단위에 제공 합니다.

Azure Stack 솔루션에서 과도 하 게 프로 비전 되지 않은 물리적 리소스 서버 메모리가 없습니다. 다른 리소스, CPU 코어, 네트워킹 대역폭 및 저장소 용량을 효과적으로 사용 가능한 리소스의 사용을 overprovisioned 수 됩니다. 솔루션에 대 한 사용 가능한 용량을 계산할 때 실제 서버 메모리가 주요 참가자입니다. 다른 리소스의 사용률은 과도 프로 비전의 비율은 가능한를 이해 한 다음 되며 의도 한 워크 로드에 따라 허용 됩니다.

Azure Stack 인프라를 호스트 하 고 전체적으로 208 GB에 대 한 124 가상 코어와 메모리 소비를 약 28 Vm 사용 됩니다.  Vm이이 수에 대 한 보안, 확장성, 서비스 및 패치 요구 사항에 맞게 필요한 서비스 분리를 충족 하는 것입니다. 이 내부 서비스 구조를 개발에 대 한 새로운 인프라 서비스의 향후 소개 수 있습니다.

모든 물리적 서버 및 인프라 소프트웨어 구성 요소 또는 패치 및 업데이트, 인프라 및 사용자 VM의 자동화 된 업데이트를 지원 하기 위해 배치 배율 단위의 모든 메모리 리소스를 사용 하지 않습니다. 배율 단위는 모든 서버에서 총 메모리 양을 솔루션의 복원 력 요구 사항을 지원 하기 위해 할당 되지 것입니다. 예를 들어 물리적 서버의 Windows Server 이미지를 업데이트할 때 서버에서 호스트 되는 Vm은 다른 곳으로 이동 배율 단위 내에서 서버의 Windows Server 이미지를 업데이트 하는 동안. 업데이트가 완료 되 면 서버를 다시 시작 하 고 워크 로드를 서비스에 반환 됩니다. 호스트 된 Vm을 중지 하지 않으려면 패치 및 Azure Stack 솔루션의 업데이트에 대 한 목표가입니다. 목표를 달성 하기 위해 노력에서 최소한 하나의 서버 메모리 용량의 배율 단위 내에서 Vm의 움직임을 할당 되지 않은 합니다. 이 배치 및 이동 사용자 또는 Azure Stack 솔루션의 테 넌 트를 대신 하 여 만든 Vm 및 인프라 Vm에 적용 됩니다. 최종 구현 결과가 필요한 VM 이동을 지 원하는 예약 된 메모리 양을 수 있도록 단일 서버의 용량 보다 훨씬 더 많은 배치 문제 때문에 Vm에 다양 한 메모리 요구 사항이 있는 경우. Windows Server 인스턴스 자체의 메모리 사용 범주에 추가 오버 헤드가 됩니다. 각 서버에 대 한 기본 운영 체제 인스턴스는 운영 체제 및 Hyper-v 호스트 된 Vm의 각 관리에서 사용 하는 메모리와 함께 해당 가상 페이지 테이블에 대 한 메모리를 사용 합니다.

이 섹션에서는 용량 계산의 복잡성에 대 한 추가 설명은 나중에 설명 합니다. 이 소개에서 솔루션 크기를 다양 한 사용 가능한 용량을 이해 하는 데 도움이 되는 다음 예제에서는 제공 됩니다. 이러한 예상 되며 결과적으로 테 넌 트 프로덕션 설치에 적용 되지 않은 VM 메모리 사용에 대 한 가정을 합니다. 이 테이블에 대 한 표준 D2 Azure VM 크기가 사용 됩니다. 표준 D2 Vm은 azure 가상 Cpu 2와 7GB 메모리를 사용 하 여 정의 됩니다.

|     |서버 용량|| 배율 단위 용량|  |  |||
|-----|-----|-----|-----|-----|-----|-----|-----|
|     | 메모리 | CPU 코어 수 | 서버 수 | 메모리 | CPU 코어 수 | 테 넌 트 Vm<sup>1</sup>     | 코어 비율<sup>2</sup>    |
|예 1|256GB|28|4|1024 GB| 112 | 54 |4:3|
|예 2|512 GB|28|4|2024 GB|112|144|4:1|
|예 3|384 GB|28|12|4608 GB|336|432|3:1|
|     |     |     |     |     |     |     |     |

> <sup>1</sup> 표준 D2 Vm.
> 
> <sup>2</sup> 가상 코어 대 실제 코어 비율입니다.

위에서 설명한 대로 VM 용량 사용 가능한 메모리에 의해 결정 됩니다. 실제 코어 비율 대 가상 코어를 사용 가능한 CPU 용량 (다른 CPU이 선택 됨) 하는 실제 코어 수가 많으면 솔루션 생성 하지 않는 한 VM 밀도 어떻게 변경 보여 줍니다. 저장소 용량 및 저장소 캐시 용량도 마찬가지입니다.

VM 밀도의 위 예제는 설명 목적 으로만 합니다. 추가 복잡성 계산 지원에는 있습니다. Microsoft는 연락처 또는 솔루션 파트너 용량 계획을 선택 하 고 결과 사용할 수 있는 용량을 자세히 파악 하도록 요구 됩니다.

이 섹션의 나머지 부분에는 계산 및 저장소에 대 한 Azure Stack 배포 요구 사항을 설명합니다. 이 정보를 사용 하 여 기본 구성 요소는 필요를 이해 하 고 최소 구성 값을 얻을 수 있습니다. 자세한 내용은 솔루션 사용 가능한 용량 및 용량 및 성능 기능을 테 넌 트와 관련 하 여 시스템의 잠재적인 제한을 구성 하는 방법을 설명 하기 위해 제공 됩니다.

> [!NOTE]
> 공용 VIP의 크기는 구성할 수 있는 네트워킹에 대 한 용량 계획 요구 사항을 최소화 됩니다. Azure Stack에 더 많은 공용 IP 주소를 추가 하는 방법에 대 한 정보를 참조 하세요. [공용 IP 주소 추가](azure-stack-add-ips.md)합니다.
