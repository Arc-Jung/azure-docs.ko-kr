# <a name="recover-from-catastrophic-data-loss"></a>치명적인 데이터 손실 로부터 복구

*적용 대상: Azure Stack 통합 시스템입니다.*

Azure Stack 데이터 센터에서 Azure 서비스를 실행합니다. Azure Stack으로 단일 랙에 설치 하는 4 개의 노드 환경에서 실행할 수 있습니다. 반면, Azure는 여러 데이터 센터 및 각 지역의 여러 영역에 40 개 이상의 지역에서 실행 됩니다. 사용자 리소스는 여러 서버, 랙, 데이터 센터 및 지역에 확장할 수 있습니다. Azure Stack을 사용 하 여 현재만 선택할을 수 있습니다 단일 랙에 전체 클라우드를 배포 합니다. 이 클라우드 데이터 센터 또는 오류 주요 제품 버그 때문에 심각한 위험에 노출합니다. 재해가 발생 하면 Azure Stack 인스턴스가 오프 라인 상태가 됩니다. 모든 데이터를 잠재적으로 복구할 수 없는 합니다.

데이터 손실의 근본 원인에 따라 단일 인프라 서비스를 복구 하거나 전체 Azure Stack 인스턴스를 복원 해야 합니다. 동일한 위치 또는 다른 위치에서 다른 하드웨어로 복원도 해야 합니다.

이 시나리오에서는 장비 및 사설 클라우드를 재배포 하면 오류가 발생할 경우 전체 설치를 복구 합니다.

| 시나리오                                                           | 데이터 손실                            | 고려 사항                                                             |
|--------------------------------------------------------------------|--------------------------------------|----------------------------------------------------------------------------|
| 재해 또는 제품 버그로 인해 치명적인 데이터 손실 로부터 복구 | 모든 인프라 및 사용자 및 앱 데이터 | 사용자 응용 프로그램 및 데이터 인프라 데이터에서 개별적으로 보호 됩니다. |

## <a name="restore"></a>복원

치명적인 데이터 손실이 발생 하는 하드웨어를 여전히 사용할 수 있지만 경우에 Azure Stack의 재배포가 필요 합니다. 다시 배포 하는 동안 저장소 위치 및 백업에 액세스 하는 데 필요한 자격 증명을 지정할 수 있습니다. 이 모드에서 복원 해야 하는 서비스를 지정할 필요가 없습니다 있습니다. 인프라 백업 컨트롤러 배포 워크플로의 일부로 컨트롤 평면 상태를 삽입합니다.

재해는 하드웨어를 사용할 수 없게 렌더링 하는 경우에 재배포는 새 하드웨어의 수만 있습니다. 재배포 대체 하드웨어를 정렬 및 데이터 센터에 도착 하는 동안 몇 주가 걸릴 수 있습니다. 언제 든 지 컨트롤 평면 데이터의 복원 가능 합니다. 그러나 복원 재배포 인스턴스의 버전이 둘 이상의 버전이 마지막 백업에 사용 된 버전 보다 큰 경우에 지원 되지 않습니다. 

| 배포 모드 | 시작 지점 | 끝점                                                                                                                                                                                                     |
|-----------------|----------------|---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| 새로 설치   | 기준 빌드 | OEM은 Azure Stack을 배포 하 고 지원 되는 최신 버전으로 업데이트 합니다.                                                                                                                                          |
| 복구 모드   | 기준 빌드 | OEM 복구 모드에서 Azure Stack을 배포 하 고 사용 가능한 최신 백업을 기반으로 하는 요구 사항 일치 하는 버전을 처리 합니다. OEM 지원 되는 최신 버전으로 업데이트 하 여 배포를 완료 합니다. |

## <a name="data-in-backups"></a>데이터 백업

Azure Stack 클라우드 복구 모드를 호출 하는 배포 유형을 지원 합니다. 이 모드는 Azure Stack 재해 후 복구를 선택 하거나 제품 버그 렌더링 솔루션을 복구할 수 없는 경우에 사용 됩니다. 이 배포 모드 솔루션에 저장 된 사용자 데이터의 복구 되지 않습니다. 이 배포 모드의 범위는 다음 데이터를 복원 하려면 제한 됩니다.

 - 배포 입력
 - 내부 id 시스템
 - 페더레이션 구성 (연결이 끊어진된 배포)를 식별 합니다.
 - 내부 인증 기관에서 사용 되는 루트 인증서
 - 구독, 계획, 제품 및 저장소에 대 한 할당량와 같은 azure 리소스 관리자 구성 사용자 데이터를 네트워크 및 리소스를 계산 합니다.
 - KeyVault 비밀 및 자격 증명 모음
 - RBAC 정책 할당 및 역할 할당 

없음 (PaaS) 서비스 리소스와 사용자 인프라 서비스 (IaaS) 또는 플랫폼으로 배포 하는 동안 복구 됩니다. 즉 IaaS Vm, 저장소 계정, blob, 테이블, 네트워크 구성 및 등등으로 손실 됩니다. 클라우드 복구의 목적에 연산자를 확인 하는 것 및 배포가 완료 된 후 사용자 포털에 다시 로그인 할 수 있습니다. 다시 로그인 하는 사용자가 해당 리소스의 표시 되지 않습니다. 사용자 복원 해당 구독이 있고는 함께 원래 계획 하 고 관리자가 정의한 정책을 제공 합니다. 시스템에 다시 로그인 하는 사용자는 재해가 발생 하기 전에 원래 솔루션에서 적용 된 동일한 제약 조건에 따라 작동 합니다. 클라우드 복구 완료 되 면 연산자 수동으로 복원할 수 있습니다는 부가 가치 타사 RPs 및 연결 된 데이터입니다.
