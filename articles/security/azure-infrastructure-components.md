# <a name="azure-information-system-components-and-boundaries"></a>Azure 정보 시스템 구성 요소 및 경계
이 문서에서는 Microsoft Azure 아키텍처 및 관리에 대한 일반적인 설명을 제공합니다. Azure 시스템 환경이 구성되는 네트워크는 다음과 같습니다.

- Azure 네트워크(Microsoft Azure 프로덕션 네트워크)
- Corpnet 네트워크(Microsoft 회사 네트워크)

별도의 IT 팀에서 Azure 네트워크 및 CorpNet 네트워크의 운영 및 유지 관리를 담당합니다.

## <a name="azure-architecture"></a>Azure 아키텍처
Microsoft Azure는 Microsoft에서 관리하는 데이터 센터의 네트워크를 통해 응용 프로그램과 서비스를 빌드, 배포 및 관리하기 위한 클라우드 컴퓨팅 플랫폼 및 인프라입니다. Azure는 고객이 지정한 리소스의 수와 리소스 요구 사항에 따라 VM을 만듭니다. 이러한 VM은 클라우드에서 사용하도록 설계되고 일반인이 액세스할 수 없는 Microsoft Azure 하이퍼바이저에서 실행됩니다.

Azure 물리적 서버 노드 각각에는 하드웨어에서 직접 실행되는 하이퍼바이저가 있습니다. 하이퍼바이저는 노드를 가변 개수의 게스트 VM(가상 머신)으로 나눕니다. 또한 각 노드에는 호스트 OS를 실행하는 특별한 하나의 "루트" VM도 있습니다. 각 VM에서 Windows 방화벽을 사용하도록 설정할 수 있습니다. 내부적으로 또는 외부적으로 열려 있고 주소를 지정할 수 있는 유일한 포트는 고객이 구성한 서비스 정의 파일에 명시적으로 정의된 포트입니다. 모든 트래픽과 디스크 및 네트워크에 대한 액세스는 하이퍼바이저 및 루트 OS에 의해 조정됩니다.

호스트 계층에서 Azure VM은 최신 Windows Server의 사용자 지정되고 강화된 버전을 실행합니다. Microsoft Azure는 VM을 호스팅하는 데 필요한 구성 요소만 포함하는 간소화된 버전의 Windows Server를 사용합니다. 이는 성능을 향상시키고 공격 표면을 줄이기 위해 수행됩니다. 컴퓨터 경계는 운영 체제 보안과 관련이 없는 하이퍼바이저에 의해 적용됩니다.

**FC(패브릭 컨트롤러)에 의한 Azure 관리**: Azure에서 물리적 서버(블레이드/노드)에서 실행되는 VM은 약 1,000개의 "클러스터"로 그룹화됩니다. VM은 FC라고 하는 확장되고 중복적인 플랫폼 소프트웨어 구성 요소에서 독립적으로 관리됩니다.

각 FC는 해당 클러스터에서 실행되는 응용 프로그램의 수명 주기를 관리하고 해당 제어 하에 있는 하드웨어의 상태를 프로비전하고 모니터링합니다. 서버가 실패했다고 판단되면 정상적인 서버에서 VM 인스턴스를 다시 환생시키는 것처럼 자율적인 작업을 모두 실행합니다. FC는 응용 프로그램 배포, 업데이트 및 확장과 같은 응용 프로그램 관리 작업도 수행합니다.

데이터 센터는 클러스터로 나뉩니다. 클러스터는 FC 수준에서 오류를 격리하고, 특정 유형의 오류가 발생하는 클러스터 이외의 서버에 영향을 미치지 않도록 합니다. 특정 Azure 클러스터에 서비스를 제공하는 FC는 FC 클러스터로 그룹화됩니다.

**하드웨어 인벤토리**: Azure 하드웨어 및 네트워크 장치의 인벤토리는 부트스트랩 구성 프로세스 중에 준비되고 datacenter.xml 구성 파일에 문서화됩니다. Azure 프로덕션 환경에 들어가는 새 하드웨어 및 네트워크 구성 요소는 부트스트랩 구성 프로세스를 따라야 합니다. FC는 datacenter.xml 구성 파일에 나열된 전체 인벤토리를 관리합니다.

**FC 관리 OS**: OS 팀은 Azure 프로덕션 환경의 모든 호스트 및 게스트 VM에 배포되는 VHD(가상 하드 디스크) 형태의 OS 이미지를 제공합니다. OS 팀은 자동화된 오프라인 빌드 프로세스를 통해 이러한 "기본 이미지"를 구성합니다. 기본 이미지는 커널 및 기타 핵심 구성 요소가 수정되고 Azure 환경을 지원하도록 최적화된 운영 체제 버전입니다.

패브릭 관리 OS 이미지에는 다음 세 가지 유형이 있습니다.

- 호스트 OS - 호스트 VM에서 실행되는 사용자 지정 운영 체제
- 원시 OS - 하이퍼바이저가 없는 테넌트(예: Azure Storage)에서 실행되는 원시 운영 체제
- 게스트 OS - 게스트 VM에서 실행되는 게스트 OS

호스트 및 원시 FC 관리 운영 체제는 클라우드에서 사용하도록 설계되었으며 공개적으로 액세스할 수 없습니다.

**호스트 및 원시 OS**: 호스트 OS 및 원시 OS는 FA(패브릭 에이전트)를 호스팅하고 계산 노드(노드에서 첫 번째 VM로 실행) 및 저장소 노드에서 실행되는 강화된 OS 이미지입니다. 호스트 및 원시 OS의 최적화된 기본 이미지를 사용하는 이점은 보안 위험이 높고 OS의 공간을 늘리는 API 또는 사용되지 않는 구성 요소에서 공개되는 노출 영역이 줄어든다는 것입니다. 이처럼 사용 공간이 적은 운영 체제에는 Azure에 필요한 구성 요소만 포함됩니다. 이렇게 하면 성능이 향상되고 공격 영역이 줄어듭니다.

**게스트 OS**: 게스트 OS VM에서 실행되는 Azure 내부 구성 요소에는 외부 고객과 달리 RDP(원격 데스크톱 프로토콜)를 실행할 기회가 없습니다. 변경 및 릴리스 관리 프로세스를 진행하려면 기본 구성 설정을 변경해야 합니다.

## <a name="azure-datacenters"></a>Azure 데이터 센터
MCIO(Microsoft Cloud Infrastructure and Operations) 팀은 모든 Microsoft 온라인 서비스에 대한 Microsoft의 물리적 인프라 및 데이터 센터 기능을 관리합니다. MCIO는 주로 데이터 센터 내의 물리적 및 환경적 제어를 관리하고, 외부 경계 네트워크 장치(에지 라우터 및 데이터 센터 라우터)도 관리하고 지원합니다. 또한 MCIO는 데이터 센터의 랙에 최소한의 서버 하드웨어를 설정해야 합니다. 고객은 Azure와 직접 상호 작용하지 않습니다.

## <a name="service-management--service-teams"></a>서비스 관리 및 서비스 팀
Azure 서비스의 지원은 '서비스 팀'이라고 하는 여러 엔지니어링 그룹에서 관리합니다. 각 서비스 팀은 Azure에 대한 지원 영역을 담당합니다. 각 서비스 팀에서 서비스의 오류를 조사하고 해결하기 위해 연중 무휴로 엔지니어를 사용할 수 있도록 해야 합니다. 서비스 팀은 기본적으로 Azure에서 작동하는 하드웨어에 물리적으로 액세스하지 않습니다.

서비스 팀에서 담당하는 영역은 다음과 같습니다.

- 응용 프로그램 플랫폼
- Azure Active Directory
- Azure Compute
- Azure 네트워크
- 클라우드 엔지니어링 서비스
- ISSD: 보안
- 다단계 인증
- SQL Database
- Storage

## <a name="types-of-users"></a>사용자 유형

| 역할 | 내부 또는 외부 | 민감도 수준 | 허가된 권한 및 수행 기능 | 액세스 형식

|Microsoft Azure 고객 | 외부 | 해당 없음 | 해당 없음 | 해당 없음 |

Azure는 고유 식별자를 사용하여 Azure 환경에 속한 모든 자산/장치에 대해 조직 사용자 및 고객(또는 조직 사용자를 대신하여 작동하는 프로세스)을 인증합니다.

**Microsoft Azure 내부 인증**: Azure 내부 구성 요소 간의 통신은 TLS 암호화로 보호됩니다. 대부분의 경우 X.509 인증서는 자체 서명됩니다. Azure 네트워크 외부에서 액세스할 수 있는 연결이 있는 인증서와 FC는 예외입니다. FC에는 신뢰할 수 있는 루트 CA에서 지원하는 Microsoft CA(인증 기관)에서 발급한 인증서가 있습니다. 이렇게 하면 FC 공개 키를 쉽게 롤오버할 수 있습니다. 또한 FC 공개 키는 Microsoft 개발자 도구에서 사용하므로 개발자가 새 응용 프로그램 이미지를 제출할 때 포함된 비밀을 보호하기 위해 FC 공개 키로 암호화됩니다.

**Microsoft Azure 하드웨어 장치 인증**: FC는 제어 하에 있는 다양한 하드웨어 장치에 인증하는 데 사용되는 자격 증명(키 및/또는 암호) 집합을 유지 관리합니다. 이러한 자격 증명을 전송, 유지 및 사용에 사용되는 시스템은 Azure 개발자, 관리자 및 백업 서비스/담당자가 중요, 기밀 또는 개인 정보에 액세스하지 못하도록 설계되었습니다.

FC의 마스터 ID 공개 키에 기반한 암호화는 FC 설정 및 FC 재구성 시간에 사용되어 네트워킹 하드웨어 장치에 액세스하는 데 사용되는 자격 증명을 전송합니다. 자격 증명은 필요할 때 FC에서 검색하고 암호 해독합니다.

**네트워크 장치**: Azure 네트워킹 팀에서 네트워크 서비스 계정을 구성하여 Microsoft Azure 클라이언트에서 네트워크 장치(라우터, 스위치 및 부하 분산 장치)에 인증할 수 있도록 합니다.

## <a name="secure-service-administration"></a>보안 서비스 관리
Microsoft Azure 운영 담당자는 SAW(보안 관리 워크스테이션 - PAW(권한 있는 액세스 워크스테이션)를 사용하여 비슷한 제어를 구현할 수 있음)를 사용해야 합니다. SAW 접근 방법은 관리 담당자를 위해 별도 관리자 및 사용자 계정을 사용하는, 권장되는 확장 구성 방법입니다. 이 방법에서는 사용자의 표준 사용자 계정과 구분되는 개별적으로 할당된 관리 계정을 사용합니다. SAW는 중요한 계정에 신뢰할 수 있는 워크스테이션을 제공함으로써 해당 계정 분리 방법을 토대로 구축되었습니다.
