# <a name="scenario-custom-isolation-for-virtual-networks-and-branches"></a>시나리오: 가상 네트워크 및 분기에 대 한 사용자 지정 격리

가상 WAN 가상 허브 라우팅으로 작업할 때 몇 가지 시나리오를 사용할 수 있습니다. Vnet (가상 네트워크) 및 분기에 대 한 사용자 지정 격리 시나리오에서 목표는 특정 Vnet 집합이 다른 Vnet 집합에 도달 하는 것을 방지 하는 것입니다. 마찬가지로 분기 (VPN/ER/User VPN)는 특정 Vnet 집합에만 도달할 수 있습니다.

또한 Azure 방화벽에서 vnet에 대해 분기를 검사 하 고 vnet 트래픽에 대해 분기를 검사 해야 하는 추가 요구 사항을 소개 하지만 vnet 간 트래픽은 제공 **하지 않습니다**  .  

가상 허브 라우팅에 대 한 자세한 내용은 [가상 허브 라우팅 정보](about-virtual-hub-routing.md)를 참조 하세요.

## <a name="design"></a><a name="design"></a>디자인

필요한 경로 테이블 수를 파악 하기 위해 연결 행렬을 작성할 수 있습니다. 이 시나리오에서 각 셀은 원본 (행)이 대상 (열)과 통신할 수 있는지 여부를 나타내는 다음과 같습니다.

| From | 아래와 같이 변경합니다.| *파란색 Vnet* | *레드 Vnet* | *빨간색 분기*| *파란색 분기*| 
|---|---|---|---|---|---|
| **파란색 Vnet** |   &#8594;|   직접     |           |   |  AzFW|
| **레드 Vnet**  |   &#8594;|              |   직접  |  AzFW  | 
| **빨간색 분기**   |   &#8594;|   |   AzFW  |  직접 | 직접
| **파란색 분기**| &#8594;| AzFW  |   |직접   | 직접

위의 표에 나와 있는 각 셀은 가상 WAN 연결 (흐름의 "From" 쪽에서 행 머리글)이 대상과 통신 하는지 여부를 설명 합니다 (흐름의 "To" 쪽에서 열 머리글은 기울임꼴로). **Direct** 는 가상 WAN을 통해 직접 트래픽 흐름을 암시 하는 반면, **AzFW** 는 대상에 전달 되기 전에 Azure 방화벽에서 트래픽을 검사 하는 것을 의미 합니다. 빈 항목은 설정에서 흐름이 차단 됨을 의미 합니다.

이 경우 Vnet에 대 한 두 경로 테이블은 경로에 Azure 방화벽이 없는 VNet 격리의 목표를 달성 하는 데 필요 합니다. 이러한 경로 테이블 **RT_BLUE** 및 **RT_RED** 를 호출 합니다.

또한 분기가 항상  **기본** 경로 테이블에 연결 되어 있어야 합니다. Azure 방화벽에서 분기로 들어오고 나가는 트래픽을 검사 하기 위해 **기본** **RT_RED** 에 고정 경로를 추가 하 고, azure 방화벽으로 트래픽을 가리키는 경로 테이블을 **RT_BLUE** 하 고, 원하는 트래픽을 허용 하는 네트워크 규칙을 설정 합니다. 또한 분기가 **RT_BLUE** 및 **RT_RED** 에 전파 **되지** 않도록 합니다.

결과적으로 최종 설계는 다음과 같습니다.

* 파란색 가상 네트워크:
  * 연결 된 경로 테이블: **RT_BLUE**
  * 경로 테이블에 전파: **RT_BLUE**
* 빨간색 가상 네트워크:
  * 연결 된 경로 테이블: **RT_RED**
  * 경로 테이블에 전파: **RT_RED** 
* 분기
  * 연결 된 경로 테이블: **기본값**
  * 경로 테이블에 전파: **기본값**
* 정적 경로:
    * **기본 경로 테이블**: 다음 홉 Azure 방화벽을 사용 하 여 주소 공간 Virtual Network
    * **RT_RED**: 다음 홉 Azure 방화벽을 사용 하는 0.0.0.0/0
    * **RT_BLUE**: 다음 홉 Azure 방화벽을 사용 하는 0.0.0.0/0
* 방화벽 네트워크 규칙:
    * **규칙** **원본 접두사** 허용: 파란색 분기 주소 접두사 **대상 접두사**: 파란색 VNet 접두사 
    * **규칙**  **원본 접두사** 허용: 빨간색 분기 주소 접두사 **대상 접두사**: 빨간색 Vnet 접두사

> [!NOTE]
> 모든 분기가 기본 경로 테이블에 연결 되어 있어야 하 고 동일한 라우팅 테이블 집합으로 전파 되어야 하므로 모든 분기에 동일한 연결 프로필이 있습니다. 즉, Vnet에 대 한 빨강/파랑 개념은 분기에 적용 될 수 없습니다. 그러나 분기에 대 한 사용자 지정 라우팅을 구현 하기 위해 분기에서 Azure 방화벽으로 트래픽을 전달할 수 있습니다.

> [!NOTE]
> Azure 방화벽은 기본적으로 0 신뢰 모델의 트래픽을 거부 합니다. 검사 된 패킷과 일치 하는 명시적 **허용** 규칙이 없는 경우 Azure 방화벽은 패킷을 삭제 합니다.

가상 허브 라우팅에 대 한 자세한 내용은 [가상 허브 라우팅 정보](about-virtual-hub-routing.md)를 참조 하세요.



## <a name="workflow"></a><a name="architecture"></a>워크플로

**그림 1** 에는 Blue와 red Vnet 및 Blue 또는 red vnet에 액세스할 수 있는 분기가 있습니다.

* 파란색으로 연결 된 Vnet는 서로 연결 될 수 있으며 모든 파란색 분기 (VPN/ER/P2S) 연결에 연결할 수 있습니다. 다이어그램에서 파란색 분기는 사이트 간 VPN 사이트입니다.
* 빨간색으로 연결 된 Vnet 서로 연결 될 수 있으며 모든 빨간색 분기 (VPN/ER/P2S) 연결에 도달할 수 있습니다. 다이어그램에서 빨간색 분기는 지점 및 사이트 간 VPN 사용자입니다.

라우팅을 설정 하는 경우 다음 단계를 고려 하십시오.

1. 이러한 Vnet에 대 한 트래픽을 사용자 지정 하기 위해 Azure Portal **RT_BLUE** 및 **RT_RED** 에서 두 개의 사용자 지정 경로 테이블을 만듭니다.
2. 경로 테이블 **RT_BLUE** 의 경우 파란색 vnet 다른 모든 BLUE vnet의 주소 접두사를 알 수 있도록 다음 설정을 적용 합니다.
   * **연결**: 모든 Blue vnet를 선택 합니다.
   * **전파**: 모든 Blue vnet를 선택 합니다.
3. Red Vnet에 대 한 **RT_RED** 경로 테이블에 대해 동일한 단계를 반복 합니다.
4. 가상 WAN에서 Azure 방화벽을 프로 비전 합니다. 가상 WAN 허브의 Azure 방화벽에 대 한 자세한 내용은 [가상 wan 허브에서 Azure 방화벽 구성](howto-firewall.md)을 참조 하세요.
5. Vnet 주소 공간 (파란색 및 빨강 모두)으로 향하는 모든 트래픽을 Azure 방화벽으로 전달 하는 가상 허브의 **기본** 경로 테이블에 고정 경로를 추가 합니다. 이 단계를 통해 분기의 모든 패킷이 검사를 위해 Azure 방화벽으로 전송 됩니다.
    * 예: **대상 접두사**: 10.0.0.0/24 **다음 홉**: Azure 방화벽
    >[!NOTE]
    > "개인 트래픽 보안" 옵션을 선택 하 여 방화벽 관리자를 통해이 단계를 수행할 수도 있습니다. 그러면 Vnet 및 분기에 적용 가능한 모든 RFC1918 개인 IP 주소에 대 한 경로가 추가 됩니다. RFC1918와 호환 되지 않는 분기 또는 가상 네트워크에서 수동으로 추가 해야 합니다. 

6. **RT_RED** 에 고정 경로를 추가 하 고 모든 트래픽을 Azure 방화벽으로 전달 **RT_BLUE** 합니다. 이 단계를 수행 하면 Vnet에서 분기에 직접 액세스할 수 없습니다. 이 단계는 방화벽 관리자를 통해 수행할 수 없습니다. 이러한 가상 네트워크는 기본 경로 테이블에 연결 되어 있지 않기 때문입니다.
    * 예: **대상 접두사**: 0.0.0.0/0 **다음 홉**: Azure 방화벽

    > [!NOTE]
    > 라우팅은 LPM (가장 긴 접두사 일치)을 사용 하 여 수행 됩니다. 따라서 0.0.0.0/0 고정 경로는 **BLUE_RT** 및 **RED_RT** 에 존재 하는 정확한 접두사 보다는 선호 **되지** 않습니다. 결과적으로, Azure 방화벽에서 Vnet 내 트래픽을 검사 하지 않습니다.
