# Azure 검색의 인덱스
> [AZURE.SELECTOR]
- [Overview](search-what-is-an-index.md)
- [Portal](search-create-index-portal.md)
- [.NET](search-create-index-dotnet.md)
- [REST](search-create-index-rest-api.md)

Azure 검색은 호스팅된 클라우드 검색 서비스로, 검색 엔진, 사용자 지정 응용 프로그램에서 Bing 또는 Google과 유사한 환경을 할 수 있는 검색 기능, Azure 검색을 웹 또는 데스크톱 응용 프로그램과 통합하기 위한 .NET 및 REST API, 그리고 *인덱스* 형태로 검색 가능한 데이터의 데이터 저장소를 제공합니다.

##인덱스란?

*인덱스*는 Azure 검색 서비스에서 사용되는 문서 및 기타 구조의 지속형 저장소입니다.

보다 구체적으로 설명하면, 여기에는 인덱스를 처리하거나 쿼리를 실행하거나 검색 결과 목록을 반환에 사용되는 검색 가능한 데이터, 패싯된 탐색 구조 또는 응용 프로그램에 대한 세부 정보 페이지가 있습니다. Azure 검색에서 인덱스는 필터 식 또는 점수 매기기 프로필(쿼리를 구체화하거나 검색 랭킹 점수를 높이는 데 사용)뿐 아니라 Azure 검색과 통합된 사용자 지정 응용 프로그램까지 범위가 지정된 구조를 포함합니다.

데이터베이스 개념에 익숙한 경우, 인덱스는 테이블과 개념적으로 유사하며 문서는 테이블의 행과 거의 비슷합니다.

이 문서에서 보다 자세히 설명된 인덱스의 주요 섹션은 다음과 같습니다.- 특정 동작을 야기하는 *문서* 본문을 정의하는 필드 - 점수 매기기 프로필 - 제안 - CORS 옵션 - 기본 분석기

응용 프로그램에서 사용하기 위한 검색 서비스를 프로비저닝의 일부로 인덱스를 생성해야 합니다. 인덱스를 만드는 것은 스키마 정의에 집중된 작업입니다. 최소한 필드를 정의하고 특성을 설정하는 것으로 구성됩니다. 필요에 따라 검색 환경을 향상시키기 위해 점수 매기기 프로필과 제안 사항을 정의하할 수 있습니다.

스키마를 만들려면 포털을 사용하거나 .NET SDK 또는 REST API로 호출하는 코드를 작성할 수 있습니다.

스키마가 정의되고 인덱스가 만들어지면, 이를 채우는 작업은 별도의 작업입니다. 인덱스를 만든 히후 데이터 수집에 대한 자세한 내용은 [Azure 검색으로 데이터 가져오기](search-what-is-data-import.md)를 참조하세요.

##Azure 검색에서 인덱스를 사용하는 방법

Azure 검색에 의해 수행된 모든 데이터 관련 작업은 인덱스의 데이터를 사용하거나 반환합니다. 예외는 없습니다. 응답에 외부 데이터를 반환할 인덱스 외에도 다른 데이터 원본에 서비스를 지정할 수 없습니다.

OLTP 데이터베이스에 데이터를 저장하는 온라인 소매 앱의 판매 트랜잭션과 같은 데이터에 대해 축적하고 작동된 응용 프로그램의 경우, Azure 검색 인덱스는 전체 솔루션에 추가 데이터 원본이 됩니다. 인덱스는 검색 서비스에 의해서만 사용되는 전용 데이터 저장소입니다. 리소스 또는 데이터에 대한 경쟁이 없기 때문에, 서비스에 근접한 전용 데이터 원본을 가지면 검색 결과의 일관성을 보장하고, 변동성을 줄이고, 응용 프로그램 및 Azure 검색 사이의 왕복 횟수를 줄기고 검색 작업의 전체 성능을 높입니다.

##스키마를 정의하기 위한 지침

스키마는 JSON 구조로 생성됩니다. Azure 검색과 통합하는 각 사용자 지정 솔루션에 대해 하나의 인덱스를 만들어야 합니다. 다중 인덱스를 연결하거나 조인시킬 수는 없지만 솔루션에 필요한 다양한 검색 시나리오를 수용하는 복잡한 인덱스를 만들 수 있습니다. 예를 들어, 응용 프로그램이 여러 언어를 지원하는 경우 인텍스의 필드 컬렉션에는 각 필드에 대한 언어별 변형이 있을 수 있습니다.

인덱스를 설계할 때 계획 단계에서 각 의사 결정에 대해 신중하게 생각합니다. 배포된 후 인덱스를 변경하려면 데이터를 다시 작성하고 다시 로드해야 합니다. 코드에서 인덱스를 만드는 경우 이 단계는 포털에서 수동으로 만든 경우보다 훨씬 더 쉽습니다.

우수한 스키마를 만들려면 원래의 원본 데이터를 확실하게 이해해야 합니다. 데이터 형식을 일치시키려면, 어떤 필드를 인덱스의 각 문서를 고유하게 식별하는 *키*로 사용할지와 어떤 필드를 검색 결과 목록 또는 세부 정보 페이지에 노출시켜야 할지 알아야 합니다.

이미지, 오디오 또는 비디오 콘텐츠는 인덱스가 리소스에 대한 URL과 함께 대표 필드를 포함하는 위치에 저장해야 합니다. 특히 이진 콘텐츠를 검색할 수 없는 모든 콘텐츠는 다른 위치에 저장한 다음 인덱스의 필드에 URI로 참조시켜야 합니다.

예상할 수 있듯이 인덱스는 솔루션에서 사용되는 다른 데이터 소스와 정기적으로 동기화되어야 합니다. 온라인 소매 카탈로그에 있는 판매 트랜잭션을 캡처하는 재고 데이터베이스에는 검색 결과 통해 나타난 데치터와 SKU, 가격, 가용성이 동일해야 합니다. 솔루션에 얼마나 많은 대기 시간을 적용할 수 있는지에 따라, 데이터 동기화가 일주일에 한 번, 하루에 한 번 또는 동시 작성을 사용하여 거의 실시간으로 재고 데이터베이스와 Azure 검색 인덱스를 찾을 수 있습니다. [Azure 검색으로 데이터 가져오기](search-what-is-data-import.md)에 사용할 수 있는 옵션에 대해 자세히 설명되어 있습니다.

##스키마 요소

인덱스는 특성 필드, 데이터 값, 그리고 쿼리 및 검색 결과가 생성되는 방식을 생성하는 기타 구문으로 구성됩니다.

- 필드는 문서의 본문을 정의합니다. 온라인 소매점은 각 SKU에 대한 문서를 필요로 합니다. 그리고 언론사는 각 기사에 대한 문서를 필요로 하며, 정보 미디어 포털은 각 공급업체 또는 상점에 대한 문서를 필요로 합니다.
- 특성(또는 속성)은 해당 필드에서 허용된 데이터 유형, 길이, 값 및 동작을 지정하는 주석입니다. 모든 필드에 대해 필드가 색 가능하거나 조회 가능하거나 정렬 가능한지 여부를 지정할 수 있습니다. 또한 필드 수준에서 언어 분석기를 사용할지 지정할 수 있습니다. *키*는 필드 컬렉션에 있는 모든 문서에 대한 고유 식별자를 제공하는 스키마의 예약된 필드입니다.
- 프로필 점수 매기기는 프로필에 의해 설정되는 특성을 가진 검색 순위를 높이는 데 사용되는 기준입니다. 예를 들어, 검색 단어가 제품 이름 및 제품 설명과 일치한다고 가정하면 제품 이름 순위를 설명에 나온 순위보다 높은 순위에 일치시킬 수 있습니다.
- 자동 완성 또는 미리 입력 쿼리라고도 알려진 제안은 인덱스의 섹션으로 정의됩니다.
- 기본 언어 분석기는 모든 필드에 전역적으로 적용 가능한 인덱스 수준에서 지정할 수 있습니다.
- CORS 옵션은 도메인 간 스크립팅을 활성화하고 인덱스 정의의 일부가 될 수도 있습니다.

## 인덱스 특성

|**속성**|설명|
|------------|-----------|
|**조회 가능**|검색 결과에서 필드를 반환할 수 있는지 여부를 지정합니다.|
|**필터링 가능**|필드를 **$filter** 쿼리에 사용할 수 있습니다.|
|**정렬 가능**|필드를 정렬 옵션으로 사용할 수 있습니다.|
|**패싯 가능**|자기 주도형 필터링을 위해 필드를 패싯 탐색 구조에 사용할 수 있습니다. 일반적으로 여러 문서(예를 들어 단일 제품 또는 서비스 범주에 속하는 여러 문서)를 그룹화하는 데 사용할 수 있는 반복되는 값을 포함한 필드는 패싯으로 가장 적합합니다.|
|**키**|문서 조회에 사용하는 각 문서의 고유 ID입니다. 모든 인덱스에는 하나의 키가 있어야 합니다. 필드 한 개만 키가 될 수 있으며, 이 필드를 Edm.String으로 설정해야 합니다.|
|**검색 가능**|전체 텍스트 검색 가능 필드로 표시합니다.|

## 클라우드 저장소

Azure 검색 하는 완벽하게 관리되는 검색 서비스이기 때문에 데이터 저장소는 내부 작업으로 처리됩니다. 프로그래머는 인덱스에 직접 연결하기 위해 Azure 검색 API를 무시할 수 없음을 이해하는 것이 중요합니다. 그렇지 않으면 포털 알림이나 API 호출이 아니면 인덱스 크기 또는 상태를 모니터링할 수 없습니다. 인덱스를 백업 또는 복원하거나 저장소 계층에서의 최적화 또는 성능 튜닝을 위한 기능이 없습니다. 숨은 기능으로 데이터는 blob 저장소로 저장되지만 물리적 저장소와 인덱스 관리에 대해 생각하는 최적의 방법은 나중에 변경할 수 있는 구현 세부 정보입니다. 서비스의 주요 이점 중 하나는 데이터 관리에 대한 자동 접근 방법입니다.

쿼리 볼륨 또는 데이터 저장소 요구 사항이 시간이 지남에 따라 변하는 경우 복제본 및 파티션을 추가하거나 이동하여 용량을 늘리거나 줄일 수 있습니다. 자세한 내용은 [Azure에서 검색 서비스 관리](search-manage.md) 또는 [서비스 제한](search-limits-quotas-capacity.md)을참조하세요.
