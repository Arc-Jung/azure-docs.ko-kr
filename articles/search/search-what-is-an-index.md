# Azure 검색의 인덱스
> [AZURE.SELECTOR]
- [Overview](search-what-is-an-index.md)
- [Portal](search-create-index-portal.md)
- [.NET](search-create-index-dotnet.md)
- [REST](search-create-index-rest-api.md)

Azure 검색은 호스팅된 클라우드 검색 서비스로, 검색 엔진, 사용자 지정 응용 프로그램에서 Bing 또는 Google과 유사한 환경을 할 수 있는 검색 기능, Azure 검색을 웹 또는 데스크톱 응용 프로그램과 통합하기 위한 .NET 및 REST API, 그리고 *인덱스* 형태로 검색과 관련된 데이터의 데이터 저장소를 제공합니다.

##인덱스란?

*인덱스*는 Azure 검색 서비스에서 사용되는 *문서* 및 기타 구조의 지속형 저장소입니다. 문서는 검색 가능한 데이터의 기본 단위입니다. 예를 들어 소매점에는 각 SKU에 대한 문서가 있을 수 있고 뉴스 조직에는 각 문서에 대한 문서가 있을 수 있으며 스트리밍 미디어 조직에는 해당 라이브러리에 각 비디오 또는 음악에 대한 문서가 있을 수 있습니다. 이러한 개념을 익숙한 데이터베이스과 동일하게 매핑: *인덱스*는 *테이블*과 개념적으로 유사하며 *문서*는 테이블의 *행*과 거의 비슷합니다.

인덱스는 대개 관계형 또는 NoSQL 데이터베이스에 저장되는 플랫 데이터 집합입니다. 플랫 데이터 집합은 판매 트랜잭션, 콘텐츠 게시, 구매 작업 등과 같이 전산화 프로세스를 통해 일반 비즈니스 작업 중에 생성되거나 캡처되는 데이터의 일반적인 하위 집합입니다. 다른 데이터 원본에서 인덱스에 행을 포함하는 데이터 집합을 푸시하거나 끌어올 때 인덱스는 문서를 가져옵니다.

##언제 인덱스 만듭니까?

응용 프로그램에서 사용하기 위한 검색 서비스를 프로비저닝의 일부로 인덱스를 생성해야 합니다. 검색 작업에 인덱스를 사용할 수 있도록 하는 것은 2가지 부분으로 구성됩니다. 첫째로, 스키마를 정의하여 Azure 검색에 게시합니다. 둘째로 스키마를 충족하는 사용자 정의 데이터 집합으로 인덱스를 채웁니다.

인덱스를 정의하는 스키마를 만들려면 포털을 사용하거나 .NET SDK 또는 REST API로 호출하는 코드를 작성합니다(이 문서 상단의 탭 참조). 인덱스를 만든 후 데이터 수집에 대한 자세한 내용은 [Azure 검색으로 데이터 가져오기](search-what-is-data-import.md)를 참조하세요.

##인덱스의 JSON 스키마 사양

인덱스는 필드 및 특성, 점수 매기기 프로필(scoring profiles), 확인기(suggesters), 기본 점수 매기기 프로필(default scoring profile), CORS 옵션(CORS options)을 포함하는 JSON 문서입니다.

 ![][1]

JSON 데이터 교환 형식으로 명시된 것처럼 Azure 검색 인덱스의 주요 섹션은 다음과 같습니다.

|섹션|설명|
|--------------|-----------|
|필드 컬렉션|필드는 문서를 정의합니다. 인덱스에 푸시 또는 끌어온 데이터 집합은 각 필드에 값 또는 null을 제공하며 이는 스키마에 표현된 데이터 형식 및 필드 길이와 호환할 수 있습니다. 필드에는 [특성](#index-attributes)이 있습니다. 이것은 필드에 대해 검색 관련 동작이 가능하도록 필드를 표시하는 데 사용되는 속성 또는 주석입니다. 예를 들어, 필드에 대한 검색 가능 여부, 조회 가능 여부, 정렬 가능 여부를 모두 필드별로 지정할 수 있습니다. 또한 필드 수준에서 재정의하는 언어 분석기를 지정할 수 있습니다.
|[확인기](https://msdn.microsoft.com/library/azure/mt131377.aspx)|자동 완성 또는 미리 입력 쿼리라고도 알려진 인덱스의 섹션으로 정의됩니다.|
|[점수 매기기 프로필](https://msdn.microsoft.com/library/azure/dn798928.aspx)|프로필에 의해 설정되는 특성을 가진 검색 순위를 높이는 데 사용되는 기준입니다. 예를 들어, 검색 단어가 제품 이름 및 제품 설명과 일치한다고 가정하면 제품 이름 순위를 설명에 나온 순위보다 높은 순위에 일치시킬 수 있습니다. 여러 프로필을 만들 수 있습니다.|
|기본 점수 매기기 프로필|선택적으로, 사용자의 점수 매기기 프로필 중 하나를 기본으로 지정하여 기본으로 제공되는 검색 순위 점수를 계산하는 논리를 무시할 수 있습니다.|
|CORS 옵션|필요에 따라 크로스-원본 리소스를 공유할 수 있도록 하고 여기서 도메인 경계를 넘어 발급된 웹 페이지에서 사용하는 리소스를 요청합니다. CORS는 특히 인덱스에 사용하지 않으면 항상 꺼집니다.|

<a name="index-attributes"></a>
##인덱스 특성

특성은 개별 필드에 설정되어 필드를 사용하는 방법을 지정합니다. 다음 테이블은 지정할 수 있는 특성을 열거합니다.

|**속성**|설명|
|------------|-----------|
|**조회 가능**|검색 결과에서 필드를 반환할 수 있는지 여부를 지정합니다.|
|**필터링 가능**|필드를 **$filter** 쿼리에 사용하도록 허용합니다.|
|**정렬 가능**|사용하면 결과 집합에서 필드를 정렬 열로 사용할 수 있습니다.|
|**패싯 가능**|사용자 자기 주도형 필터링을 위해 필드를 패싯 탐색 구조에 사용할 수 있습니다. 일반적으로 여러 문서(예를 들어 단일 브랜드 또는 서비스 범주에 속하는 여러 문서)를 그룹화하는 데 사용할 수 있는 반복되는 값을 포함한 필드는 패싯으로 가장 적합합니다.|
|**키**|문서 조회에 사용하는 각 문서의 고유 ID를 제공하는 문자열입니다. 모든 인덱스에는 하나의 키가 있어야 합니다. 필드 한 개만 키가 될 수 있으며, 이 필드를 Edm.String으로 설정해야 합니다.|
|**검색 가능**|전체 텍스트 검색 가능 필드로 표시합니다.|


##Azure 검색에서 인덱스를 사용하는 방법

인덱스의 데이터는 검색 결과 목록, 패싯 탐색 구조 또는 단일 문서에 대한 정보 페이지를 구성하는 데 사용됩니다. 또한 Azure 검색에서 인덱스는 검색 점수 순위를 높이는 데 사용되는 조건을 제공하는 필터 식 또는 점수 매기기 프로필에서 내부적으로 사용되는 검색할 수 없는 데이터를 포함할 수 있습니다.

Azure 검색에 의해 수행된 모든 데이터 관련 작업은 인덱스의 데이터를 사용하거나 반환합니다. 예외는 없습니다. 검색 서비스에서 내보낸 응답에 외부 데이터를 반환할 인덱스 외에도 다른 데이터 원본에 서비스를 지정할 수 없습니다.

분명히, 온라인 소매 앱의 판매 트랜잭션과 같은 데이터에 대해 축적하고 작동된 응용 프로그램의 경우, Azure 검색 인덱스는 전체 솔루션에 추가 데이터 원본이 됩니다. 검색만을 위해 전용 데이터 저장소를 두는 것이 불필요해 보일 수 있지만, 이러한 저장소를 사용하면 검색 결과의 일관성, 변동성 감소, 응용 프로그램과 Azure 검색간의 라운드 트립 감소, 리소스 또는 데이터 경합 부재로 인한 전반적인 검색 작업 성능 향상 등과 같은 혜택이 있습니다.

##스키마를 정의하기 위한 지침

스키마는 JSON 구조로 생성됩니다. Azure 검색과 통합하는 각 사용자 지정 솔루션에 대해 하나의 인덱스를 만들어야 합니다. 다중 인덱스를 연결하거나 조인시킬 수는 없지만 솔루션에 필요한 다양한 검색 시나리오를 수용하는 복잡한 인덱스를 만들 수 있습니다. 예를 들어, 응용 프로그램이 여러 언어를 지원하는 경우 인텍스의 필드 컬렉션에는 각 필드에 대한 언어별 변형이 있을 수 있습니다.

인덱스를 설계할 때 계획 단계에서 각 의사 결정에 대해 신중하게 생각합니다. 배포된 후 인덱스를 변경하려면 데이터를 다시 작성하고 다시 로드해야 합니다. 코드에서 인덱스를 만드는 경우 이 단계는 포털에서 수동으로 만든 경우보다 훨씬 더 쉽습니다.

우수한 스키마를 만들려면 원래의 원본 데이터를 확실하게 이해해야 합니다. 데이터 형식을 일치시키려면, 어떤 필드를 인덱스의 각 문서를 고유하게 식별하는 *키*로 사용할 지와 어떤 필드를 검색 결과 목록 또는 세부 정보 페이지에 노출시켜야 할지 알아야 합니다.

**관계형 데이터 시작**

관계형 데이터는 플랫 데이터 집합으로 변환하기 어려울 수 있습니다. 가능하면 사용 가능한 경우 귀사의 데이터 웨어하우스나 보고 데이터베이스를 사용해봅니다. 모델이 비정규화되기 때문에 일반적으로 더 좋은 선택입니다. 그렇지 않은 경우 쿼리를 사용하여 필요한 행 집합을 가져와야 합니다. 이 작업을 수행하는 방법에 대한 예 및 설명은 [Azure 검색에 대한 AdventureWorks 재고 데이터베이스 모델링](http://blogs.technet.com/b/onsearch/archive/2015/09/08/modeling-the-adventureworks-inventory-database-for-azure-search.aspx)을 참조하세요.

**이진 데이터 포함**

검색 결과에 이미지, 오디오 또는 비디오 콘텐츠가 필요할 수 있습니다. 이 경우에 파일은 리소스에 대한 URL과 함께 대표 필드를 포함하는 인덱스를 사용하여 다른 곳에 저장되어야 합니다. 특히 이진 콘텐츠를 검색할 수 없는 모든 콘텐츠는 저렴한 저장소에 저장한 다음 인덱스의 필드에 URI로 참조시켜야 합니다.

**데이터 동기화**

예상할 수 있듯이 인덱스는 솔루션에서 사용되는 다른 데이터 소스와 정기적으로 동기화되어야 합니다. 온라인 소매 카탈로그에 있는 판매 트랜잭션을 캡처하는 재고 데이터베이스에는 검색 결과 통해 나타난 데치터와 SKU, 가격, 가용성이 동일해야 합니다. 솔루션에 얼마나 많은 대기 시간을 적용할 수 있는지에 따라, 데이터 동기화가 일주일에 한 번, 하루에 한 번 또는 동시 작성을 사용하여 거의 실시간으로 재고 데이터베이스와 Azure 검색 인덱스를 찾을 수 있습니다. [Azure 검색으로 데이터 가져오기](search-what-is-data-import.md)에 사용할 수 있는 옵션에 대해 자세히 설명되어 있습니다.

## 클라우드 저장소

Azure 검색 하는 완벽하게 관리되는 검색 서비스이기 때문에 데이터 저장소는 내부 작업으로 처리됩니다. 프로그래머는 인덱스에 직접 연결하기 위해 Azure 검색 API를 무시할 수 없음을 이해하는 것이 중요합니다. 그렇지 않으면 포털 알림이나 API 호출이 아니면 인덱스 크기 또는 상태를 모니터링할 수 없습니다. 인덱스를 백업 또는 복원하거나 저장소 계층에서의 최적화 또는 성능 튜닝을 위한 기능이 없습니다. 숨은 기능으로 데이터는 blob 저장소로 저장되지만 물리적 저장소와 인덱스 관리에 대해 생각하는 최적의 방법은 나중에 변경할 수 있는 구현 세부 정보입니다. 서비스의 주요 이점 중 하나는 물리적으로 검색 응용 프로그램의 계산 리소스를 관리하는 자동 접근 방법입니다.

쿼리 볼륨 또는 데이터 저장소 요구 사항이 시간이 지남에 따라 변하는 경우 복제본 및 파티션을 추가하거나 이동하여 용량을 늘리거나 줄일 수 있습니다. 자세한 내용은 [Azure에서 검색 서비스 관리](search-manage.md) 또는 [서비스 제한](search-limits-quotas-capacity.md)을 참조하세요.
