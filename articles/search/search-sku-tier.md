# <a name="choose-a-pricing-tier-for-azure-cognitive-search"></a>Azure 인지 검색에 대한 가격 책정 계층 선택

Azure Cognitive Search 서비스를 만들 때 서비스 수명 동안 고정된 가격 책정 계층(또는 SKU)에서 [리소스가 만들어집니다.](search-create-service-portal.md) 계층에는 무료, 기본, 표준 및 스토리지 최적화가 포함됩니다. 표준 및 스토리지 최적화는 여러 구성 및 용량으로 제공됩니다.

대부분의 고객은 서비스를 평가할 수 있도록 프리 티어로 시작합니다. 사후 평가는 개발 및 프로덕션 배포를 위한 상위 계층 중 하나에서 두 번째 서비스를 만드는 것이 일반적입니다.

## <a name="feature-availability-by-tier"></a>계층별 기능 가용성

무료를 포함하여 거의 모든 계층에서 사용할 수 있지만 충분한 용량을 제공하지 않는 한 리소스를 많이 사용하는 기능이나 워크플로가 잘 작동하지 않을 수 있습니다. 예를 들어 [AI 보강에는](cognitive-search-concept-intro.md) 데이터 집합이 작지 않는 한 무료 서비스에 시간이 걸리는 장기 실행 기술이 있습니다.

다음 표에서는 계층 관련 기능 제약 조건을 설명합니다.

| 기능 | 제한 사항 |
|---------|-------------|
| [인덱서](search-indexer-overview.md) | 인덱서S3 HD에서 사용할 수 없습니다. |
| [고객 관리 암호화 키](search-security-manage-encryption-keys.md) | 프리 티어에는 사용할 수 없습니다. |

## <a name="tiers-skus"></a>계층(SUS)

계층은 다음과 같은 구분으로 구분됩니다.

+ 만들 수 있는 인덱스 및 인덱서의 수량
+ 파티션(실제 스토리지)의 크기와 속도

선택한 계층에 청구 가능 요금이 결정됩니다. Azure 포털의 다음 스크린샷은 사용 가능한 계층에서 가격(포털 및 [가격 책정 페이지에서](https://azure.microsoft.com/pricing/details/search/)찾을 수 있음)을 뺀 값으로 보여 주며 있습니다. **무료**, **기본**및 **표준은** 가장 일반적인 계층입니다.

**Free는** 빠른 시작 및 자습서를 포함하여 소규모 프로젝트에 대한 제한된 검색 서비스를 만듭니다. 내부적으로 복제본과 파티션은 여러 구독자 간에 공유됩니다. 무료 서비스를 확장하거나 중요한 워크로드를 실행할 수 없습니다.

**기본** 및 **표준은** 가장 일반적으로 사용되는 청구 가능한 계층이며 **표준이** 기본값입니다. 전용 리소스를 사용하면 더 큰 프로젝트를 배포하고 성능을 최적화하며 용량을 설정할 수 있습니다.

![Azure 인지 검색의 가격 책정 계층](media/search-sku-tier/tiers.png "Azure 인지 검색의 가격 책정 계층")

일부 계층은 특정 유형의 작업에 최적화되어 있습니다. 예를 들어 **표준 3 고밀도(S3 HD)는** S3의 *호스팅 모드로,* 기본 하드웨어는 많은 수의 작은 인덱스에 최적화되어 있으며 다중 테넌시 시나리오를 위한 것입니다. S3 HD는 S3와 동일한 단위 당 충전을 가지고 있지만 하드웨어는 많은 수의 작은 인덱스에서 빠른 파일 읽기에 최적화되어 있습니다.

**스토리지 최적화** 계층은 표준 계층보다 TB당 더 저렴한 가격으로 더 큰 스토리지 용량을 제공합니다. 기본 절충은 쿼리 대기 시간이 더 높으며 특정 응용 프로그램 요구 사항에 대해 유효성을 검사해야 합니다.  이 계층의 성능 고려 사항에 대해 자세히 알아보려면 [성능 및 최적화 고려 사항을](search-performance-optimization.md)참조하십시오.

[가격 책정 페이지,](https://azure.microsoft.com/pricing/details/search/) [Azure Cognitive Search](search-limits-quotas-capacity.md) 문서의 서비스 제한 및 서비스를 프로비전할 때 포털 페이지에서 다양한 계층에 대해 자세히 알아볼 수 있습니다.

## <a name="billable-events"></a>청구 가능한 이벤트

Azure Cognitive Search를 기반으로 구축된 솔루션은 다음과 같은 방법으로 비용이 발생할 수 있습니다.

+ 최소 구성(하나의 파티션 및 복제본)에서 24x7을 실행하는 서비스 자체의 고정 비용
+ 확장 시 증분 비용(복제본 또는 파티션 추가)
+ 대역폭 요금(아웃바운드 데이터 전송) 
+ 인지 검색(AI 보강을 위해 코그너티브 서비스 연결 또는 지식 저장소에 Azure 저장소 사용)

### <a name="service-costs"></a>서비스 비용

요금을 피하기 위해 "일시 중지"될 수 있는 가상 시스템 또는 기타 리소스와 달리 Azure Cognitive Search 서비스는 전용 하드웨어에서 항상 사용할 수 있습니다. 따라서 서비스 만들기는 서비스를 만들 때 시작되고 서비스를 삭제할 때 종료되는 청구 가능한 이벤트입니다. 

최소 요금은 청구 가능한 요금의 첫 번째 검색 단위(복제본 x 파티션 1개)입니다. 이 최소 값은 서비스가 이 구성보다 적은 서비스에서 실행될 수 없기 때문에 서비스 수명 동안 고정됩니다. 최소 값 이상으로 복제본과 파티션을 서로 독립적으로 추가할 수 있습니다. 복제본 및 파티션을 통한 용량이 점진적으로 증가하면 다음 [수식(복제본 x 파티션 x 속도)에](#search-units)따라 요금이 증가하며, 청구되는 요금은 선택한 가격 책정 계층에 따라 다릅니다.

검색 솔루션의 비용을 추정할 때는 가격 및 용량이 선형적이지 않다는 점에 유의하세요. (용량을 두 배로 늘리면 비용이 두 배 이상 됩니다.) 수식의 작동 방식에 대한 예는 [복제본 및 파티션을 할당하는 방법을 참조하세요.](search-capacity-planning.md#how-to-allocate-replicas-and-partitions)

### <a name="bandwidth-charges"></a>대역폭 요금

[Azure Cognitive Search 인덱서를](search-indexer-overview.md) 사용하면 서비스 위치에 따라 청구에 영향을 줄 수 있습니다. 데이터와 동일한 지역에서 Azure Cognitive Search 서비스를 만드는 경우 데이터 송신 요금을 완전히 제거할 수 있습니다. [대역폭 가격 책정 페이지의](https://azure.microsoft.com/pricing/details/bandwidth/)몇 가지 정보는 다음과 같습니다.

+ Microsoft는 Azure의 서비스 또는 Azure Cognitive Search의 아웃바운드 데이터에 대한 인바운드 데이터에 대해 요금을 부과하지 않습니다.
+ 다중 서비스 솔루션에서는 모든 서비스가 동일한 지역에 있을 때 와이어를 교차하는 데이터에 대한 요금이 부과되지 않습니다.

서비스가 다른 지역에 있는 경우 아웃바운드 데이터에 요금이 부과됩니다. 이러한 요금은 실제로 Azure 인지 검색 청구서의 일부가 아닙니다. 데이터 또는 AI가 풍부한 인덱서를 사용하여 다른 지역의 데이터를 가져오는 경우 전체 청구서에 비용이 반영되므로 여기에 언급되어 있습니다.

[AI 보강의](cognitive-search-concept-intro.md)경우 종량제 처리를 위해 S0 가격 책정 계층에서 Azure Cognitive Search와 동일한 지역에 [청구 가능한 Azure Cognitive Services 리소스를 첨부할](cognitive-search-attach-cognitive-services.md)계획입니다. 코그너티브 서비스 연결과 관련된 고정 비용은 없습니다. 필요한 처리에 대해서만 비용을 지불합니다.

| 작업(Operation) | 청구 영향 |
|-----------|----------------|
| 문서 크래킹, 텍스트 추출 | 무료 |
| 문서 크래킹, 이미지 추출 | 문서에서 추출한 이미지 수에 따라 청구됩니다. [인덱서 구성에서](https://docs.microsoft.com/rest/api/searchservice/create-indexer#indexer-parameters) **imageAction은** 이미지 추출을 트리거하는 매개 변수입니다. **imageAction이** "없음"(기본값)으로 설정된 경우 이미지 추출에 대한 요금이 청구되지 않습니다. 이미지 추출 속도는 Azure 인지 검색의 [가격 세부 정보](https://azure.microsoft.com/pricing/details/search/) 페이지에 설명되어 있습니다.|
| [기본 제공 인지 기술](cognitive-search-predefined-skills.md) | 코그너티브 서비스를 직접 사용하여 작업을 수행한 것과 동일한 비율로 청구됩니다. |
| 사용자 지정 기술 | 사용자 지정 기술은 제공하는 기능입니다. 사용자 지정 기술을 사용하는 비용은 사용자 지정 코드가 다른 유료 서비스를 호출하는지 여부에 따라 달라집니다. |

## <a name="billing-formula-r-x-p--su"></a>청구 수식(R x P = SU)

Azure 인지 검색 작업에 대해 이해해야 할 가장 중요한 청구 개념은 검색 단위(SU)입니다. *search unit* Azure Cognitive Search는 인덱싱 및 쿼리를 위한 복제본과 파티션모두에 종속되므로 둘 중 하나만 청구하는 것은 의미가 없습니다. 대신, 청구는 둘 다의 복합을 기준으로 합니다.

SU는 서비스에서 사용하는 *복제본* 및 *파티션의* **제품입니다.**

모든 서비스는 최소값으로 1SU(복제본 1개 x 파티션 1개)에서 시작합니다. 모든 서비스의 최대 값은 36 개의 US입니다. 이 최대값은 여러 가지 방법으로 도달할 수 있습니다: 6개의 파티션 x 6 복제본, 3개의 파티션 x 12 복제본. 총 용량 미만(예: 3-복제, 9 개의 S로 청구되는 3파티션 서비스)을 사용하는 것이 일반적입니다. 유효한 [조합은 파티션 및 복제본 조합](search-capacity-planning.md#chart) 차트를 참조하십시오.

청구 요금은 SU당 시간당 요금입니다. 각 계층의 비율은 점진적으로 더 높습니다. 계층이 높을수록 파티션이 더 크고 빨라지므로 해당 계층의 시간당 요금이 전반적으로 높아지게 됩니다. [가격 세부 정보](https://azure.microsoft.com/pricing/details/search/) 페이지에서 각 계층의 요금을 확인할 수 있습니다.

대부분의 고객은 총 용량의 일부만 온라인 상태로 유지하고 나머지는 보류 상태로 둡니다. 청구의 경우 SU 수식으로 계산된 온라인 상태로 가져오는 파티션 및 복제본의 수가 시간단위로 지불하는 금액을 결정합니다.

## <a name="how-to-manage-costs"></a>비용 관리 방법

다음 제안사항을 통해 비용을 최소화할 수 있습니다.

+ 대역폭 요금을 최소화하거나 제거하기 위해 동일한 리전 또는 가능한 한 적은 수의 리전에 있는 모든 리소스를 만듭니다.

+ 모든 서비스를 Azure 인지 검색, 인지 서비스 및 솔루션에 사용되는 다른 Azure 서비스와 같은 하나의 리소스 그룹으로 통합합니다. Azure 포털에서 리소스 그룹을 찾고 **비용 관리** 명령을 사용하여 실제 및 예상 지출에 대한 통찰력을 제공합니다.

+ 요청 및 응답이 데이터 센터 경계 내에 유지되도록 프런트 엔드 응용 프로그램에 Azure Web App을 고려합니다.

+ 인덱싱과 같은 리소스 집약적인 작업에 맞게 확장한 다음 일반 쿼리 워크로드에 대해 아래쪽으로 다시 조정합니다. Azure Cognitive Search(하나의 파티션과 하나의 복제본으로 구성된 SU 하나)에 대한 최소 구성으로 시작한 다음 사용자 활동을 모니터링하여 더 많은 용량이 필요하다는 것을 나타내는 사용 패턴을 식별합니다. 예측 가능한 패턴이 있는 경우 규모와 활동(이를 자동화하기 위해 코드를 작성해야 합니다)과 동기화할 수 있습니다.

또한 [지출과](https://docs.microsoft.com/azure/billing/billing-getting-started) 관련된 기본 제공 도구 및 기능에 대한 청구 및 비용 관리를 방문하십시오.

검색 서비스를 일시적으로 종료할 수 없습니다. 전용 리소스는 항상 작동하며 서비스 수명 동안 단독으로 사용할 수 있습니다. 서비스 삭제는 영구적이며 관련 데이터도 삭제합니다.

서비스 자체의 관점에서 청구서를 낮추는 유일한 방법은 복제본및 파티션을 여전히 허용 가능한 성능과 [SLA 준수를](https://azure.microsoft.com/support/legal/sla/search/v1_0/)제공하는 수준으로 줄이거나 더 낮은 계층에서 서비스를 만드는 것입니다(S1 시간당 요금이 S2 또는 S3 요금보다 낮습니다). 부하 예측의 하단에 서비스를 프로비전한다고 가정하면 서비스를 능가하는 경우 두 번째 더 큰 계층서비스를 만들고 두 번째 서비스에서 인덱스를 다시 작성한 다음 첫 번째 서비스를 삭제할 수 있습니다.

## <a name="how-to-evaluate-capacity-requirements"></a>용량 요구 사항을 평가하는 방법

Azure Cognitive Search에서 용량은 *복제본* 및 *파티션으로*구성됩니다.

+ 복제본은 검색 서비스의 인스턴스입니다. 각 복제본은 인덱스의 로드 균형 사본을 하나 호스팅합니다. 예를 들어 6개의 복제본이 있는 서비스에는 서비스에 로드된 모든 인덱스의 복사본이 6개 있습니다.

+ 분할은 인덱스를 저장하고 검색 가능한 데이터를 자동으로 분할합니다. 두 개의 파티션은 인덱스를 반으로 나누고 세 개의 파티션은 3개의 파티션으로 분할됩니다. 용량 측면에서 *파티션 크기는* 계층 간의 기본 차별화 기능입니다.

> [!NOTE]
> 모든 표준 및 스토리지 최적화 계층은 [복제본과 파티션의 유연한 조합을](search-capacity-planning.md#chart) 지원하므로 저울을 변경하여 [시스템을 속도 또는 스토리지에 맞게 최적화할](search-performance-optimization.md) 수 있습니다. 기본 계층은 고가용성을 위해 최대 3개의 복제본을 제공하지만 파티션은 하나만 있습니다. 프리 티어는 전용 리소스를 제공하지 않습니다: 컴퓨팅 리소스는 여러 구독자가 공유합니다.

### <a name="evaluating-capacity"></a>용량 평가

용량과 서비스 운영 비용은 함께 진행됩니다. 계층은 저장소와 리소스의 두 가지 수준에 제한을 가합니다. 먼저 도달하는 제한이 유효 한도이기 때문에 둘 다 생각해 보아야 합니다.

비즈니스 요구 사항은 일반적으로 필요한 인덱스 수를 지정합니다. 예를 들어 대규모 문서 리포지토리에 대한 글로벌 인덱스가 필요할 수 있습니다. 또는 지역, 응용 프로그램 또는 비즈니스 틈새 시장을 기반으로 여러 인덱스가 필요할 수 있습니다.

인덱스의 크기를 결정하려면 [하나를 빌드](search-create-index-portal.md)해야 합니다. 크기는 제안자, 필터링 및 정렬을 사용하도록 설정하는지 여부와 같이 가져온 데이터 및 인덱스 구성을 기반으로 합니다. 크기에 미치는 구성 영향에 대한 자세한 내용은 [기본 인덱스 만들기를 ](search-what-is-an-index.md)참조하십시오.

전체 텍스트 검색의 경우 기본 데이터 구조는 원본 데이터와 다른 특성을 가지는 [반전된 인덱스](https://en.wikipedia.org/wiki/Inverted_index) 구조입니다. 반전된 인덱스의 경우 크기 및 복잡성은 콘텐츠에 따라 결정되며, 반드시 피드하는 데이터의 양에 의해 결정되는 것은 아닙니다. 중복성이 높은 큰 데이터 원본은 매우 가변적인 콘텐츠를 포함하는 작은 데이터 집합보다 인덱스가 작을 수 있습니다. 따라서 원래 데이터 집합의 크기에 따라 인덱스 크기를 추론하는 것은 거의 불가능합니다.

> [!NOTE] 
> 인덱스 와 스토리지에 대한 미래의 요구를 예측하는 것은 추측처럼 느껴질 수 있지만, 그것은 할 가치가있다. 계층의 용량이 너무 낮으면 상위 계층에서 새 서비스를 프로비전한 다음 [인덱스를 다시 로드해야 합니다.](search-howto-reindex.md) 한 SKU에서 다른 SKU로 서비스를 업그레이드하는 것은 없습니다.
>

### <a name="estimate-with-the-free-tier"></a>프리 티어로 추정

용량을 예측하기 위한 한 방법은 무료 계층으로 시작하는 것입니다. 무료 서비스는 최대 3개의 인덱스, 50MB의 저장소 및 2분의 인덱싱 시간을 제공한다는 점을 기억하십시오. 이러한 제약 조건으로 예상 인덱스 크기를 예측하는 것은 어려울 수 있지만 다음 단계는 다음과 같습니다.

+ [무료 서비스를 만듭니다.](search-create-service-portal.md)
+ 작은 대표적인 데이터 집합을 준비합니다.
+ [포털에서 초기 인덱스를 작성하고](search-create-index-portal.md) 크기를 기록합니다. 기능 및 특성은 저장소에 영향을 미칩니다. 예를 들어 제안자(검색형 쿼리)를 추가하면 저장소 요구 사항이 증가합니다. 동일한 데이터 집합을 사용하여 각 필드에 다른 특성을 가진 여러 버전의 인덱스를 만들어 저장소 요구 사항이 어떻게 다른지 확인할 수 있습니다. 자세한 내용은 [기본 인덱스 만들기에서 "저장소 의미"를](search-what-is-an-index.md#index-size)참조하십시오.

대략적인 추정치를 사용하면 두 개의 인덱스(개발 및 생산)에 대한 예산을 두 배로 늘리고 그에 따라 계층을 선택할 수 있습니다.

### <a name="estimate-with-a-billable-tier"></a>청구 가능한 계층으로 예상

전용 리소스는 더 큰 샘플링 및 처리 시간을 수용할 수 있으며 개발 중에 인덱스 수량, 크기 및 쿼리 볼륨을 보다 사실적으로 추정할 수 있습니다. 일부 고객은 청구 가능한 계층으로 바로 이동한 다음 개발 프로젝트가 성숙함에 따라 다시 평가합니다.

1. [각 계층의 서비스 제한을 검토하여](https://docs.microsoft.com/azure/search/search-limits-quotas-capacity#index-limits) 하위 계층이 필요한 인덱스 수를 지원할 수 있는지 여부를 확인합니다. 기본, S1 및 S2 계층에서 인덱스 제한은 각각 15, 50 및 200입니다. 저장소 최적화 계층은 매우 큰 인덱스의 낮은 수를 지원하도록 설계되었기 때문에 10개의 인덱스로 제한됩니다.

1. [청구 가능 계층에서 서비스 만들기](search-create-service-portal.md):

    + 예상 하중에 대해 잘 모르는 경우 기본 또는 S1에서 낮게 시작합니다.
    + 대규모 인덱싱 및 쿼리 로드가 있을 것이라는 것을 알고 있다면 S2 또는 S3에서 높은 시작하십시오.
    + L1 또는 L2에서 스토리지 최적화로 시작하여 많은 양의 데이터를 인덱싱하고 내부 비즈니스 응용 프로그램과 마찬가지로 쿼리 로드가 상대적으로 낮습니다.

1. 원본 데이터가 인덱스로 변환되는 방법을 결정하려면 [초기 인덱스를 만듭니다](search-create-index-portal.md). 인덱스 크기를 추정하는 유일한 방법입니다.

1. 포털에서 [스토리지, 서비스 제한, 쿼리 볼륨 및 대기 시간을 모니터링](search-monitor-usage.md)합니다. 포털에는 초당 쿼리, 제한된 쿼리 및 검색 대기 시간이 표시됩니다. 이러한 모든 값은 올바른 계층을 선택했는지 여부를 결정하는 데 도움이 될 수 있습니다. 

인덱스 번호와 크기는 분석에 똑같이 중요합니다. 이는 저장소(파티션)의 전체 사용또는 리소스(인덱스, 인덱서 등)에 대한 최대 제한을 통해 최대 제한에 도달하기 때문입니다. 포털을 사용하면 둘 모두 추적할 수 있으며, 개요 페이지에 현재 사용량 및 최대 한도가 나란히 표시됩니다.

> [!NOTE]
> 문서에 불필요한 데이터가 포함된 경우 저장소 요구 사항을 부풀려야 할 수 있습니다. 이상적으로 문서에는 검색 경험에 필요한 데이터만 포함됩니다. 이진 데이터는 검색할 수 없으며 별도로 저장해야 합니다(Azure 테이블 또는 Blob 저장소에 있을 수 있음). 그런 다음 외부 데이터에 대한 URL 참조를 보유하도록 인덱스에 필드를 추가해야 합니다. 개별 문서의 최대 크기는 16MB입니다(한 번의 요청으로 여러 문서를 일괄 업로드하는 경우 그 이하). 자세한 내용은 [Azure 인지 검색의 서비스 제한을](search-limits-quotas-capacity.md)참조하십시오.
>

**쿼리 볼륨 고려 사항**

QPS(쿼리당 쿼리)는 성능 튜닝 중에 중요한 메트릭이지만, 처음에는 쿼리 볼륨이 높을 것으로 예상되는 경우 일반적으로 계층 고려 사항일 뿐입니다.

표준 계층은 복제본과 파티션의 균형을 제공할 수 있습니다. 로드 분산을 위한 복제본을 추가하거나 병렬 처리를 위한 파티션을 추가하여 쿼리 처리 를 늘릴 수 있습니다. 그런 다음 서비스를 프로비전한 후 성능을 조정할 수 있습니다.

처음부터 높은 지속적인 쿼리 볼륨을 예상하는 경우 더 강력한 하드웨어가 뒷받침하는 더 높은 표준 계층을 고려해야 합니다. 그런 다음 파티션과 복제본을 오프라인으로 전환하거나 쿼리 볼륨이 발생하지 않는 경우 하위 계층 서비스로 전환할 수 있습니다. 쿼리 처리량을 계산하는 방법에 대한 자세한 내용은 [Azure Cognitive Search 성능 및 최적화를](search-performance-optimization.md)참조하십시오.

저장소 최적화 계층은 대규모 데이터 워크로드에 유용하며 쿼리 대기 시간 요구 사항이 덜 중요한 경우에 대해 보다 전반적인 사용 가능한 인덱스 저장소를 지원합니다. 로드 균형 조정에 추가 복제본을 사용하고 병렬 처리를 위해 추가 파티션을 사용해야 합니다. 그런 다음 서비스를 프로비전한 후 성능을 조정할 수 있습니다.

**서비스 수준 계약**

프리 티어 및 미리 보기 기능은 [서비스 수준 계약(SLA)을](https://azure.microsoft.com/support/legal/sla/search/v1_0/)제공하지 않습니다. 모든 청구 가능 계층에 대해 서비스에 충분한 중복성을 프로비전할 때 SLA가 적용됩니다. 쿼리(읽기) SLA에 대해 두 개 이상의 복제본이 있어야 합니다. 쿼리 및 인덱싱(읽기-쓰기) SLA를 위해 세 개 이상의 복제본이 있어야 합니다. 파티션 수는 SLA에 영향을 주지 않습니다.

## <a name="tips-for-tier-evaluation"></a>계층 평가에 대한 팁

+ 메트릭이 쿼리를 중심으로 빌드하고 사용 패턴(업무 시간 중 쿼리, 사용량이 많은 시간 동안 인덱싱)에 대한 데이터를 수집할 수 있도록 허용합니다. 이 데이터를 사용하여 서비스 프로비저닝 결정을 알립니다. 시간별 또는 일별 흐름에서는 실용적이지 않지만 파티션과 리소스를 동적으로 조정하여 계획된 쿼리 볼륨 변경 사항을 수용할 수 있습니다. 레벨이 조치를 취할 수 있을 만큼 오래 유지되는 경우 계획되지 않았지만 지속적인 변경 사항을 수용할 수도 있습니다.

+ 프로비저닝의 유일한 단점은 실제 요구 사항이 예측보다 큰 경우 서비스를 중단해야 할 수도 있다는 것입니다. 서비스 중단을 방지하려면 상위 계층에서 새 서비스를 만들고 모든 앱 및 요청이 새 끝점을 대상으로 지정할 때까지 나란히 실행합니다.

## <a name="next-steps"></a>다음 단계

프리 티어로 시작하여 데이터의 하위 집합을 사용하여 초기 인덱스를 작성하여 해당 특성을 이해합니다. Azure 인지 검색의 데이터 구조는 반전된 인덱스 구조입니다. 반전된 인덱스의 크기와 복잡성은 내용에 따라 결정됩니다. 중복성이 큰 콘텐츠는 결과적으로 불규칙성이 큰 콘텐츠보다 더 작은 인덱스가 되는 경향이 있습니다. 따라서 데이터 집합의 크기가 아닌 콘텐츠 특성에 따라 인덱스 저장소 요구 사항이 결정됩니다.

인덱스 크기를 초기에 예상한 후 이 문서에서 설명하는 계층 중 하나인 기본, 표준 또는 저장소 최적화에 [대해 청구 가능한 서비스를 프로비전합니다.](search-create-service-portal.md) 데이터 크기 조정에 대한 인위적인 제약 조건을 완화하고 검색할 모든 데이터를 포함하도록 [인덱스를 다시 빌드합니다.](search-howto-reindex.md)

필요한 성능과 규모를 가져오려면 필요에 따라 [파티션 및 복제본을 할당](search-capacity-planning.md)합니다.

성능과 용량이 괜찮으면 작업이 완료된 것입니다. 그렇지 않으면 사용자의 요구에 맞춰 더 밀접하게 조정하는 다른 계층에서 검색 서비스를 다시 만듭니다.
