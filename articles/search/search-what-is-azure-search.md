# Azure 검색이란?

Azure 검색 서비스는 호스팅되는 완전 관리 클라우드 서비스로, 개발자가 .NET SDK 또는 REST API를 사용하여 풍부한 검색 응용 프로그램을 빌드할 수 있도록 합니다. 이 서비스는 콘텐츠 전체를 범위로 설정한 전체 텍스트 검색 및 입력한 용어의 일부에 기초한 자동 완성 쿼리 제안, 적중 항목 강조 및 패싯 탐색 등 상용 웹 검색 엔진에서 볼 수 있는 기능과 유사한 고급 검색을 포함하고 있습니다. 자연 언어 지원은 지정된 언어에 적절한 언어 규칙을 사용하여 기본 제공됩니다.

향상된 검색 또는 저장소 용량의 필요에 따라 서비스를 확장할 수 있습니다. 예를 들어, 소매 업체는 휴일 쇼핑 또는 홍보 이벤트와 관련된 추가 볼륨에 맞게 용량을 늘릴 수 있습니다.

Azure 검색은 웹 서비스 및 HTTP를 사용하여 작업하는 방법을 아는 개발자와 시스템 통합 업체를 위한 API 기반 서비스입니다. Azure 검색은 클라우드 검색 서비스 관리에서 복잡한 과정을 없애고 검색 기반 웹 및 모바일 응용 프로그램을 만드는 과정을 간소화합니다.

##작동 방법

Azure 검색은 서버 및 인프라 관리를 Microsoft에 위임하고 개발자와 시스템 통합 업체는 검색 데이터를 채우고 응용 프로그램에서 액세스하는 즉시 사용 가능한 서비스만 처리하게 하는 [PaaS 서비스](https://wikipedia.org/wiki/Platform_as_a_service)입니다. 서비스를 구성하는 방법에 따라 다른 Azure 검색 구독자와 공유되는 무료 서비스 또는 서비스에만 사용하는 전용 리소스를 제공하는 표준 가격 책정 계층을 사용합니다. 표준 검색은 증가된 저장소 또는 쿼리 부하의 수요에 맞는 옵션을 사용하여 확장할 수 있습니다.

Azure 검색은 전체 텍스트 쿼리를 통해 검색할 수 있는 인덱스에 데이터를 저장합니다. 이러한 인덱스의 스키마는 Azure 포털에서 만들거나 클라이언트 라이브러리 또는 REST API를 사용하여 프로그래밍 방식으로 만들 수 있습니다. 스키마를 정의한 다음 데이터를 Azure 검색 서비스에 업로드할 수 있으며 이후에는 이 서비스에서 데이터가 인덱싱됩니다.

밀어넣기 또는 끌어오기 모델을 사용하여 데이터를 인덱스에 업로드할 수 있습니다. 끌어오기 모델은 요청 시 또는 예약 업데이트에 맞게 구성할 수 있는 인덱서를 통해 제공되며([인덱서 작업(Azure 검색 서비스 REST API)](https://msdn.microsoft.com/library/azure/dn946891.aspx) 참조), Azure DocumentDB, Azure SQL 데이터베이스 또는 Azure VM에서 호스트되는 SQL Server에서 데이터 및 데이터 변경 내용을 쉽게 수집할 수 있습니다. 밀어넣기 모델은 SDK 또는 업데이트된 문서를 인덱스에 보내기 위해 사용되는 REST API를 통해 제공됩니다. JSON 형식으로 되어 있다면 거의 모든 데이터 집합에서 데이터를 밀어넣을 수 있습니다. 데이터 로드에 대한 지침은 [문서 추가, 업데이트 또는 삭제](https://msdn.microsoft.com/library/azure/dn798930.aspx) 또는 [.NET SDK를 사용하는 방법](search-howto-dotnet-sdk.md)을 참조하세요.

일부 개발자는 인덱서가 제공하는 편의성 때문에 인덱서를 선택합니다. 다른 개발자의 경우 추가 작업이 다소 적은 밀어넣기 모델이 유용합니다. 밀어넣기 모델을 선택하는 이유는 두 가지입니다. 첫째, 크롤러 스타일 메커니즘 때문에 데이터 서버에 가해지는 추가 부하를 방지할 수 있습니다. 둘째, 예약 인덱싱 때문에 발생하는 고유한 대기 시간을 방지할 수 있습니다. 사이버 먼데이 또는 블랙 프라이데이에는 사용 가능한 재고의 일상적인 상태를 반영하는 검색을 원할 것입니다. 밀어넣기 모델은 그러한 정도의 정밀도를 제공할 수 있습니다.

##만들고 저장할 내용

일반적인 워크플로는 로컬 개발 환경에서 스키마와 문서를 만든 다음 .NET SDK 또는 REST API를 사용하여 데이터를 업로드, 처리 및 최종적으로 쿼리하는 것입니다. 인덱싱된 모든 데이터는 성능 향상 및 검색 작업 간의 일관성 확보를 위해 Azure 검색 내에 유지됩니다.

포털의 기본 제공 편집기를 사용하여 인덱스 스키마 및 점수 매기기 프로필을 만들 수 있으며, 이는 프로토타입 생성에 최적입니다. 반복 가능하고 자동화된 접근이 필요한 개발자는 코드에 인덱스를 만들 수 있습니다. 새로운 기능은 API에 먼저 추가되므로 응용 프로그램 요구 사항에 따라 프로그래밍 방식만이 유일한 선택인 경우도 있습니다.

스키마를 만드는 경우 검색 응용 프로그램에서 지원되는 필드와 필드 속성을 정의합니다. 필드에는 제품 이름, 설명, 고객의 의견, 브랜드, 가격, 홍보 알림 등과 같은 검색 가능한 데이터를 포함합니다. 특성은 수행할 수 있는 작업의 유형을 결정합니다. 일반적으로 사용되는 특성의 예로 필드가 전체 텍스트 검색(searchable=true), 필터(filterable=true) 또는 패싯(facetable = true)을 지원하는지 여부가 있습니다.

또한 필드는 내부적으로 필터와 점수 매기기 프로필에 사용되는 값 또는 아마도 다른 저장소 플랫폼에 저장된 콘텐츠(예: BLOB 저장소에 보관된 비디오 또는 이미지 파일)에 대한 URL 등과 같은 검색 응용 프로그램과 관련된 검색 불가능한 데이터도 포함합니다. 자주 인용되는 내부 전용 필드의 예로 이윤 또는 잠재적 수익을 나타내는 다른 값이 있습니다. 아마 검색 엔진은 회사에 더 큰 재정적 이익을 가져다 주는 특정 항목을 홍보해야 할 것입니다. 스키마는 이러한 종류의 검색 동작이 가능하게 하는 필드 특성을 포함할 수 있습니다.

문서는 검색 엔진이 검색 결과에 반환하는 자세한 데이터입니다. 예를 들어 검색 응용 프로그램이 온라인 카탈로그인 경우 각 SKU에 대해 하나의 문서를 생성하고 검색 용어와 일치하는 문서에서 반환된 값을 사용하여 검색 결과 페이지를 만들 수 있습니다.

##기능 드릴다운

Azure 검색은 프로비저닝과 규모, 프로그래밍 기능, 액세스 및 제어를 비롯한 여러 범주와 사용자 지정 검색 응용 프로그램에 구현하도록 선택한 기능에서 가치를 제공합니다.

다음 기능 검사 목록은 Azure 검색에 대해 검색 응용 프로그램 요구를 평가하는 데 유용할 수 있습니다. Azure 검색에 대한 최신 업데이트에서 새로운 기능 공지를 찾을 수 있습니다. 또한 아직 구현되지 않은 기능의 상태를 확인하려면 Azure 검색 기능 요청 페이지를 검토할 수 있습니다.

특정 기능에 대한 질문이 있습니까? [Azure 검색 MSDN 포럼](https://social.msdn.microsoft.com/forums/azure/home?forum=azuresearch)을 살펴보십시오. 또한 아직 구현되지 않은 기능의 상태를 확인하려면 [Azure 검색 기능 요청 페이지](http://feedback.azure.com/forums/263029-azure-search)를 검토할 수 있습니다.

###용량 및 규모 기능 및 제한 사항

이 서비스를 표준 또는 공유 서비스 배포로 실행할 수 있습니다. 표준 검색은 워크로드에 따라 확장할 수 있는 전용 리소스를 지원합니다. 공유 서비스는 무료이며 성능이 보장되지 않으므로 서비스 기능을 테스트하기 위한 것입니다.

저장소 및 문서 개수(파티션) 증가 시 또는 쿼리 부하(복제본)의 **확장성**. 각 복제본은 인덱스의 사본 하나를 실행합니다. 고가용성을 달성하려면 쿼리 워크로드의 경우 복제본 2개 및 읽기-쓰기 (쿼리 및 인덱싱) 워크로드의 경우 복제본 3개가 필요합니다. 용량 계획에 대한 자세한 내용은 제한 및 제약 조건(Azure 검색)을 참조하세요.

Azure 검색은 자동으로 이 서비스에 대해 할당한 파티션 전체의 인덱스와 문서를 범위로 설정합니다. 즉, 인덱스 및 문서를 어떤 파티션과 복제본의 집합으로 고정할 수 없습니다.

파티션 및 복제본은 서비스 전체에 적용되는 리소스이며 모든 인덱스는 모든 복제본에서 실행됩니다. 인덱스 격리가 필요하거나 아마도 지리적으로 분산된 서로 다른 데이터 센터의 서비스와 리소스에 대한 요구 사항이 있는 경우 두 번째 서비스를 만들 수 있습니다.

저장소 및 서비스에 로드되는 인덱스와 문서 수에는 제한이 있습니다. 효과적인 제한은 물리적 저장소 고갈 또는 인덱스와 문서 개수의 상위 제한 도달 등과 같은 먼저 발생하는 제한일 것입니다. 자세한 내용은 [제한 및 제약 조건(Azure 검색)](search-limits-quotas-capacity.md)을 참조하세요.

###프로그래밍 기능

REST API는 내용이 JSON 형식으로 되어 있는 HTTP 요청 및 응답으로 구성됩니다. 서비스에 액세스하기 위한 API 및 서비스를 관리하기 위한 API가 있습니다. 자세한 내용은 [Azure 검색 서비스 REST API](https://msdn.microsoft.com/library/azure/dn798935.aspx) 및 [Azure 검색 관리 REST API](https://msdn.microsoft.com/library/azure/dn832684.aspx)를 참조하세요.

.NET SDK에는 HTTP 및 JSON을 직접 다룰 필요 없이 Azure 검색 서비스를 쉽게 사용할 수 있는 클래스가 포함되어 있습니다. 자세한 내용은 [Azure 검색 .NET SDK를 사용하는 방법](search-howto-dotnet-sdk.md)을 참조하세요.

###액세스 및 제어

검색은 HTTPS를 통해서만 액세스할 수 있습니다.

호스트 응용 프로그램에서의 Azure 검색 인증은 HTTP 헤더의 admin api-key를 통해 이루어집니다. 사용자별 인증 또는 권한 부여 모델은 없습니다. 그러나 $filter를 사용하면 사용자 ID별로 액세스를 제한할 수 있습니다. 또한 서로 다른 클라이언트 응용 프로그램에 할당할 수 있는 여러 쿼리 api-key를 사용할 수도 있습니다. 키 관리에 대한 자세한 내용은 [Microsoft Azure에서 검색 서비스 관리](search-manage.md)를 참조하세요. $filter에 대한 자세한 내용은 [문서 검색(Azure 검색 REST API)](https://msdn.microsoft.com/library/azure/dn798927.aspx)을 참조하세요.

###인덱스 및 문서

여러 인덱스를 사용할 수 있습니다(가격 책정 계층에 따른 제한은 [제한 및 제약 조건(Azure 검색)](search-limits-quotas-capacity.md) 참조). 참고로 현재 인덱스 조인은 지원되지 않습니다. 검색 요청은 하나의 인덱스를 지정할 수 있습니다.

**문서**는 필드와 관련 특성을 포함하고 있습니다. 필드는 검색 가능한 텍스트, 필터 및 점수 매기기 프로필에 주로(또는 심지어 독점적으로) 사용하는 값을 포함하고 있으며 다른 데이터 저장소의 콘텐츠(이미지 등)에 대한 URL 또는 포인터를 포함할 가능성이 매우 높습니다. 많은 검색 응용 프로그램은 여러 형태의 저장소를 사용합니다. 이미지 또는 비디오를 Azure Blob 저장소 등의 다른 저장 미디어에 저장하면 비용이 절감될 수 있습니다.

**인덱서**를 사용하면 Azure SQL 데이터베이스, Azure VM의 SQL Server 또는 Azure DocumentDB의 변경된 데이터에서 인덱스 업데이트를 예약할 수 있습니다. 자세한 내용은 [인덱서 작업(Azure 검색 서비스 REST API)](https://msdn.microsoft.com/library/azure/dn946891.aspx)를 참조하세요.

부울 필터에 OData 구문을 사용하고 전체 텍스트 검색에 간단한 쿼리 구문을 사용하여 **쿼리**를 형성할 수 있습니다. 자세한 내용은 [Azure 검색에 대한 OData 식 구문](https://msdn.microsoft.com/library/azure/dn798921.aspx) 및 [Azure 검색의 간단한 쿼리 구문](https://msdn.microsoft.com/library/azure/dn798920.aspx)을 참조하세요.

문서 ID에 따라 특정 문서를 조회하면 특정 항목을 빠르게 검색할 수 있습니다. 이 방법은 사용자가 특정 검색 결과를 클릭할 때 해당 문서에 대한 특정 세부 정보를 조회하려는 경우 유용합니다. 자세한 내용은 [문서 조회(Azure 검색 서비스 REST API)](https://msdn.microsoft.com/library/azure/dn798929.aspx)를 참조하세요.

##검색 응용 프로그램 기능

**전체 텍스트 검색**(텍스트 필드에 애해 기본적으로 설정되어 있음)은 검색 가능한 특성이 있는 필드에 사용할 수 있습니다. 전체 텍스트 검색은 접두사 일치를 기반으로 합니다. 즉, 단어의 중간이나 끝이 아닌 검색 용어의 첫 부분을 바탕으로 일치 여부를 결정합니다. 인덱스를 정의하는 방법의 단계는 [인덱스 만들기(Azure 검색 서비스 REST API)](https://msdn.microsoft.com/library/azure/dn798941.aspx) 또는 [포털에서 인덱스 만들기](search-create-index-portal.md)를 참조하세요.

**점수 매기기 프로필**은 비즈니스 목표에 따라 검색을 최적화하는 순위 모델을 작성하는 데 사용됩니다. 예를 들어 최신 제품 또는 할인 제품ㅇ를 검색 결과의 더 높은 부분에 나타내려고 할 수 있습니다. 또한 따로 추적하고 저장한 고객 검색 기본 설정에 따라 개인화된 점수에 태그를 사용하여 점수 매기기 프로필을 만들 수도 있습니다. 세부 정보는 [검색 인덱스(Azure 검색 서비스 REST API)에 점수 매기기 프로필 추가](https://msdn.microsoft.com/library/azure/dn798928.aspx)를 참조하세요.

**언어 지원**은 잘 알려진 Lucene 분석기 및 Microsoft Office와 Bing(미리 보기에만 해당)의 원동력인 Microsoft 자체의 분석기를 비롯한 자연 언어 처리 스택의 여러 옵션을 사용하여 50가지 언어에 대해 기본 제공됩니다. 자연 언어 프로세서에 대한 자세한 내용은 Lucene에 대한 [언어 지원(Azure 검색 서비스 REST API)](https://msdn.microsoft.com/library/azure/dn879793.aspx) 및 [Azure 검색 서비스 REST API 버전 2015-02-28-미리 보기](search-api-2015-02-28-Preview.md)를 참조하세요.

**패싯 탐색**은 주로 브랜드, 모델, 크기 또는 데이터에 적합한 범주를 기반으로 자기 주도 검색에 사용되는 분류 트리를 가리킵니다. 패싯 탐색은 인덱스의 특성과 쿼리에 제공된 패싯 필드를 결합하여 구현됩니다. 자세한 내용은 [패싯 탐색](search-faceted-navigation.md)을 참조하세요.

자동 완성 기능 또는 자동 완성 쿼리에 대한 **제안**은 인덱스의 특성을 통해 지원됩니다. Azure 검색은 유사 항목 일치 및 접사 일치(필드 콘텐츠의 일부에 대한 일치)를 모두 지원합니다. 두 가지 일치를 모두 수행할 수 있으며 두 가지 일치를 함께 사용하는 것도 가능합니다. 제안을 사용하도록 설정에 대해 자세한 내용은 [인덱스 만들기(Azure 검색 서비스 REST API)](https://msdn.microsoft.com/library/azure/dn798941.aspx), 제안 사용에 대해서는 [제안(Azure 검색 서비스 REST API)](https://msdn.microsoft.com/library/azure/dn798936.aspx)을 참조하세요.

**필터**는 패싯 탐색 구현, 쿼리 구성 또는 검색 작업을 코드에서 설정한 필터 기준으로 전체적으로 제한하기 위해 사용할 수 있습니다. 필터는 인덱스 스키마를 통해 특정 필드에 대해 그리고 $Filter 검색 매개 변수를 통해 활성화됩니다. 자세한 내용은 [인덱스 만들기(Azure 검색 서비스 REST API)](https://msdn.microsoft.com/library/azure/dn798941.aspx) 및 [문서 검색(Azure 검색 서비스 REST API)](https://msdn.microsoft.com/library/azure/dn798927.aspx)을 참조하세요.

**정렬**도 인덱스 스키마를 통해 특정 필드에 대해 사용하도록 설정된 후 $orderBy 매개 변수를 통해 검색 매개 변수로 구현됩니다. 이 기능에 대한 자세한 내용도 [인덱스 만들기(Azure 검색 서비스 REST API)](https://msdn.microsoft.com/library/azure/dn798941.aspx) 및 [문서 검색(Azure 검색 서비스 REST API)](https://msdn.microsoft.com/library/azure/dn798927.aspx)에서 참조할 수 있습니다.

쿼리에 대해 반환되는 검색 적중 **개수** 및 한 번에 반환되는 적중 수를 제한하는 기능은 $top 및 $skip을 통해 구현됩니다. 자세한 내용은 [문서 검색(Azure 검색 REST API)](https://msdn.microsoft.com/library/azure/dn798927.aspx)을 참조하세요.

**강조 표시된 적중 수**는 강조 표시 쿼리 매개 변수를 통해 지정되며, 이 기능을 사용하여 사용자가 입력한 검색 용어에서 발견된 키워드를 강조하는 텍스트의 조각을 사용자에게 표시할 수 있습니다. 쿼리 매개 변수에 대한 자세한 내용은 [문서 검색(Azure 검색 서비스 REST API)](https://msdn.microsoft.com/library/azure/dn798927.aspx)을 참조하세요.

##보고 및 분석

리소스 사용 현황은 저장소 사용 방법을 빨리 파악할 수 있도록 서비스 대시보드에 표시됩니다.

저장소 사용량, 문서 개수 및 인덱스 개수는 포털에서 사용할 수 있습니다. API를 사용할 수도 있습니다. 자세한 내용은 [인덱스 통계 가져오기(Azure 검색 서비스 REST API)](https://msdn.microsoft.com/library/azure/dn798942.aspx)를 참조하세요.

쿼리 처리량 또는 다른 서비스 작업을 측정하기 위한 기본 제공 메커니즘은 없습니다. 또한 로깅 또는 검색 결과 분석(예를 들어 결과가 없는 검색 용어 목록 검색 또는 검색 요청 원본에 대한 보고)에 대한 지원은 현재 없습니다.
