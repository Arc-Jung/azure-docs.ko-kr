# <a name="create-an-effective-anchor-experience-by-using-azure-spatial-anchors"></a>Azure 공간 앵커를 사용하여 효과적인 앵커 환경 만들기

이 문서에서는 공간 앵커를 사용하여 앵커를 효과적으로 만들고 찾는 데 도움이 되는 지침과 고려 사항을 제공합니다.

## <a name="good-anchors"></a>좋은 앵커

공간 앵커는 좋은 앵커를 만드는 데 도움이 됩니다. 좋은 앵커를 만들기 위해 UX(사용자 환경)에서 사용자를 교육하거나 안내하는 데 시간을 투자하는 것이 중요합니다. 좋은 앵커를 미리 만드는 데 투자하면 최종 사용자가 앵커를 안정적으로 찾을 수 있습니다.

- 다양한 장치에서.
- 여러 번.
- 다른 조명 조건에서.
- 공간 내에서 원하는 관점에서.

## <a name="static-and-dynamic-locations"></a>정적 및 동적 위치

앵커 환경을 디자인하는 일의 일부는 위치를 선택하는 것입니다. 위치가 정적이고 공간 관리자가 정의합니까? 아니면 사용자가 동적이고 정의할 것인가?

소매점 관리자는 사용자가 방문하도록 유도하기 위해 정적 매장 내 환경을 원할 수 있습니다. 그러나 혼합 현실 보드 게임의 개발자는 사용자가 재생할 위치를 선택할 수 있도록 할 가능성이 높습니다.

정적 위치의 경우 관리자에게 좋은 앵커로 공간을 큐레이팅하는 데 시간을 할애하도록 가르칠 수 있습니다.

동적 위치의 경우 UX에서 사용자를 가르치거나 안내하여 좋은 앵커를 만드는 방법을 고려해야 합니다.

## <a name="stable-visual-features"></a>안정적인 시각적 기능

혼합 현실 및 증강 현실 장치에 사용되는 시각적 추적 시스템은 환경의 시각적 기능에 의존합니다. 가장 신뢰할 수 있는 환경을 얻으려면 다음을 수행하십시오.

- 안정적인 시각적 특징(즉, 자주 변경되지 않는 기능)이 있는 위치에 앵커를 *만드세요.*

- 뚜렷한 특성이 없는 큰 빈 서피스에 앵커를 만들지 *마십시오.*

- 반사율이 높은 재질에 앵커를 만들지 *마십시오.*

- 패턴이 반복되는 표면에 카펫이나 벽지와 같이 앵커를 만들지 *마십시오.*

## <a name="various-viewing-perspectives"></a>다양한 관점

앵커를 만들 때 나중에 앵커를 찾으려고 할 사람들에 대해 생각해 보십시오.

예를 들어 두 개의 문이 있는 방 중앙에 있는 앵커를 생각해 보십시오. 사용자가 어느 문에서든 룸에 들어갈 수 있도록 허용할 수 있습니다. 앵커를 작성할 때 두 출입구에서 해당 위치를 스캔해야 합니다. 사용자가 양쪽 문에서 앵커를 찾을 수 있도록 원근을 변경하여 앵커 주변의 환경 데이터를 캡처합니다.

일반적으로 앵커를 만들 때 앵커를 찾으려는 사람들의 관점에서 스캔합니다. 따라서 야외 조각품에 가상 콘텐츠를 배치하는 경우 앵커를 만들 때 조각을 스캔하면서 조각을 돌아다니는 것이 합리적입니다. 앵커가 방 구석에 있는 경우 앵커에 접근할 방향은 하나뿐입니다. 이 앵커를 작성할 때 이 관점에서만 스캔할 수 있습니다.

## <a name="multiple-anchors"></a>여러 앵커

조명은 앱이 감지하는 시각적 기능에 차이를 만들 수 있습니다. 강한 자연광으로 생성된 앵커는 인공 조명에서 찾기 어렵고 그 반대의 경우도 마찬가지입니다.

이 문제가 있는 경우 두 개의 앵커를 만드는 데 도움이 될 수 있습니다. 같은 지점에서 일광에 하나의 앵커와 인공 조명을 만듭니다. 그러면 앱에서 두 앵커를 모두 쿼리할 수 있습니다. 두 앵커 중 하나가 있으면 앱에 앵커에 대한 포즈가 있습니다.

마찬가지로 대부분의 개체가 이동하기 때문에 시각적 기능이 변경되는 환경에서는 여러 앵커가 도움이 될 수 있습니다. 환경의 중대한 변경으로 인해 앵커를 찾기가 너무 어려워지면 앵커를 새 앵커로 바꿀 수 있습니다. 예를 들어 몇 개월마다 레이아웃이 새로 고쳐지는 소매점에서 이 작업을 수행할 수 있습니다.

## <a name="targets-and-rooms"></a>대상 및 룸

대부분의 경우 앵커는 앱 환경의 진입점입니다. 사용자가 사용자 경험을 입력할 수 있도록 이 단계를 빠르고 안정적으로 진행해야 합니다. 사용자가 앵커를 찾는 방법에 시간을 할애하는 것은 중요한 디자인 단계입니다. *대상과* *룸이라는*두 가지 광범위한 시나리오에서 앵커를 찾는 것이 좋습니다.

### <a name="targets"></a>대상

대상 시나리오에서 앵커의 위치는 잘 알려져 있습니다. 예를 들어 가상의 혼합 현실 페인팅 앱에서 한 사용자가 벽에 가상 캔버스를 배치합니다. 그녀는 방에 있는 다른 사용자에게 벽의 같은 장소에서 장치를 가리키도록 지시하여 앵커를 찾아 환경을 시작하도록 지시합니다.

대상 시나리오의 또 다른 예로는 커피숍에서 "거래 스캔"이라는 기호가 있을 수 있습니다. 커피숍은 여기에 앵커를 배치했습니다. 사용자가 표지판을 스캔할 때 앵커를 찾아 증강 현실 경험을 입력하여 커피 거래를 찾습니다.

대상 시나리오에서 사진이 도움이 될 수 있습니다. 기기에 의도한 대상 사진을 사용자에게 표시하면 실제 환경에서 스캔할 대상을 신속하게 식별할 수 있습니다. 예를 들어 사용자가 GPS를 사용하여 의도한 대상의 일반 영역 내에 도착하는 데 도움을 줄 수 있습니다. 사용자가 도착하면 앱에 대상 사진이 표시됩니다. 사용자는 공간을 둘러보고 대상을 찾은 다음 앵커를 검사합니다.

![사용자의 모바일 장치에서 대상 사진을 보여주는 앵커 그림](./media/start-here-edit.png)

### <a name="rooms"></a>대화방

룸 시나리오에서 사용자는 어딘가에 앵커가 있다는 것을 알기만 하면 공백에 들어갑니다. 사용자는 장치로 공간을 스캔하고 신속하게 앵커를 찾습니다.

이 환경에서는 일반적으로 다양한 보기 관점에서 설명한 대로 잘 선별된 앵커를 만들어야 합니다. 앵커를 만들 때 여러 관점에서 룸을 스캔한 경우 사용자가 위치를 찾으려고 할 때 거의 모든 곳에서 회의실을 스캔할 수 있습니다.

![사용자가 회의실을 스캔하여 앵커를 찾는 방법 그림](./media/scan-room.png)

기본적으로 나중에 사용자가 앵커를 빠르게 스캔하고 찾을 수 있도록 앵커를 만들 때 공간을 검색하는 데 더 많은 시간을 할애할 수 있습니다. 환경을 만들 때 이 중요한 장단점을 고려해야 합니다.

앞에서 설명한 혼합 현실 페인팅 앱의 예는 룸 시나리오로는 잘 작동하지 않습니다. 여기서 앵커를 배치하는 사용자는 다른 사용자가 신속하게 경험에 참여하기를 원합니다. 사용자는 룸을 잘 스캔할 때까지 환경을 시작하기를 기다리지 않습니다. 모든 사용자가 장치를 위치를 정확히 파악하여 앵커를 찾을 수 있으므로 이 예제는 대상 시나리오로 더 잘 작동합니다.

## <a name="anchor-location"></a>앵커 위치

시각적 추적 시스템은 환경의 시각적 기능에 의존합니다. 스캔에 포함된 시각적 기능이 많을수록 앵커를 찾을 가능성이 높아집니다.

이 섹션의 일반적인 지침을 따라 환경의 유용한 검사를 권장하는 UX를 빌드합니다.

첫째, 사용자가 몇 초 내에 앵커를 찾지 못하면 앱에서 사용자가 장치를 이동하여 더 많은 관점을 캡처하도록 권장해야 합니다. 또한 이 앱은 사용자가 환경 주위로 이동하여 더 많은 관점에서 앵커를 스캔하도록 유도할 수 있습니다. 장치가 보는 기능 관점이 많을수록 더 좋습니다.

대상 시나리오의 경우 사용자에게 대상 을 이동하여 다른 관점에서 볼 수 있도록 요청합니다. 즉, 앵커가 위치할 때까지 사용자에게 새 관점에서 대상을 캡처하도록 요청합니다.

룸 시나리오의 경우 사용자에게 룸을 천천히 스캔하도록 요청합니다. 예를 들어, 사용자에게 180도 또는 360도 의 방을 캡처하도록 요청합니다. 또는 사용자에게 새로운 관점에서 회의실을 보도록 요청합니다.

가장 의미있는 방법은 방을 가로 질러 스캔하는 것입니다. 예를 들어, 방을 가로지르는 스캔은 주변 벽을 스캔하는 것보다 환경의 더 많은 시각적 특징을 캡처합니다. 근처 벽을 스캔하면 환경의 유용한 시각적 특징이 많이 캡처되지 않습니다.

앵커를 검색할 때 장치를 좌우로 반복적으로 이동하는 것은 도움이 되지 않습니다. 이것은 단순히 동일한 관점에서 동일한 점을 캡처합니다.

## <a name="experience-tests"></a>경험 테스트

이 문서에서는 일반적인 지침에 대해 설명했습니다. 공간 앵커를 사용하면 실제 세계와 상호 작용하는 앱을 작성할 수 있습니다. 따라서 실제 환경에서 앱의 앵커 시나리오를 테스트하는 데 시간을 할애해야 합니다. 특히 사용자가 앱을 사용할 것으로 예상되는 위치를 나타내는 환경에서는 이러한 적용이 가능합니다.
