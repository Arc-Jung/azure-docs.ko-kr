# <a name="azure-storage-redundancy"></a>Azure Storage 중복성

Azure Storage는 일시적인 하드웨어 오류, 네트워크 또는 정전 및 대규모 자연 재해를 포함하여 계획되고 계획되지 않은 이벤트로부터 보호되도록 항상 데이터의 여러 복사본을 저장합니다. 중복성은 저장소 계정이 장애가 발생한 경우에도 [Azure 저장소에 대한 SLA(서비스 수준 계약)를](https://azure.microsoft.com/support/legal/sla/storage/) 충족하는지 확인합니다.

시나리오에 가장 적합한 중복 옵션을 결정할 때는 낮은 비용과 높은 가용성 및 내구성 간의 장단점을 고려해야 합니다. 선택해야 할 중복 옵션을 결정하는 데 도움이 되는 요소는 다음과 같습니다.  

- 데이터가 기본 리전에서 복제되는 방법
- 데이터가 지리적으로 기본 지역과 멀리 떨어진 두 번째 위치로 복제되어 지역 재해로부터 보호되는지 여부
- 어떤 이유로든 기본 지역을 사용할 수 없게 되는 경우 응용 프로그램에서 보조 지역의 복제된 데이터에 대한 읽기 액세스가 필요한지 여부

## <a name="redundancy-in-the-primary-region"></a>기본 리전의 중복성

Azure Storage 계정의 데이터는 항상 기본 리전에서 세 번 복제됩니다. Azure Storage는 데이터가 기본 리전에서 복제되는 방법에 대한 두 가지 옵션을 제공합니다.

- **LRS(로컬 중복 저장소)는** 기본 지역의 단일 물리적 위치 내에서 데이터를 3번 동시에 복사합니다. LRS는 가장 비용이 많이 드는 복제 옵션이지만 고가용성이 필요한 응용 프로그램에는 권장되지 않습니다.
- **ZRS(영역 중복 저장소)는** 기본 리전의 세 Azure 가용성 영역에서 데이터를 동기적으로 복사합니다. 고가용성이 필요한 응용 프로그램의 경우 기본 리전에서 ZRS를 사용하고 보조 지역으로 복제하는 것이 좋습니다.

### <a name="locally-redundant-storage"></a>LRS(로컬 중복 스토리지)

LRS(로컬 중복 저장소)는 기본 지역의 단일 물리적 위치 내에서 데이터를 세 번 복제합니다. LRS는 특정 연도동안 최소 99.99999999999%(119999999)의 내구성을 제공합니다.

LRS는 가장 저렴한 중복 옵션이며 다른 옵션에 비해 내구성이 가장 낮습니다. LRS는 서버 랙 및 드라이브 오류로부터 데이터를 보호합니다. 그러나 데이터 센터 내에서 화재 나 홍수와 같은 재해가 발생하면 LRS를 사용하는 저장소 계정의 모든 복제본이 손실되거나 복구 할 수 없습니다. 이러한 위험을 완화하기 위해 ZRS(영역 [중복 저장소),](#zone-redundant-storage) 지역 중복 저장소(GRS) 또는 [geo-redundant storage](#geo-redundant-storage) [지역 영역 중복 저장소(미리 보기)(GZRS)를](#geo-zone-redundant-storage-preview) 사용하는 것이 좋습니다.

LRS를 사용하는 저장소 계정에 대한 쓰기 요청이 동기적으로 발생합니다. 데이터가 세 복제본 모두에 기록된 후에만 쓰기 작업이 성공적으로 반환됩니다.

LRS는 다음 시나리오에 적합합니다.

- 데이터 손실이 발생하더라도 쉽게 재구성할 수 있는 데이터를 애플리케이션에 저장하는 경우 LRS를 선택할 수 있습니다.
- 응용 프로그램이 데이터 거버넌스 요구 사항으로 인해 국가 또는 지역 내에서만 데이터를 복제하도록 제한되는 경우 LRS를 선택할 수 있습니다. 경우에 따라 데이터가 지리적으로 복제되는 페어링된 영역이 다른 국가 또는 지역에 있을 수 있습니다. 쌍을 이루는 지역에 대한 자세한 내용은 [Azure 지역](https://azure.microsoft.com/regions/)을 참조하세요.

### <a name="zone-redundant-storage"></a>영역 중복 스토리지

ZRS(영역 중복 저장소)는 기본 리전의 세 Azure 가용성 영역에서 Azure Storage 데이터를 동기적으로 복제합니다. 각 가용성 영역은 독립적인 전력, 냉각 및 네트워킹을 갖춘 별도의 물리적 위치입니다. ZRS는 지정된 연도에 걸쳐 최소 99.9999999999999%(129초)의 Azure Storage 데이터 개체에 대한 내구성을 제공합니다.

ZRS를 사용하면 영역을 사용할 수 없게 되더라도 읽기 및 쓰기 작업 모두에 대해 데이터에 계속 액세스할 수 있습니다. 영역을 사용할 수 없게 되면 Azure는 DNS 다시 가리키기와 같은 네트워킹 업데이트를 수행합니다. 이러한 업데이트는 업데이트가 완료되기 전에 데이터에 액세스하는 경우 응용 프로그램에 영향을 줄 수 있습니다. ZRS용 응용 프로그램을 디자인할 때는 지수 백오프를 통해 재시도 정책을 구현하는 것을 포함하여 일시적인 오류 처리 에 대한 관행을 따릅니다.

ZRS를 사용하는 저장소 계정에 대한 쓰기 요청이 동기적으로 발생합니다. 데이터가 세 가지 가용성 영역에 걸쳐 있는 모든 복제본에 기록된 후에만 쓰기 작업이 성공적으로 반환됩니다.

일관성, 내구성 및 고가용성이 필요한 시나리오에는 기본 지역에서 ZRS를 사용하는 것이 좋습니다. ZRS는 일시적으로 사용할 수 없는 경우 데이터에 대해 뛰어난 성능, 낮은 대기 시간 및 복원력을 제공합니다. 그러나 ZRS 자체는 여러 영역이 영구적으로 영향을 받는 지역 재해로부터 데이터를 보호하지 못할 수 있습니다. 지역 재해를 방지하기 위해 Microsoft는 기본 지역에서 ZRS를 사용하고 데이터를 보조 지역으로 지리적 복제하는 [GZRS(지역 영역 중복 저장소)를](#geo-zone-redundant-storage-preview) 사용하는 것이 좋습니다.

다음 표에서는 어떤 유형의 저장소 계정이 어떤 지역에서 ZRS를 지원하는지 보여 주며,

|    Storage 계정 유형    |    지원되는 지역    |    지원되는 서비스    |
|----------------------------|---------------------------------------------------------------------------------------------------------------------------------------------------------------------|---------------------------------------------------------------------------------------------------------------------------------------------|
|    범용 v2<sup>1</sup>    | 동남아시아<br /> 오스트레일리아 동부<br /> 북유럽<br />  서유럽<br /> 프랑스 중부<br /> 일본 동부<br /> 남아프리카 북부<br /> 영국 남부<br /> 미국 중부<br /> 미국 동부<br /> 미국 동부 2<br /> 미국 서부 2    |    블록 Blob<br /> 페이지<sup>Blob2</sup><br /> 파일 공유(표준)<br /> 테이블<br /> 큐<br /> |
|    블록블블스토리지<sup>1</sup>    | 서유럽<br /> 미국 동부    |    블록 Blob만 해당    |
|    파일 스토리지    | 서유럽<br /> 미국 동부    |    Azure 파일만    |

GRS를 사용하는 Azure 관리되지 않는 디스크에 대한 저장소 계정을 만들 수 있지만 비동기 지역 복제에 대한 일관성이 있는 잠재적인 문제로 인해 권장되지 않습니다. 관리되거나 관리되지 않는 디스크는 ZRS 또는 GZRS를 지원하지 않습니다. 관리 디스크에 대한 자세한 내용은 [Azure 관리 디스크의 가격 책정을](https://azure.microsoft.com/pricing/details/managed-disks/)참조하십시오.

ZRS를 지원하는 지역에 대한 자세한 내용은 [Azure 가용성 영역의](../../availability-zones/az-overview.md) **지역별 서비스 지원을** 참조하십시오.

## <a name="redundancy-in-a-secondary-region"></a>보조 지역의 중복성

고가용성이 필요한 애플리케이션의 경우 저장소 계정의 데이터를 기본 리전에서 수백 마일 떨어진 보조 지역으로 추가로 복사하도록 선택할 수 있습니다. 저장소 계정이 보조 지역으로 복사된 경우 전체 지역 중단 또는 기본 지역을 복구할 수 없는 재해가 발생한 경우에도 데이터가 지속됩니다.

스토리지 계정을 만들 때 계정에 대한 기본 지역을 선택합니다. 쌍을 이루는 보조 지역은 주 지역에 따라 결정되며 변경할 수 없습니다. Azure에서 지원하는 지역에 대한 자세한 내용은 [Azure 지역을](https://azure.microsoft.com/global-infrastructure/regions/)참조하십시오.

Azure Storage는 데이터를 보조 지역으로 복사하는 두 가지 옵션을 제공합니다.

- **지리적 중복 저장소(GRS)는 LRS를** 사용하여 기본 지역의 단일 물리적 위치 내에서 데이터를 3회 동시에 복사합니다. 그런 다음 데이터를 비동기적으로 보조 지역의 단일 물리적 위치로 복사합니다.
- **지리 영역 중복 저장소(GZRS)** (미리 보기)는 ZRS를 사용하여 기본 지역의 세 Azure 가용성 영역에서 데이터를 동기적으로 복사합니다. 그런 다음 데이터를 비동기적으로 보조 지역의 단일 물리적 위치로 복사합니다.

GRS와 GZRS의 주요 차이점은 데이터가 기본 영역에서 복제되는 방식입니다. 보조 위치 내에서 데이터는 항상 LRS를 사용하여 세 번 동기적으로 복제됩니다.

GRS 또는 GZRS를 사용하면 보조 지역에 장애 조치(failover)가 없는 한 보조 위치의 데이터를 읽기 또는 쓰기 액세스에 사용할 수 없습니다. 보조 위치에 대한 읽기 액세스를 위해 읽기 액세스 지리적 중복 저장소(RA-GRS) 또는 읽기 액세스 지리적 영역 중복 저장소(RA-GZRS)를 사용하도록 저장소 계정을 구성합니다. 자세한 내용은 [보조 지역의 데이터에 대한 액세스 읽기를](#read-access-to-data-in-the-secondary-region)참조하십시오.

기본 영역을 사용할 수 없는 경우 보조 영역(미리 보기)으로 장애 조치(failover)하도록 선택할 수 있습니다. 장애 조치(failover)가 완료되면 보조 지역이 기본 영역이 되고 데이터를 다시 읽고 쓸 수 있습니다. 재해 복구에 대한 자세한 정보 및 보조 지역으로 장애 조치하는 방법을 알아보려면 [재해 복구 및 계정 장애 조치(미리 보기)를](storage-disaster-recovery-guidance.md)참조하십시오.

> [!IMPORTANT]
> 데이터가 비동기적으로 보조 지역으로 복제되기 때문에 기본 영역에 영향을 주는 오류로 인해 기본 영역을 복구할 수 없는 경우 데이터가 손실될 수 있습니다. 주 영역에 대한 가장 최근 쓰기와 보조 영역에 대한 마지막 쓰기 사이의 간격을 RPO(복구 지점 목표)라고 합니다. RPO는 데이터를 복구할 수 있는 시점을 나타냅니다. Azure Storage에는 일반적으로 RPO가 15분 미만이지만 현재 보조 지역으로 데이터를 복제하는 데 걸리는 SLA는 없습니다.

### <a name="geo-redundant-storage"></a>지역 중복 스토리지

지리적 중복 저장소(GRS)는 LRS를 사용하여 기본 지역의 단일 물리적 위치 내에서 데이터를 3회 동시에 복사합니다. 그런 다음 데이터를 기본 리전에서 수백 마일 떨어진 보조 지역의 단일 물리적 위치로 비동기적으로 복사합니다. GRS는 지정된 연도에 걸쳐 최소 99.999999999999999999999%(169초)의 Azure Storage 데이터 개체에 대한 내구성을 제공합니다.

쓰기 작업은 먼저 기본 위치에 커밋되고 LRS를 사용하여 복제됩니다. 그런 다음 업데이트가 보조 영역에 비동기적으로 복제됩니다. 데이터가 보조 위치에 기록되는 경우 LRS를 사용하여 해당 위치 내에도 복제됩니다.

### <a name="geo-zone-redundant-storage-preview"></a>지리적 영역 중복 저장소(미리 보기)

지리 영역 중복 저장소(GZRS) (미리 보기)는 가용성 영역 간에 중복성이 제공하는 고가용성과 지역 복제에서 제공하는 지역 정전으로부터 보호합니다. GZRS 저장소 계정의 데이터는 기본 지역의 세 [Azure 가용성 영역에](../../availability-zones/az-overview.md) 복사되며 지역 재해로부터 보호하기 위해 보조 지리적 영역으로 복제됩니다. 최대 일관성, 내구성 및 가용성, 우수한 성능 및 재해 복구를 위한 복원력이 필요한 응용 프로그램에는 GZRS를 사용하는 것이 좋습니다.

GZRS 저장소 계정을 사용하면 가용성 영역을 사용할 수 없거나 복구할 수 없는 경우 데이터를 계속 읽고 쓸 수 있습니다. 또한 전체 지역 정전 또는 기본 지역을 복구할 수 없는 재해의 경우에도 데이터가 지속됩니다. GZRS는 특정 연도에 걸쳐 최소 99.999999999999999999%(169초) 내구성을 제공하도록 설계되었습니다.

범용 v2 저장소 계정만 GZRS 및 RA-GZRS를 지원합니다. 애플리케이션 계정 유형에 대한 자세한 내용은 [Azure Storage 계정 개요](storage-account-overview.md)를 참조하세요. GZRS 및 RA-GZRS는 블록 Blob, 페이지 Blob(VHD 디스크 제외), 파일, 테이블 및 큐를 지원합니다.

GZRS 및 RA-GZRS는 현재 다음 리전에서 미리 보기를 사용할 수 있습니다.

- 동남아시아
- 북유럽
- 서유럽
- 일본 동부
- 영국 남부
- 미국 동부
- 미국 동부 2
- 미국 중부
- 미국 서부 2

Microsoft는 추가 Azure 지역에서 GZRS 및 RA-GZRS를 계속 사용하도록 설정합니다. 지원되는 지역에 대한 자세한 내용은 [Azure 서비스 업데이트](https://azure.microsoft.com/updates/) 페이지를 정기적으로 확인하십시오.

미리 보기 가격 책정에 대한 자세한 내용은 [Blob,](https://azure.microsoft.com/pricing/details/storage/blobs) [파일,](https://azure.microsoft.com/pricing/details/storage/files/) [큐](https://azure.microsoft.com/pricing/details/storage/queues/)및 [표에](https://azure.microsoft.com/pricing/details/storage/tables/)대한 GZRS 미리 보기 가격을 참조하십시오.

> [!IMPORTANT]
> 프로덕션 워크로드에 미리 보기 기능을 사용하지 않도록 하는 것이 좋습니다.

## <a name="read-access-to-data-in-the-secondary-region"></a>보조 지역의 데이터에 대한 액세스 읽기

GRS 또는 GZRS를 사용하여 지리적 중복 저장소는 데이터를 보조 지역의 다른 물리적 위치로 복제하여 지역 정전으로부터 보호합니다. 그러나 해당 데이터는 고객 또는 Microsoft가 기본 에서 보조 지역으로 장애 조치(failover)를 시작하는 경우에만 읽을 수 있습니다. 보조 영역에 대한 읽기 액세스를 사용하도록 설정하면 기본 지역을 사용할 수 없는 경우 데이터를 읽을 수 있습니다. 보조 영역에 대한 읽기 액세스를 위해 읽기 액세스 지리적 중복 저장소(RA-GRS) 또는 읽기 액세스 지리적 영역 중복 저장소(RA-GZRS)를 사용하도록 설정합니다.

### <a name="design-your-applications-for-read-access-to-the-secondary"></a>보조 에 대한 읽기 액세스를 위해 응용 프로그램 설계

저장소 계정이 보조 지역에 대한 읽기 액세스를 위해 구성된 경우 어떤 이유로든 기본 지역을 사용할 수 없게 되는 경우 보조 리전에서 데이터를 읽도록 응용 프로그램을 원활하게 이동하도록 디자인할 수 있습니다. 보조 영역은 항상 읽기 액세스에 사용할 수 있으므로 응용 프로그램을 테스트하여 가동 중단 시 보조 영역에서 읽을 수 있는지 확인할 수 있습니다. 고가용성을 위해 응용 프로그램을 디자인하는 방법에 대한 자세한 내용은 [읽기 액세스 지리적 중복 저장소를 사용하여 가용성이 높은 응용 프로그램 설계를](storage-designing-ha-apps-with-ragrs.md)참조하십시오.

보조 에 대한 읽기 액세스가 활성화되면 저장소 계정의 기본 끝점뿐만 아니라 보조 끝점에서 데이터를 읽을 수 있습니다. 보조 끝점은 계정 이름에 *보조 접미사를* 더합니다. 예를 들어 Blob 저장소의 기본 끝점이 `myaccount.blob.core.windows.net`다음과 같은 경우 `myaccount-secondary.blob.core.windows.net`보조 끝점은 . 저장소 계정의 계정 액세스 키는 기본 끝점과 보조 끝점 모두에 대해 동일합니다.

### <a name="check-the-last-sync-time-property"></a>마지막 동기화 시간 속성 확인

데이터가 비동기적으로 보조 지역으로 복제되기 때문에 보조 영역은 종종 기본 영역 뒤에 있습니다. 기본 리전에서 오류가 발생하면 기본 에 대한 모든 쓰기가 아직 보조 로 복제되지 않았을 수 있습니다.

보조 지역에 복제된 쓰기 작업을 확인하려면 응용 프로그램에서 저장소 계정에 대한 **마지막 동기화 시간** 속성을 확인할 수 있습니다. 마지막 동기화 시간 이전에 기본 영역에 작성된 모든 쓰기 작업이 보조 지역으로 성공적으로 복제되어 보조 영역에서 읽을 수 있습니다. 마지막 동기화 시간 이후에 기본 영역에 작성된 모든 쓰기 작업은 보조 지역으로 복제될 수도 있고 복제되지 않았을 수도 있습니다.

Azure PowerShell, Azure CLI 또는 Azure Storage 클라이언트 라이브러리 중 하나를 사용하여 **마지막 동기화 시간** 속성의 값을 쿼리할 수 있습니다. **마지막 동기화 시간** 속성은 GMT 날짜/시간 값입니다. 자세한 내용은 [저장소 계정의 마지막 동기화 시간 속성을 참조하십시오.](last-sync-time-get.md)

## <a name="data-integrity"></a>데이터 무결성

Azure Storage는 주기적으로 CRC(순환 중복 검사)를 사용하여 저장된 데이터의 무결성을 정기적으로 확인합니다. 데이터 손상이 감지되면 중복 데이터를 사용하여 복구됩니다. 또한 Azure Storage는 데이터를 저장하거나 검색할 때 데이터 패킷의 손상을 감지하기 위해 모든 네트워크 트래픽에 대한 체크섬을 계산합니다.
