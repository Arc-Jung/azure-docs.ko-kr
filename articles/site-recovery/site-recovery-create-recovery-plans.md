# <a name="create-a-recovery-plan-by-using-site-recovery"></a>Site Recovery를 사용하여 복구 계획 만들기

이 문서에서는 [Azure Site Recovery](site-recovery-overview.md)에서 복구 계획을 만들고 사용자 지정하는 방법을 설명합니다.

다음을 수행할 복구 계획을 만듭니다.

* 함께 장애 조치(failover)된 다음, 함께 시작되는 컴퓨터 그룹을 정의합니다.
* 복구 계획 그룹에서 함께 그룹화하여 컴퓨터 간의 종속성을 모델링합니다. 예를 들어 특정 응용 프로그램을 장애 조치(failover)하고 실행하려면 해당 응용 프로그램의 모든 VM을 동일한 복구 계획 그룹에 포함시킵니다.
* 장애 조치(Failover)를 실행합니다. 복구 계획에서 테스트를 실행하고 계획되거나 계획되지 않은 장애 조치를 실행할 수 있습니다.

## <a name="why-use-a-recovery-plan"></a>복구 계획을 사용하는 이유

복구 계획은 관리할 수 있는 독립적인 작은 단위를 만들어서 체계적인 복구 프로세스를 준비하는 데 도움이 됩니다. 단위는 일반적으로 환경 내 응용 프로그램을 나타냅니다. 복구 계획은 VM이 시작되는 순서를 정의하는 데 도움이 됩니다. 또한 복구 계획을 사용하여 복구 중 일반적인 작업을 자동화할 수 있습니다.

> [!TIP]
> 클라우드 마이그레이션 또는 재해 복구에 대한 준비가 되었는지 확인하는 한 가지 방법은 각각의 응용 프로그램이 복구 계획에 속하도록 하는 것입니다. 또한 각 복구 계획에서 Azure로 복구를 테스트해야 합니다. 이러한 준비를 통해 전체 데이터 센터를 Azure로 확실하게 마이그레이션하거나 장애 조치(failover)할 수 있습니다.
 
복구 계획에는 세 가지 핵심적인 가치 제안이 있습니다.

* 종속성을 캡처하도록 응용 프로그램을 모델링합니다.
* RTO를 줄이기 위해 대부분의 복구 작업을 자동화합니다.
* 재해에 대비하기 위해 장애 조치(failover)를 테스트합니다.

### <a name="model-an-application-to-capture-dependencies"></a>종속성을 캡처하도록 응용 프로그램 모델링

복구 계획은 일반적으로 응용 프로그램을 포함하고 장애 조치(failover)가 함께 수행되는 가상 머신의 그룹입니다. 복구 계획 구문을 사용하면 응용 프로그램별 속성을 캡처하도록 복구 계획 그룹을 향상시킬 수 있습니다. 

이 문서에서는 SQL 백 엔드, 미들웨어 및 웹 프런트 엔드를 포함할 수도 있는 일반적인 3계층 응용 프로그램을 사용합니다. 복구 계획을 사용자 지정하여 장애 조치 후에 가상 머신이 올바른 순서로 시작되도록 할 수 있습니다. SQL 백엔드가 먼저 시작되고, 다음으로 미들웨어가 시작되어야 하며, 웹 프런트 엔드가 마지막으로 시작되어야 합니다. 이러한 순서를 통해 마지막 가상 머신이 시작되면 응용 프로그램이 작동하도록 합니다. 

예를 들어 미들웨어가 시작되면 SQL 계층에 연결을 시도합니다. 복구 계획은 SQL 계층이 이미 실행되고 있도록 합니다. 프런트 엔드 서버가 마지막에 시작되도록 하면 모든 구성 요소가 시작되어 실행되고 응용 프로그램이 요청을 수락할 준비가 되기 전에 최종 사용자가 응용 프로그램 URL에 연결되지 않도록 할 수도 있습니다. 이러한 종속성을 구축하려면 복구 계획을 사용자 지정하고 그룹을 추가한 다음, 가상 머신을 선택합니다. 그룹 간에 가상 머신을 이동하려면 가상 머신의 그룹을 변경합니다.

![Site Recovery의 복구 계획 예제 스크린샷](./media/site-recovery-create-recovery-plans/rp.png)

사용자 지정을 완료한 후에는 정확한 복구 단계를 시각화할 수 있습니다. 다음은 복구 계획의 장애 조치 중 실행되는 단계의 순서입니다.

1. 종료 단계는 온-프레미스 가상 머신을 끄려고 합니다. (테스트 장애 조치(failover)를 제외하고, 그 동안 기본 사이트는 계속 실행되어야 합니다.)
2. 끄려는 시도는 복구 계획의 모든 가상 머신에 대한 장애 조치(failover)를 병렬로 트리거합니다. 장애 조치(failover) 단계는 복제된 데이터를 사용하여 가상 머신 디스크를 준비합니다.
3. 시작 그룹이 순서대로 실행되고 각 그룹에서 가상 머신이 시작됩니다. 먼저 그룹 1이 실행되고 그룹 2가 실행되고 마지막으로 그룹 3이 실행됩니다. 그룹에 가상 머신이 둘 이상 있으면(예: 부하가 분산된 웹 프런트 엔드) 모든 가상 머신이 병렬로 시작됩니다.

> [!TIP]
> 그룹 전체를 시퀀싱하면 다양한 응용 프로그램 계층 간에 종속성을 유지할 수 있습니다. 병렬 처리는 적절히 사용되면 응용 프로그램 복구의 RTO를 향상시킵니다.

   > [!NOTE]
   > 단일 그룹에 속하는 컴퓨터는 병렬로 장애 조치(failover)됩니다. 서로 다른 그룹에 속하는 컴퓨터는 그룹 순서대로 장애 조치(failover)됩니다. 그룹 2의 컴퓨터는 그룹 1의 모든 컴퓨터가 장애 조치(failover)되고 시작된 후에야 장애 조치를 시작합니다.

### <a name="automate-most-recovery-tasks-to-reduce-rto"></a>RTO를 줄이기 위해 대부분의 복구 작업 자동화

대규모 응용 프로그램 복구는 복잡한 작업일 수 있습니다. 혼란스러운 시기에 기억해야 할 수작업 단계가 많으면 작업이 어렵고 프로세스에 오류가 발생하기 쉽습니다. 또한 실제로 장애 조치(failover)를 트리거하는 응용 프로그램의 복잡성을 인식하지 못하는 사람이 작업을 수행할 수도 있습니다. 

복구 계획을 사용하면 각 단계에서 수행해야 하는 필수 작업을 자동화할 수 있습니다. 필요한 작업은 Azure Automation Runbook을 사용하여 설정할 수 있습니다. Runbook를 사용하면 다음 예제의 작업과 같은 일반적인 복구 작업을 자동화할 수 있습니다. 자동화할 수 없는 작업의 경우 복구 계획에 수동 작업을 삽입할 수 있습니다.

* **장애 조치(failover) 후 Azure 가상 머신에 대한 작업**: 이러한 작업은 일반적으로 가상 머신에 연결이 가능해야 합니다. 예제:
    * 장애 조치(failover) 후 가상 머신에 공용 IP 주소를 만듭니다.
    * 장애 조치(failover) 후 가상 머신의 NIC에 네트워크 보안 그룹을 할당합니다.
    * 가용성 집합에 부하 분산 장치를 추가합니다.
* **장애 조치 후 가상 머신 내 작업**: 이러한 작업은 새로운 환경에서 응용 프로그램이 올바르게 작동하도록 응용 프로그램을 재구성합니다. 예제:
    * 가상 머신 내 데이터베이스 연결 문자열을 수정합니다.
    * 웹 서버 구성 또는 규칙을 변경합니다.

> [!TIP]
> Automation Runbook을 사용하여 복구 후 작업을 자동화하는 완벽한 복구 계획을 만들어서 한 번의 클릭으로 장애 조치(failover)를 수행하고 RTO를 최적화하십시오.

### <a name="test-failover-to-be-ready-for-a-disaster"></a>재해에 대비하기 위한 테스트 장애 조치(failover)

복구 계획을 사용하여 장애 조치 또는 테스트 장애 조치를 트리거할 수 있습니다. 장애 조치를 수행하기 전에 항상 응용 프로그램에서 테스트 장애 조치를 완료하십시오. 테스트 장애 조치는 응용 프로그램이 복구 사이트에서 작동되는지 여부를 확인하는 데 도움이 됩니다. 놓친 부분이 있으면 쉽게 정리를 트리거한 다음, 테스트 장애 조치를 다시 실행할 수 있습니다. 응용 프로그램이 원활하게 복구된다는 확신이 들 때까지 테스트 장애 조치를 여러 번 수행하십시오.

![Site Recovery의 테스트 복구 계획 예제 스크린샷](./media/site-recovery-create-recovery-plans/rptest.png)

> [!TIP]
> 각 응용 프로그램은 고유하기 때문에 각 응용 프로그램에 맞게 사용자 지정된 복구 계획을 구축해야 합니다. 
>
> 또한 오늘날의 동적인 데이터 센터 중심 환경에서 응용 프로그램과 그 종속성은 자주 변경됩니다. 복구 계획이 최신인지 확인하려면 분기마다 응용 프로그램에 대한 장애 조치 테스트를 수행하십시오.

## <a name="create-a-recovery-plan"></a>복구 계획 만들기

1. **복구 계획** > **복구 계획 만들기**를 선택합니다.
   복구 계획 이름, 원본 및 대상을 지정합니다. 원본 위치에는 장애 조치(Failover) 및 복구를 사용하도록 설정한 가상 머신이 있어야 합니다. 복구 계획에 포함시킬 가상 머신을 기반으로 원본 및 대상을 선택합니다. 

   |시나리오                   |원본               |대상           |
   |---------------------------|---------------------|-----------------|
   |Azure 간             |Azure 지역         |Azure 지역     |
   |VMware에서 Azure로            |구성 서버 |Azure            |
   |VMM(Virtual Machine Manager)-Azure               |VMM 표시 이름    |Azure            |
   |Azure에 Hyper-V 사이트      |Hyper-v 사이트 이름    |Azure            |
   |물리적 컴퓨터에서 Azure로 |구성 서버 |Azure            |
   |VMM에서 VMM으로                 |VMM 식별 이름    |VMM 표시 이름|

   > [!NOTE]
   > 복구 계획에는 원본 및 대상이 동일한 가상 머신이 포함될 수 있습니다. VMware 및 System Center VMM(Virtual Machine Manager) 가상 머신은 동일한 복구 계획에 있을 수 없습니다. 하지만 VMware 가상 머신과 물리적 컴퓨터는 동일한 복구 계획에 추가할 수 있습니다. 이 경우 두 머신 모두에 대한 원본은 구성 서버입니다.

2. **가상 머신 선택**에서 복구 계획 내 기본 그룹(그룹 1)에 추가하고자 하는 가상 머신(또는 복제 그룹)을 선택합니다. 원본에서 보호되고(복구 계획에 선택되어) 대상에서 보호되는(복구 계획에 선택되어) 가상 머신만 선택할 수 있습니다.

## <a name="customize-and-extend-recovery-plans"></a>복구 계획 사용자 지정 및 확장

복구 계획을 사용자 지정하고 확장하려면 Site Recovery 복구 계획 리소스 창으로 이동합니다. **사용자 지정** 탭을 선택합니다. 다음 옵션을 사용하여 복구 계획을 사용자 지정하고 확장할 수 있습니다.

- **새 그룹 추가**: 기본 그룹에 최대 7개의 추가 복구 계획 그룹을 추가할 수 있습니다. 그런 다음, 해당 복구 계획 그룹에 컴퓨터나 복제 그룹을 추가할 수 있습니다. 그룹은 추가한 순서대로 번호가 지정됩니다. 가상 머신이나 복제 그룹은 1개의 복구 계획 그룹에만 포함할 수 있습니다.
- **수동 작업 추가**: 복구 계획 그룹의 앞 또는 뒤에 실행되는 수동 작업을 추가할 수 있습니다. 복구 계획을 실행하는 경우 수동 작업을 삽입하는 지점에서 중지됩니다. 대화 상자는 수동 작업이 완료되도록 지정하라는 메시지를 표시합니다.
- **스크립트 추가**: 복구 계획 그룹 앞 또는 뒤에 실행되는 스크립트를 추가할 수 있습니다. 스크립트를 추가하면 해당 그룹에 대해 새로운 작업 집합이 추가됩니다. 예를 들어, 그룹 1에 대한 사전 단계 집합이 *Group 1: pre-steps*라는 이름으로 생성됩니다. 모든 사전 단계가 이 집합 내에 나열됩니다. VMM 서버가 배포된 경우에만 주 사이트에 스크립트를 추가할 수 있습니다. 자세한 내용은 [복구 계획에 VMM 스크립트 추가](site-recovery-how-to-add-vmmscript.md)를 참조하세요.
- **Azure Runbook 추가**: Azure Runbook을 사용하여 복구 계획을 확장할 수 있습니다. 예를 들어, Runbook을 사용하여 작업을 자동화하거나 단일 단계 복구를 만들 수 있습니다. 자세한 내용은 [복구 계획에 Azure Automation Runbook 추가](site-recovery-runbook-automation.md)를 참조하세요.


## <a name="add-a-script-runbook-or-manual-action-to-a-plan"></a>계획에 스크립트, Runbook 또는 수동 작업 추가

기본 복구 계획 그룹에 가상 머신이나 복제 그룹을 추가한 후에 복구 계획 그룹에 스크립트나 수동 작업을 추가할 수 있습니다.

1. 복구 계획을 엽니다.
2. **단계** 목록에서 항목을 하나 선택합니다. 그런 다음, **스크립트** 또는 **수동 작업**을 선택합니다.
3. 스크립트나 작업을 선택한 항목의 앞 또는 뒤에 추가할 것인지 지정합니다. 스크립트 위치를 위 또는 아래로 이동하려면 **위로 이동** 및 **아래로 이동** 단추를 선택합니다.
4. VMM 스크립트를 추가하면 **VMM 스크립트에 대한 장애 조치**를 선택합니다. **스크립트 경로**에 공유에 대한 상대 경로를 입력합니다. 예: **\RPScripts\RPScript.PS1**.
5. Automation Runbook을 추가하는 경우 Runbook이 있는 Automation 계정을 지정합니다. 그런 다음, Azure Runbook 스크립트를 선택합니다.
6. 스크립트가 예상대로 작동하는지 확인하려면, 복구 계획의 장애 조치를 실행합니다.

스크립트나 Runbook 옵션은 장애 조치(failover) 또는 장애 복구(failback) 중 다음 시나리오에서만 사용할 수 있습니다. 장애 조치(failover)와 장애 복구(failback) 모두에서 수동 작업이 가능합니다.


|시나리오               |장애 조치(failover) |장애 복구 |
|-----------------------|---------|---------|
|Azure 간         |Runbook |Runbook  |
|VMware에서 Azure로        |Runbook |해당 없음       | 
|VMM에서 Azure로           |Runbook |스크립트   |
|Azure에 Hyper-V 사이트  |Runbook |해당 없음       |
|VMM에서 VMM으로             |스크립트   |스크립트   |
