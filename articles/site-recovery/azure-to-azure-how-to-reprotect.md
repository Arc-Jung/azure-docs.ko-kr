# <a name="reprotect-failed-over-azure-vms-to-the-primary-region"></a>주 지역으로 장애 조치(failover)된 Azure VM 다시 보호

Azure [사이트 복구를](site-recovery-overview.md)사용하여 한 리전에서 다른 리전으로 [장애 조치하면](site-recovery-failover.md) **보호되지 않은** 상태에서 보조 지역에서 VM이 부팅됩니다. VM을 기본 지역으로 장애 조치하려면 다음 작업을 수행하십시오.

1. VM이 주 지역으로 복제를 시작하도록 보조 지역의 VM을 다시 보호합니다.
1. 다시 보호가 완료되고 VM이 복제되면 보조 지역에서 기본 지역으로 장애 조치할 수 있습니다.

## <a name="prerequisites"></a>사전 요구 사항

- 주 지역에서 보조 지역으로 VM 장애 조치(failover)를 커밋해야 합니다.
- 주 대상 사이트가 사용 가능하고 해당 지역에서 리소스를 만들거나 액세스할 수 있어야 합니다.

## <a name="reprotect-a-vm"></a>VM 다시 보호

1. **Vault** > **복제 항목에서**VM에서 실패한 항목을 마우스 오른쪽 단추로 클릭하고 **다시 보호를**선택합니다. 다시 보호 방향이 보조에서 주로 표시됩니다.

   ![다시 보호](./media/site-recovery-how-to-reprotect-azure-to-azure/reprotect.png)

1. 리소스 그룹, 네트워크, 스토리지 및 가용성 집합을 검토합니다. 그런 다음 **확인**을 클릭합니다. 새 리소스로 표시된 리소스가 있으면 다시 보호 프로세스의 일부로 만들어집니다.
1. 다시 보호 작업은 최신 데이터로 대상 사이트를 시드합니다. 작업이 완료되면 델타 복제가 수행됩니다. 그런 다음 주 사이트로 다시 장애 조치(failover)할 수 있습니다. 사용자 지정 옵션을 사용하여 다시 보호 중에 사용할 네트워크나 스토리지 계정을 선택할 수 있습니다.

   ![옵션 사용자 지정](./media/site-recovery-how-to-reprotect-azure-to-azure/customize.png)

### <a name="customize-reprotect-settings"></a>다시 보호 설정 사용자 지정

다시 보호하는 동안 대상 VM의 다음 속성을 사용자 지정할 수 있습니다.

![사용자 지정](./media/site-recovery-how-to-reprotect-azure-to-azure/customizeblade.png)

|속성 |메모  |
|---------|---------|
|대상 리소스 그룹 | VM이 만들어진 대상 리소스 그룹을 수정합니다. 다시 보호의 일부로 대상 VM이 삭제됩니다. 장애 조치(failover) 후 VM을 만들 새 리소스 그룹을 선택할 수 있습니다. |
|대상 가상 네트워크 | 다시 보호 작업 동안 대상 네트워크를 변경할 수 없습니다. 네트워크를 변경하려면 네트워크 매핑을 다시 실행합니다. |
|대상 저장소(보조 VM은 관리 디스크를 사용하지 않습니다) | 장애 조치(failover) 후 VM이 사용하는 스토리지 계정을 변경할 수 있습니다. |
|복제본 관리 디스크(보조 VM이 관리 디스크를 사용함) | Site Recovery는 보조 VM의 관리 디스크를 미러링하는 주 지역에서 관리 디스크의 복제본을 만듭니다. |
|캐시 스토리지 | 복제하는 동안 사용할 캐시 스토리지 계정을 지정할 수 있습니다. 기본적으로 새 캐시 저장소 계정이 없는 경우 만들어집니다. |
|가용성 집합 | 보조 지역의 VM이 가용성 집합의 일부인 경우 주 지역의 대상 VM에 대한 가용성 집합을 선택할 수 있습니다. 기본적으로 Site Recovery는 주 지역에서 기존 가용성 집합을 찾고 사용합니다. 사용자 지정하는 동안 새 가용성 집합을 지정할 수 있습니다. |

### <a name="what-happens-during-reprotection"></a>다시 보호하는 동안 어떻게 되나요?

기본적으로 다음과 같은 현상이 발생합니다.

1. 장애 조치된 VM이 실행 중인 지역에 캐시 스토리지 계정이 만들어집니다.
1. 대상 스토리지 계정(주 지역의 원래 스토리지 계정)이 없는 경우 새로 만들어집니다. 할당된 저장소 계정 이름은 보조 VM에서 사용하는 저장소 계정의 이름입니다. `asr`
1. VM에서 관리 디스크를 사용하는 경우 복제본 관리 디스크는 보조 VM의 디스크에서 복제된 데이터를 저장하기 위해 주 지역에 생성됩니다.
1. 대상 가용성 집합이 없는 경우 필요한 경우 다시 보호 작업의 일부로 새 새 가 만들어집니다. 재보호 설정을 사용자 지정한 경우 선택한 집합이 사용됩니다.

다시 보호 작업을 트리거하고 대상 VM이 있는 경우 다음이 발생합니다.

1. 대상 쪽 VM은 실행 중인 경우 꺼집니다.
1. VM이 관리 디스크를 사용하는 경우 원래 디스크의 복사본이 `-ASRReplica` 접미사로 만들어집니다. 원래 디스크가 삭제됩니다. 복사본은 `-ASRReplica` 복제에 사용됩니다.
1. VM이 관리되지 않는 디스크를 사용할 경우 대상 VM의 데이터 디스크가 분리되어 복제에 사용됩니다. OS 디스크의 복사본이 만들어지고 VM에서 연결됩니다. 원래의 OS 디스크가 분리되며 복제에 사용됩니다.
1. 원본 디스크와 대상 디스크 간의 변경 내용만 동기화됩니다. 차등 백업은 디스크를 모두 비교하여 계산된 다음, 전달됩니다. 아래에서 재보호를 완료할 예상 시간을 확인하십시오.
1. 동기화가 완료되면 델타 복제가 시작되고 복제 정책에 따라 복구 지점이 만들어집니다.

다시 보호 작업을 트리거하고 대상 VM 및 디스크가 존재하지 않는 경우 다음과 같은 작업이 발생합니다.

1. VM에서 관리디스크를 사용하는 경우 복제본 디스크가 접미사로 `-ASRReplica` 만들어집니다. 복사본은 `-ASRReplica` 복제에 사용됩니다.
1. VM이 관리되지 않는 디스크를 사용하는 경우 복제 디스크는 대상 스토리지 계정에 만들어집니다.
1. 전체 디스크가 장애 조치된 지역에서 새 대상 지역에 복사됩니다.
1. 동기화가 완료되면 델타 복제가 시작되고 복제 정책에 따라 복구 지점이 만들어집니다.

#### <a name="estimated-time-to-do-the-reprotection"></a>재보호를 수행하는 예상 시간

대부분의 경우 Azure 사이트 복구는 전체 데이터를 원본 리전에 복제하지 않습니다.
다음 조건에 따라 복제되는 데이터의 양을 결정합니다.

1. 리소스 그룹 변경/삭제와 같은 이유로 원본 VM 데이터가 삭제, 손상 또는 액세스할 수 없는 경우 다시 보호하는 동안 원본 지역에서 사용할 수 있는 데이터가 없기 때문에 전체 초기 복제가 발생합니다.
1. 원본 VM 데이터에 액세스할 수 있는 경우 두 디스크를 비교한 다음 전송하여 차동만 계산됩니다. 예상 시간을 얻으려면 아래 표를 확인하십시오.

|예시 상황 | 재보호에 걸린 시간 |
|---|---|
|소스 영역에는 1TB 표준 디스크가 있는 1VM이 있습니다.<br/>127GB 데이터만 사용되고 나머지 디스크는 비어 있습니다.<br/>디스크 유형은 60MiB/S 처리량이 있는 표준입니다.<br/>장애 조치 후 데이터가 변경되지 않습니다.| 대략적인 시간: 45 분 - 1.5 시간.<br/>다시 보호하는 동안 사이트 복구는 약 45분 동안 127GB/45GB가 소요되는 모든 데이터의 체크본을 채웁니다.<br/>사이트 복구가 자동 크기 조정에 약 20-30분 동안 필요한 경우 약간의 오버헤드 시간이 필요합니다.<br/>송신 요금은 없습니다. |
|소스 영역에는 1TB 표준 디스크가 있는 1VM이 있습니다.<br/>127GB 데이터만 사용되고 나머지 디스크는 비어 있습니다.<br/>디스크 유형은 60MiB/S 처리량이 있는 표준입니다.<br/>장애 조치 후 45GB 데이터가 변경됩니다.| 대략적인 시간: 1 시간 - 2 시간.<br/>다시 보호하는 동안 사이트 복구는 약 45분 동안 127GB/45GB가 소요되는 모든 데이터의 체크본을 채웁니다.<br/>전송 시간은 45GB/ 45MBps, 약 17분인 45GB의 변경 사항을 적용합니다.<br/>송신 요금은 체크섬이 아닌 45GB 데이터 변경에 대한 요금입니다. |

VM이 기본 지역으로 다시 장애토된 후 다시 보호되면(즉, VM이 기본 리전에서 DR 영역으로 다시 보호되는 경우) 대상 VM 및 관련 NIC가 삭제됩니다.

VM이 DR 영역에서 기본 영역으로 다시 보호되는 경우 어설프게 된 기본 VM 및 관련 NIC를 삭제하지 않습니다.

## <a name="next-steps"></a>다음 단계

VM을 보호한 후 장애 조치(failover)를 시작할 수 있습니다. 장애 조치(failover)는 보조 지역의 VM을 종료하고 이 프로세스 중에 짧은 가동 중지 시간으로 기본 영역에서 VM을 만들고 부팅합니다. 이 프로세스에 적합한 시간을 선택하고 기본 사이트에 대한 전체 장애 조치(failover)를 시작하기 전에 테스트 장애 조치(failover)를 실행하는 것이 좋습니다. Azure 사이트 복구 장애 조치(failover)에 대해 [자세히 알아보세요.](site-recovery-failover.md)
