# <a name="about-on-premises-disaster-recovery-failoverfailback"></a>온-프레미스 재해 복구 장애 조치/장애 복구 에 대해

이 문서에서는 Azure [사이트](site-recovery-overview.md)복구를 사용하여 온-프레미스 컴퓨터를 Azure에 복구하는 동안 장애 조치 및 장애 복구에 대한 개요를 제공합니다.

## <a name="recovery-stages"></a>복구 단계

사이트 복구에서 장애 조치 및 장애 복구에는 다음 네 단계가 있습니다.

- **1단계: 온-프레미스에서 장애 조치**: 온-프레미스 시스템에 대한 복제를 Azure에 설정한 후 온-프레미스 사이트가 다운되면 해당 컴퓨터를 Azure로 장애 조치합니다. 장애 조치(Failover) 후 복제된 데이터에서 Azure VM이 만들어집니다.
- **2단계: Azure VM 다시 보호**: Azure에서 Azure VM을 다시 보호하여 온-프레미스 사이트로 복제하기 시작합니다. 데이터 일관성을 보장하기 위해 재보호 중에 온-프레미스 VM(사용 가능한 경우)이 꺼져 있습니다.
- **3단계: Azure에서 장애 조치**: 온-프레미스 사이트가 정상으로 다시 실행되는 경우 다른 장애 조치(failover)를 실행하여 이번에는 Azure VM을 온-프레미스 사이트로 장애 복구합니다. 실패한 원래 위치 또는 대체 위치로 장애 조치할 수 있습니다.
- **4단계: 온-프레미스 컴퓨터 다시 보호**: 실패한 후 온-프레미스 컴퓨터를 Azure로 다시 복제할 수 있습니다.

## <a name="failover"></a>장애 조치

비즈니스 연속성 및 BCDR(재해 복구) 전략의 일부로 장애 조치(failover)를 수행합니다.

- BCDR 전략의 첫 번째 단계로 온-프레미스 컴퓨터를 지속적으로 Azure에 복제합니다. 사용자는 온-프레미스 소스 컴퓨터에서 실행되는 워크로드 및 앱에 액세스합니다.
- 예를 들어 온-프레미스에 가동 중단이 있는 경우 와 같은 필요가 발생하는 경우 복제 컴퓨터를 Azure로 장애 조치합니다. Azure VM은 복제된 데이터를 사용하여 만들어집니다.
- 비즈니스 연속성을 위해 사용자는 Azure VM에서 앱에 계속 액세스할 수 있습니다.

장애 조치(Failover)는 2단계 활동입니다.

- **장애 조치**: 선택한 복구 지점을 사용하여 Azure VM을 만들고 가져오는 장애 조치입니다.
- **커밋**: 장애 조치 후 Azure에서 VM을 확인합니다.
    - 그런 다음 장애 조치(failover)를 선택한 복구 지점에 커밋하거나 커밋에 대해 다른 지점을 선택할 수 있습니다.
    - 장애 조치(failover)를 커밋한 후에는 복구 지점을 변경할 수 없습니다.

## <a name="failover-processing"></a>장애 조치 처리

일부 시나리오에서는 장애 조치(failover)를 위해서는 추가 처리가 필요하며 이러한 작업을 완료하는 데는 약 8~10분이 소요됩니다. 다음에 대한 테스트 장애 조치(failover) 시간이 길어질 수 있습니다.

* 9.8 이전의 모바일 서비스 버전을 실행하는 VMware VM
* 실제 서버
* VMware Linux VM
* 물리적 서버로 보호되는 Hyper-V VM
* DHCP 서비스를 사용하도록 설정되지 않은 VMware VM
* 다음 부팅 드라이버가 없는 VMware VM: storvsc, vmbus, storflt, intelide, atapi.

## <a name="recovery-point-options"></a>복구 지점 옵션

장애 조치 중에 여러 복구 지점 옵션을 선택할 수 있습니다.

**옵션** | **세부 정보**
--- | ---
**최신(최저 RPO)** | 이 옵션은 가장 낮은 복구 지점 목표(RPO)를 제공합니다. 먼저 사이트 복구 서비스로 전송된 모든 데이터를 처리하여 각 VM에 대한 복구 지점을 만든 다음 장애처리합니다. 이 복구 지점에는 장애 조치(failover)가 트리거될 때 사이트 복구에 복제된 모든 데이터가 있습니다.
**최신 처리**  | 이 옵션은 VM에서 사이트 복구에서 처리된 최신 복구 지점으로 실패합니다. 특정 VM에 대한 최신 복구 지점을 보려면 VM 설정에서 **최신 복구 지점을** 확인합니다. 이 옵션은 처리되지 않은 데이터를 처리하는 데 시간이 투입되지 않으므로 낮은 RTO(복구 시간 목표)를 제공합니다.
**최신 앱 일관성** |  이 옵션은 앱 일관된 복구 지점이 활성화된 경우 사이트 복구에서 처리되는 최신 응용 프로그램 일관된 복구 지점으로 VM을 통해 실패합니다. VM 설정의 최신 복구 지점을 확인합니다.
**최신 다중 VM 처리** | 이 옵션은 다중 VM 일치가 사용하도록 설정된 하나 이상의 VM이 포함된 복구 계획에 사용할 수 있습니다. 이 설정이 사용하도록 설정된 VM은 최신 일반 다중 VM 일치 복구 지점으로 장애 조치(Failover)됩니다. 계획의 다른 VM은 최신 처리된 복구 지점으로 장애 조치됩니다.
**최신 멀티 VM 앱 일관성** |  이 옵션은 다중 VM 일치가 사용하도록 설정된 하나 이상의 VM이 포함된 복구 계획에 사용할 수 있습니다. 복제 그룹에 속하는 VM은 공통된 최신 다중 VM 애플리케이션 일관성 복구 지점으로 장애 조치(failover)됩니다. 다른 VM은 최신 애플리케이션 일치 복구 지점으로 장애 조치(Failover)됩니다.
**주문** | 이 옵션을 사용하여 특정 VM을 특정 복구 시점까지 장애 조치합니다. 복구 계획에는 이 옵션을 사용할 수 없습니다.

> [!NOTE]
> 복구 지점은 다른 복구 서비스 자격 증명 모음으로 마이그레이션할 수 없습니다.

## <a name="reprotectionfailback"></a>재보호/장애 조치

Azure로 장애 조치(failover) 후 복제된 Azure VM은 보호되지 않은 상태입니다.

- 온-프레미스 사이트로 다시 장애 복구하는 첫 번째 단계로 온-프레미스로 복제하는 Azure VM을 시작해야 합니다. 재보호 프로세스는 실패한 컴퓨터 유형에 따라 다릅니다.
- 컴퓨터가 Azure에서 온-프레미스로 복제된 후 Azure에서 온-프레미스 사이트로 장애 조치(failover)를 실행할 수 있습니다.
- 컴퓨터가 온-프레미스에서 다시 실행된 후 재해 복구를 위해 복제를 Azure에 복제하도록 복제를 활성화할 수 있습니다.

장애 조치(Failback)는 다음과 같이 작동합니다.

- 장애 복구하려면 VM에 장애 복구하려면 하나 이상의 복구 지점이 필요합니다. 복구 계획에서 계획의 모든 VM에는 하나 이상의 복구 지점이 필요합니다.
- **최신** 복구 지점을 사용하여 장애 복구를 사용하는 것이 좋습니다(충돌 일관된 지점).
    - 앱일관된 복구 지점 옵션이 있습니다. 이 경우 단일 VM은 사용 가능한 최신 앱 일관된 복구 지점으로 복구됩니다. 복제 그룹이 있는 복구 계획의 경우 각 복제 그룹은 사용 가능한 공통 복구 지점으로 복구됩니다.
    - 앱 일관된 복구 지점은 시간에 뒤처질 수 있으며 데이터가 손실될 수 있습니다.
- Azure에서 온-프레미스 사이트로 장애 조치하는 동안 사이트 복구는 Azure VM을 종료합니다. 장애 조치(failover)를 커밋하면 사이트 복구는 Azure에서 실패한 다시 Azure VM을 제거합니다.


## <a name="vmwarephysical-reprotectionfailback"></a>VMware/물리적 재보호/장애 조치

VMware 컴퓨터와 물리적 서버를 Azure에서 온-프레미스로 다시 보호하고 장애 복구하려면 장애 조치 인프라가 필요하며 여러 가지 요구 사항이 있습니다.

- **Azure의 임시 프로세스 서버**: Azure에서 장애 복구하려면 Azure에서 복제를 처리하는 프로세스 서버로 작동하도록 Azure VM을 설정합니다. 장애 복구를 완료한 후 이 VM을 삭제할 수 있습니다.
- **VPN 연결**: 장애 복구하려면 Azure 네트워크에서 온-프레미스 사이트로의 VPN 연결(또는 ExpressRoute)이 필요합니다.
- **별도의 마스터 대상 서버**: 기본적으로 구성 서버와 함께 온-프레미스 VMware VM에 설치된 마스터 대상 서버에서 장애 복구를 처리합니다. 대량 트래픽을 장애 복구해야 하는 경우 별도의 온-프레미스 마스터 대상 서버를 이 용도로 설정합니다.
- **장애 복구 정책**: 온-프레미스 사이트에 다시 복제하려면 장애 복구 정책이 필요합니다. 이 정책은 온-프레미스에서 Azure로의 복제 정책을 만들 때 자동으로 만들어졌습니다.
    - 이 정책은 구성 서버와 자동으로 연결됩니다.
    - 이 정책을 편집할 수 없습니다.
    - 정책 값: RPO 임계값 - 15분; 복구 지점 보존 - 24 시간; 앱 일관된 스냅샷 빈도 - 60분.

VMware/물리적 재보호 및 장애 조치(failback)에 대해 자세히 알아보십시오.
- 다시 보호 및 장애 조치(failback)에 대한 추가 요구 사항을 [검토합니다.](vmware-azure-reprotect.md#before-you-begin)
- Azure에서 프로세스 서버를 [배포합니다.](vmware-azure-prepare-failback.md#deploy-a-process-server-in-azure)
- 별도의 마스터 대상 서버를 [배포합니다.](vmware-azure-prepare-failback.md#deploy-a-separate-master-target-server)

Azure VM을 온-프레미스로 다시 보호하는 경우 원래 위치 또는 대체 위치로 장애 조치할 지 않도록 지정할 수 있습니다.

- **원래 위치 복구**: 이 가 있는 경우 동일한 소스 온-프레미스 컴퓨터에 Azure에서 다시 실패 합니다. 이 시나리오에서는 변경 내용만 온-프레미스로 다시 복제됩니다.
- **대체 위치 복구**: 온-프레미스 컴퓨터가 없는 경우 Azure에서 대체 위치로 장애 조치할 수 있습니다. Azure VM을 온-프레미스로 다시 보호하면 온-프레미스 컴퓨터가 만들어집니다. 전체 데이터 복제는 Azure에서 온-프레미스로 수행됩니다. - 위치 장애 조치의 요구 사항 및 제한 사항을 [검토합니다.](concepts-types-of-failback.md)



## <a name="hyper-v-reprotectionfailback"></a>하이퍼 V 재보호/장애 조치

Azure에서 온-프레미스로 Hyper-V VM을 다시 보호하고 장애 조치하려면 다음을 수행하십시오.

- 저장소 계정을 사용하여 복제하는 Hyper-V VM만 장애 조치할 수 있습니다. 관리되는 디스크를 사용하여 복제하는 Hyper-V VM의 장애 복구는 지원되지 않습니다.
- 온-프레미스 하이퍼-V 호스트(또는 사용되는 경우 시스템 센터 VMM)는 Azure에 연결해야 합니다.
- Azure에서 온-프레미스로 계획된 장애 복구를 실행합니다.
- 하이퍼 VVM 장애 조치(Hyper-VM VM) 장애 에 대해 특정 구성 요소를 설정할 필요가 없습니다.
- 계획된 장애 조치 중에 장애 조치 전에 데이터를 동기화하는 옵션을 선택할 수 있습니다.
    - **장애 조치 전에 데이터 동기화**: 이 옵션은 컴퓨터를 종료하지 않고 동기화할 때 가상 시스템의 가동 중지 시간을 최소화합니다.
        - 1단계: Azure VM의 스냅숏을 만들고 온-프레미스 Hyper-V 호스트에 복사합니다. 컴퓨터가 Azure에서 계속 실행됩니다.
        - 2단계: Azure VM을 종료하여 새 변경 사항이 발생하지 않도록 합니다. 최종 델타 변경 집합은 온-프레미스 서버로 전송되고 온-프레미스 VM이 시작됩니다.
    - **장애 조치 중에만 데이터 동기화**: 대부분의 디스크가 변경되어 체크섬 계산을 수행하지 않으므로 이 옵션이 더 빠릅니다. 디스크의 다운로드를 수행합니다. VM이 한동안 Azure에서 실행중이거나 온-프레미스 VM이 삭제된 경우 이 옵션을 사용하는 것이 좋습니다.

하이퍼-V 재보호 및 장애 조치(failback)에 대해 [자세히 알아보십시오.](hyper-v-azure-failback.md)

Azure VM을 온-프레미스로 다시 보호하는 경우 원래 위치 또는 대체 위치로 장애 조치할 지 않도록 지정할 수 있습니다.

- **원래 위치 복구**: 이 가 있는 경우 동일한 소스 온-프레미스 컴퓨터에 Azure에서 다시 실패 합니다. 이 시나리오에서는 이전 절차에 설명 된 동기화 옵션 중 하나를 선택 합니다.
- **대체 위치 복구**: 온-프레미스 컴퓨터가 없는 경우 Azure에서 대체 위치로 장애 조치할 수 있습니다. Azure VM을 온-프레미스로 다시 보호하면 온-프레미스 컴퓨터가 만들어집니다. 이 옵션을 사용하면 장애 조치 전에 데이터를 동기화할 수 있는 옵션을 선택하는 것이 좋습니다.
- 위치 장애 조치의 요구 사항 및 제한 사항을 [검토합니다.](hyper-v-azure-failback.md)


온-프레미스 사이트로 다시 실패한 후 **역복제를** 사용하여 VM을 Azure로 복제하고 주기를 완료하도록 합니다.
