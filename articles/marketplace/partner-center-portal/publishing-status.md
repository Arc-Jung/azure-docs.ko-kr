# <a name="check-the-publishing-status-of-your-commercial-marketplace-offer"></a>상업적 Marketplace 제안의 게시 상태를 확인 합니다.

파트너 센터의 [상용 Marketplace 포털](https://partner.microsoft.com/dashboard/commercial-marketplace/offers) 의 **제품 개요** 탭에서 현재 **게시 상태** 를 볼 수 있습니다.

각 제품에 대해 다음 상태 표시기 중 하나가 표시되어야 합니다.

| **Status**    | **설명**  |
| :---------- | :-------------------|
| **초안** | 제안이 만들어졌지만 게시 되 고 있지 않습니다. |
| **게시 진행 중** | 제안/계획은 게시 프로세스의 단계를 통해 작동 합니다. |
| **주의가 필요함** | Microsoft 또는 게시 단계에서 인증 하는 동안 중요 한 문제가 발견 되었습니다. |
| **미리 보기** | 제품은 Microsoft에서 인증 되었으며 이제 게시자에의 한 최종 확인을 기다립니다 합니다. 라이브 전환을 선택 하 여 제품을 라이브 상태로 만듭니다. |
| **라이브** | 제품은 marketplace에 살고 있으며 고객이 확인 하 고 얻을 수 있습니다. |
| **보류 중인 판매 중지** | 게시자가 제품 또는 계획에서 "판매 중지"를 선택 했지만 작업이 아직 완료 되지 않았습니다. |
| **Marketplace에서 사용할 수 없음** | Marketplace에서 이전에 게시 된 제품/계획이 제거 되었습니다. |

## <a name="automated-validation"></a>자동 유효성 검사

게시 프로세스의 첫 번째 단계는 자동 유효성 검사 집합입니다. 각 유효성 검사 단계는 제품을 만들 때 사용 하도록 선택한 기능에 해당 합니다. 해당 기능을 사용 하도록 설정 하지 않은 경우 유효성 검사는 다음 게시 단계로 건너뜁니다. 게시 상태가 승인 되기 전에 각 유효성 검사를 완료 해야 합니다.

- **구매 흐름 설정 제공 (<10 분)**

이 단계에서는 Azure Portal를 통해 고객이 구매할 때 제품이 충족 될 수 있도록 합니다. 이 단계는 Microsoft를 통해 판매 되는 제품에만 적용 됩니다.

- **테스트 드라이브 데이터 유효성 검사 (~ 5 분)**

이 단계에서는 제품의 test drive 기술 구성 섹션에서 제공한 데이터의 유효성을 검사 합니다. 시험 구동 기능을 테스트 하 고 승인 합니다. 이 단계는 테스트 드라이브를 사용 하도록 설정한 경우에만 적용할 수 있습니다.

- **테스트 드라이브 프로 비전 (~ 30 분)**

이 단계에서는 이전 단계에서 테스트 드라이브의 데이터 및 기능에 대 한 유효성을 검사 한 후 고객이 사용할 수 있도록 테스트 드라이브의 인스턴스를 배포 하 고 복제 합니다.  이 단계는 테스트 드라이브를 사용 하도록 설정한 경우에만 적용할 수 있습니다.

- **리드 관리 유효성 검사 및 등록 (15 분 <)**

이 단계에서는 고객 관리 시스템에서 제품 설치에 제공 된 세부 정보에 따라 고객 리드를 받을 수 있음을 확인 합니다. 이 단계는 잠재 고객 관리를 사용 하도록 설정한 경우에만 적용 됩니다.

## <a name="certification"></a>인증

게시 하기 전에 파트너 센터에서 상업적 Marketplace에 제출 된 제품은 인증을 받아야 합니다. 제출 된 제품은 [Azure Marketplace 인증 정책](https://docs.microsoft.com/legal/marketplace/general-policies)에 대 한 검사를 비롯 하 여 엄격한 테스트, 일부 자동화 및 기타 설명서를 제공 합니다. 게시 흐름의 다음 단계로 진행 하기 전에 제품 제출을 인증 적격으로 표시 해야 합니다.

### <a name="types-of-validation-that-take-place-during-certification"></a>인증 하는 동안 발생 하는 유효성 검사 유형

제출 된 각 제품에 대 한 인증 프로세스에는 세 가지 수준의 유효성 검사가 포함 됩니다.

- 게시자 비즈니스 자격
- 콘텐츠 유효성 검사
- 기술 유효성 검사

#### <a name="publisher-business-eligibility"></a>게시자 비즈니스 자격

각 제품 유형은 게시자가 충족 해야 하는 기본 자격 조건 집합을 검사 합니다. 자격 조건에는 게시자의 MPN 상태, 역량 보유, 역량 수준 등이 포함 될 수 있습니다.

#### <a name="content-validation"></a>콘텐츠 유효성 검사

콘텐츠 유효성을 검사 하는 동안 제품을 만들 때 입력 한 정보를 품질 및 관련성이 있는지 확인 합니다. 이러한 검사는 세부 정보, 가격 책정, 가용성, 관련 계획 등을 나열 하는 marketplace의 항목을 검토 합니다. Azure Marketplace 및/또는 AppSource 목록 기준을 충족 하기 위해 제품에 다음과 같은 내용이 포함 되어 있는지 확인 합니다.

- 제품을 정확 하 게 설명 하는 제목입니다.
- 철저 한 개요 및 가치 제안을 제공 하는 잘 작성 된 설명입니다.
- 품질 스크린샷 및 함께 제공 되는 비디오 하거나
- 제품이 Microsoft 플랫폼 및 도구를 활용 하는 방법에 대 한 설명입니다.

[일반 목록 정책을](https://docs.microsoft.com/legal/marketplace/certification-policies#100-general)읽어 콘텐츠 유효성 검사 조건에 대해 자세히 알아보세요.

#### <a name="technical-validation"></a>기술 유효성 검사

기술 유효성을 검사 하는 동안 제품 (패키지 또는 이진)은 다음 검사를 수행 합니다.
- 맬웨어 검색 됨
- 모니터링 된 네트워크 호출
- 패키지 분석 됨
- 제품의 실제 기능을 철저 하 게 검사

이 제품은 강력한 성능을 보장 하기 위해 다양 한 플랫폼 및 버전에서 테스트 됩니다.

이 문서의 기술 구성 섹션에서 제품에 필요한 특정 구성 세부 정보를 검토 합니다.

### <a name="certification-failure-report"></a>인증 실패 보고서

검토가 완료 되 면 제품이 인증을 통과 한 경우 게시 프로세스의 다음 단계로 이동 합니다. 제품이 목록, 기술 또는 정책 검사를 모두 수행 하지 못했거나 해당 유형의 제품을 제출할 수 없는 경우 인증 실패 보고서가 생성 되어 사용자에 게 전자 메일로 전송 됩니다.

이 보고서에는 검토 메모와 함께 실패 한 모든 정책에 대 한 설명이 포함 되어 있습니다. 이 전자 메일 보고서를 검토 하 고, 문제를 해결 하 고, 필요한 경우 제품을 업데이트 하 고, 파트너 센터에서 [상용 마켓플레이스 포털](https://partner.microsoft.com/dashboard/commercial-marketplace/offers) 을 사용 하 여 제품을 다시 제출 합니다. (인증을 통과할 때까지 필요한 만큼 제품을 다시 제출할 수 있습니다.)

## <a name="preview-creation"></a>미리 보기 만들기

**미리 보기 생성** 단계에서 제품의 미리 보기 섹션에서 지정한 대상 그룹에만 액세스할 수 있는 제품의 버전을 만듭니다.

>[!Note]
> 이 단계를 사용 하 여 조직에서 제공 하는 제품을 외부 사용자에 게 제공 하지 마십시오. 대신 **개인 제품** 옵션을 사용 하세요. 이 시점에서 제품이 완전히 테스트 및 검증 되지 않았으며 외부 배포를 위한 준비가 되지 않았습니다.

## <a name="publisher-approval"></a>게시자 승인

이 단계에서는 최종 게시 단계 전에 제품 미리 보기를 검토 하 고 승인 하도록 요청 하는 전자 메일을 받게 됩니다.

Microsoft를 통해 제품을 판매 하도록 선택한 경우 제품의 획득 및 배포를 테스트 하 여이 미리 보기 승인 단계에서 요구 사항을 충족 하는지 확인할 수 있습니다. 귀하의 제품은 아직 공개 marketplace에서 제공 되지 않습니다. 이 미리 보기를 테스트 하 고 승인한 후에는 [**제품 개요**](https://partner.microsoft.com/dashboard/commercial-marketplace/overview) 대시보드에서 **라이브** 를 선택 해야 합니다.

이 미리 보기 단계에서 제품을 변경 하려는 경우 편집 하 고 다시 제출 하 여 새 미리 보기를 게시할 수 있습니다. 추가 변경 내용에 대 한 자세한 내용은 [기존 marketplace 제품 업데이트](#update-existing-marketplace-offers) 문서를 참조 하세요.

제품이 이미 라이브 이며 marketplace에서 공개에 사용할 수 있는 경우 [**제품 개요**](https://partner.microsoft.com/dashboard/commercial-marketplace/overview) 대시보드에서 **라이브** 를 선택할 때까지 업데이트가 라이브 상태가 되지 않습니다.

### <a name="publish-offer-to-the-public"></a>공개에 제품 게시

파트너 센터에 로그인 하 고 제품에 액세스 합니다. **제품 개요** 페이지로 리디렉션됩니다. 이 페이지의 맨 위에서 **라이브 전환**옵션이 표시 됩니다. **라이브 전환을** 선택 하 고 확인 후 제품이 공개로 게시 되기 시작 합니다. 제품이 살고 있으면 전자 메일 알림을 받게 됩니다.

## <a name="publish"></a>게시

이제 **제품을 사용 하도록 선택** 하 여 marketplace에서 사용할 수 있도록 선택 했으므로 live 제품이 제품의 미리 보기 버전 처럼 구성 되도록 하는 일련의 최종 유효성 검사가 수행 됩니다.

- **구매 흐름 설정 제공 (>10 분)**

이 단계에서는 Azure Portal를 통해 고객이 구매할 때 제품이 충족 될 수 있도록 합니다. 이 단계는 Microsoft를 통해 판매 되는 제품에만 적용 됩니다.

- **테스트 드라이브 데이터 유효성 검사 (~ 5 분)**

이 단계에서는 제품의 test drive 기술 구성 섹션에서 제공한 데이터의 유효성을 검사 합니다. 시험 구동 기능을 테스트 하 고 승인 합니다. 이 단계는 테스트 드라이브를 사용 하도록 설정한 경우에만 적용할 수 있습니다.

- **테스트 드라이브 프로 비전 (~ 30 분)**

이 단계에서는 고객이 사용할 수 있도록 테스트 드라이브의 인스턴스를 배포 하 고 복제 합니다.  이 단계는 테스트 드라이브를 사용 하도록 설정한 경우에만 적용할 수 있습니다.

- **리드 관리 유효성 검사 및 등록 (15 분 >)**

이 단계에서는 고객 관리 시스템에서 제품 설치에 제공 된 세부 정보에 따라 고객 리드를 받을 수 있음을 확인 합니다. 이 단계는 잠재 고객 관리를 사용 하도록 설정한 경우에만 적용 됩니다.

- **최종 게시 (>30 분)**

이 단계에서는 제품이 marketplace에서 공개적으로 제공 되는 것을 확인 합니다.
