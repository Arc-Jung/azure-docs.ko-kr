# <a name="azure-load-balancer-portal-settings"></a>Azure Load Balancer 포털 설정

Azure Load Balancer를 만들 때이 문서의 정보는 개별 설정 및 올바른 구성에 대해 자세히 알아보는 데 도움이 됩니다.

## <a name="create-load-balancer"></a>부하 분산 장치 만들기

Azure Load Balancer는 백 엔드 풀의 VM 인스턴스에 트래픽을 분산 하는 네트워크 부하 분산 장치입니다.

### <a name="basics"></a>기본 사항

부하 분산 장치 포털 만들기 페이지의 **기본 사항** 탭에 다음 정보가 표시 됩니다.



| Setting |  세부 정보 |
| ---------- | ---------- |
| Subscription  | 구독을 선택합니다. 이 선택 항목은 부하 분산 장치를 배포 하려는 구독입니다. |
| 리소스 그룹 | **새로 만들기** 를 선택 하 고 텍스트 상자에 리소스 그룹의 이름을 입력 합니다. 기존 리소스 그룹을 만든 경우 해당 그룹을 선택 합니다. |
| Name | 이 설정은 Azure Load Balancer의 이름입니다. |
| 지역 | 부하 분산 장치를 배포할 Azure 지역을 선택 합니다. |
| Type | 부하 분산 장치에는 두 가지 유형이 있습니다. </br> **내부 (전용)** </br> **공용 (외부)**.</br> ILB (내부 부하 분산 장치)는 개인 IP 주소를 통해 백 엔드 풀 멤버로 트래픽을 라우팅합니다.</br> 공용 부하 분산 장치는 인터넷을 통해 클라이언트의 요청을 백 엔드 풀로 보냅니다.</br> [부하 분산 장치 유형에](components.md#frontend-ip-configuration-)대해 자세히 알아보세요.|
| SKU  | **표준**을 선택합니다. </br> 부하 분산 장치에는 **기본** 및 **표준**의 두 가지 sku가 있습니다. </br> Basic은 기능이 제한 되어 있습니다. </br> **Standard** 는 프로덕션 워크 로드에 권장 됩니다. </br> [Sku](skus.md)에 대해 자세히 알아보세요. |

형식으로 **공용** 을 선택 하는 경우 다음 정보가 표시 됩니다.

| Setting |  세부 정보 |
| ---------- | ---------- |
| 공용 IP 주소 | **새로 만들기** 를 선택 하 여 공용 부하 분산 장치에 대 한 공용 IP 주소를 만듭니다. </br> 기존 공용 IP가 있는 경우 **기존 사용**을 선택 합니다.  |
| 공용 IP 주소 이름 | 공용 IP 주소의 이름입니다.|

| 할당 | **Static** 은 표준에 대해 자동으로 선택 됩니다. </br> 기본 공용 Ip에는 **동적** 및 **정적**이라는 두 가지 유형이 있습니다. </br> 동적 공용 IP 주소는 생성 될 때까지 할당 되지 않습니다. </br> 리소스가 삭제 되 면 Ip가 손실 될 수 있습니다. </br> 고정 IP 주소를 권장 합니다. |

| 라우팅 기본 설정 | **Microsoft 네트워크**를 선택 합니다. </br> Microsoft 네트워크는 Microsoft 글로벌 네트워크를 통해 트래픽이 라우팅되는 것을 의미 합니다. </br> 인터넷은 트래픽이 인터넷 서비스 공급자 네트워크를 통해 라우팅되는 것을 의미 합니다. </br> [라우팅 기본 설정](../virtual-network/routing-preference-overview.md) 에 대 한 자세한 정보|

## <a name="frontend-ip-configuration"></a>프런트 엔드 IP 구성

Azure Load Balancer의 IP 주소입니다. 클라이언트의 연락 지점입니다. 

하나 또는 여러 프런트 엔드 IP 구성을 사용할 수 있습니다. 위의 기본 사항 섹션을 거치는 경우 부하 분산 장치에 대 한 프런트 엔드를 이미 만든 것입니다. 

프런트 엔드 IP 구성을 부하 분산 장치에 추가 하려면 Azure Portal의 부하 분산 장치로 이동 하 고 **프런트 엔드 ip 구성**을 선택한 다음 **+ 추가**를 선택 합니다.

| Setting |  세부 정보 |
| ---------- | ---------- |
| 속성 | 프런트 엔드 IP 구성의 이름입니다. |
| IP 버전 | 프런트 엔드에 포함할 IP 주소 버전입니다. </br> 부하 분산 장치는 IPv4 및 IPv6 프런트 엔드 IP 구성을 모두 지원 합니다. |
| IP 유형 | Ip 형식은 ip 접두사를 사용 하 여 단일 IP 주소를 프런트 엔드 또는 IP 주소 범위와 연결 하는지 여부를 결정 합니다. </br> [공용 IP 접두사](../virtual-network/public-ip-address-prefix.md) 는 동일한 끝점에 반복적으로 연결 해야 하는 경우에 유용 합니다. 접두사는 SNAT 포트 문제를 지원 하기 위해 충분 한 포트가 제공 되도록 합니다. |
| 공용 IP 주소 (위의 접두사를 선택한 경우 접두사) | 부하 분산 장치 프런트 엔드에 대 한 새 공용 IP (또는 접두사)를 선택 하거나 만듭니다. |

## <a name="backend-pools"></a>백 엔드 풀

부하 분산 장치에 백 엔드 풀을 추가 하려면 Azure Portal 부하 분산 장치로 이동 하 여 **백 엔드 풀**을 선택한 다음 **+ 추가**를 선택 합니다.

| Setting | 세부 정보 |
| ---------- |  ---------- |
| 속성 | 백 엔드 풀의 이름입니다. |
| 가상 네트워크 | 백 엔드 인스턴스에 대 한 가상 네트워크입니다. |
| IP 버전 | 옵션은 **IPv4** **또는 i**p v 6입니다. |

Azure Load Balancer의 백 엔드 풀에 가상 머신 또는 가상 머신 확장 집합을 추가할 수 있습니다. 먼저 가상 머신 또는 가상 머신 확장 집합을 만듭니다. 그런 다음 포털에서 부하 분산 장치에 추가 합니다.

## <a name="health-probes"></a>상태 프로브

상태 프로브는 백 엔드 Vm 또는 인스턴스의 상태를 모니터링 하는 데 사용 됩니다. 상태 프로브 상태는 새 연결이 상태 검사에 따라 인스턴스로 전송 되는 시기를 결정 합니다. 

부하 분산 장치에 상태 프로브를 추가 하려면 Azure Portal의 부하 분산 장치로 이동 하 고 **상태 프로브**를 선택한 다음 **+ 추가**를 선택 합니다.

| Setting | 세부 정보 |
| ---------- | ---------- |
| 속성 | 상태 프로브 이름입니다. |
| 프로토콜 | 선택한 프로토콜에 따라 백 엔드 인스턴스가 정상 인지 확인 하는 데 사용 되는 검사 유형이 결정 됩니다. </br> 옵션은 다음과 같습니다. </br> **TCP** </br> **HTTPS** </br> **HTTP** </br> 올바른 프로토콜을 사용 하 고 있는지 확인 하세요. 이 선택은 응용 프로그램의 특성에 따라 달라 집니다. </br> 상태 프로브 및 프로브 응답의 구성은 새 흐름을 수신할 백 엔드 풀 인스턴스를 결정 합니다. </br> 상태 프로브를 사용 하 여 백 엔드 끝점에서 응용 프로그램의 오류를 검색할 수 있습니다. </br> [상태 프로브](load-balancer-custom-probe-overview.md)에 대해 자세히 알아보세요. |
| 포트 | 상태 프로브에 대 한 대상 포트입니다. </br> 이 설정은 상태 프로브가 인스턴스 상태를 확인 하는 데 사용 하는 백 엔드 인스턴스의 포트입니다. |
| 간격 | 프로브 시도 사이의 시간 (초)입니다. </br> 이 간격에 따라 상태 프로브가 백 엔드 인스턴스에 도달 하려고 시도 하는 빈도가 결정 됩니다. </br> 5를 선택 하면 5 초 후에 두 번째 프로브 시도가 수행 됩니다. |
| 비정상 임계값 | VM이 비정상으로 간주 되기 전에 발생 해야 하는 연속 프로브 오류 수입니다.</br> 2를 선택 하면 두 번의 연속 실패 후 새 흐름이이 백 엔드 인스턴스로 설정 되지 않습니다. |

## <a name="load-balancing-rules"></a>부하 분산 규칙

들어오는 트래픽을 백 엔드 풀 내의 모든 인스턴스에 배포 하는 방법을 정의 합니다. 부하 분산 규칙은 지정 된 프런트 엔드 IP 구성 및 포트를 여러 백 엔드 IP 주소 및 포트에 매핑합니다.

부하 분산 장치에 부하 분산 장치 규칙을 추가 하려는 경우 Azure Portal 부하 분산 장치로 이동 하 여 **부하 분산 규칙**을 선택한 다음 **+ 추가**를 선택 합니다.
    
| Setting | 세부 정보 |
| ---------- | ---------- |
| 속성 | 부하 분산 장치 규칙의 이름입니다. |
| IP 버전 | 옵션은 **IPv4** **또는 i**p v 6입니다.  |
| 프런트 엔드 IP 주소 | 프런트 엔드 IP 주소를 선택 합니다. </br> 부하 분산 장치 규칙을 연결 하려는 부하 분산 장치의 프런트 엔드 IP 주소입니다.|
| 프로토콜 | Azure Load Balancer는 계층 4 네트워크 부하 분산 장치입니다. </br> 옵션은 **TCP** 또는 **UDP**입니다. |
| 포트 | 이 설정은이 부하 분산 규칙에 따라 트래픽을 분산 하려는 프런트 엔드 IP와 연결 된 포트입니다. |
| 백 엔드 포트 | 이 설정은 부하 분산 장치에서 트래픽을 보낼 백 엔드 풀의 인스턴스에 대 한 포트입니다. 이 설정은 프런트 엔드 포트와 동일 하거나 응용 프로그램의 유연성이 필요한 경우에 다를 수 있습니다. |
| 백 엔드 풀 | 이 부하 분산 장치 규칙을 적용 하려는 백 엔드 풀입니다. |
| 상태 프로브 | 백 엔드 풀에 있는 인스턴스의 상태를 확인 하기 위해 만든 상태 프로브입니다. </br> 정상 인스턴스만 새 트래픽을 받게 됩니다. |
| 세션 지속성 |  옵션은 다음과 같습니다. </br> **없음** </br> **클라이언트 IP** </br> **클라이언트 IP 및 프로토콜**</br> </br> 클라이언트의 트래픽을 백 엔드 풀의 동일한 가상 머신으로 유지 합니다. 이 트래픽은 세션 기간 동안 유지 관리 됩니다. </br> **없음** 을 지정 하면 동일한 클라이언트의 후속 요청을 모든 가상 컴퓨터에서 처리할 수 있습니다. </br> **클라이언트 ip** 는 동일한 클라이언트 ip 주소의 연속 요청을 동일한 가상 컴퓨터에서 처리 하도록 지정 합니다. </br> **클라이언트 ip 및 프로토콜** 동일한 클라이언트 ip 주소 및 프로토콜의 후속 요청은 동일한 가상 컴퓨터에서 처리 되는지 확인 합니다. </br> [배포 모드](load-balancer-distribution-mode.md)에 대해 자세히 알아보세요. |
| 유휴 제한 시간(분) | 클라이언트에 의존 하 여 연결 유지 메시지를 보내지 않고 **TCP** 또는 **HTTP** 연결을 열어 두기 |  
| TCP 다시 설정 | 부하 분산 장치는 연결이 유휴 상태일 때 보다 예측 가능한 응용 프로그램 동작을 만들기 위해 **TCP 재설정** 을 보낼 수 있습니다. </br> [TCP 다시 설정](load-balancer-tcp-reset.md) 에 대 한 자세한 정보|
| 부동 IP | 부동 IP는 **DSR (Direct Server Return)** 이라고 하는 항목의 일부에 대 한 Azure의 용어입니다. </br> DSR은 다음과 같은 두 부분으로 구성됩니다. <br> 1. Flow 토폴로지 </br> 2. 플랫폼 수준의 IP 주소 매핑 스키마입니다. </br></br> Azure Load Balancer는 부동 IP가 사용 되는지 여부에 관계 없이 항상 DSR flow 토폴로지에서 작동 합니다. </br> 이 작업은 흐름의 아웃 바운드 부분이 항상 올바르게 다시 작성 되어 원본으로 직접 전달 됨을 의미 합니다. </br> 부동 IP를 사용 하지 않으면 Azure는 기존 부하 분산 IP 주소 매핑 체계 인 VM 인스턴스의 IP를 노출 합니다. </br> 부동 IP를 사용 하도록 설정 하면 부하 분산 장치의 프런트 엔드 IP에 매핑되는 IP 주소를 변경 하 여 유연성을 높일 수 있습니다. </br> 자세한 내용은 [Azure Load Balancer에 대 한 여러 프런트 엔드](load-balancer-multivip-overview.md)를 참조 하세요.|
| 암시적 아웃바운드 규칙 만들기 | **아니오**를 선택합니다. </br> 기본값: **disableoutboundsnat 추가 됨 = false**  </br> 이 경우 아웃 바운드는 동일한 프런트 엔드 IP를 통해 발생 합니다. </br></br> **Disableoutboundsnat 추가 됨 = true** </br>이 경우 아웃 바운드에 아웃 바운드 규칙이 필요 합니다. |

## <a name="inbound-nat-rules"></a>인바운드 NAT 규칙

인바운드 NAT 규칙은 프런트 엔드 IP 주소 및 포트 조합으로 전송 된 들어오는 트래픽을 전달 합니다. 

트래픽은 백 엔드 풀의 특정 가상 머신 또는 인스턴스로 전송됩니다. 포트 전달은 부하 분산과 동일한 해시 기반 배포를 통해 수행됩니다.

시나리오에서 백 엔드 풀의 VM 인스턴스를 분리 하는 데 RDP (원격 데스크톱 프로토콜) 또는 Secure Shell (SSH) 세션이 필요한 경우 여러 내부 끝점을 동일한 프런트 엔드 IP 주소의 포트에 매핑할 수 있습니다. 

프런트 엔드 IP 주소는 추가 점프 상자 없이 Vm을 원격으로 관리 하는 데 사용할 수 있습니다.

인바운드 nat 규칙을 부하 분산 장치에 추가 하려면 Azure Portal의 부하 분산 장치로 이동 하 여 **인바운드 nat 규칙**을 선택한 다음 **+ 추가**를 선택 합니다.

| Setting | 세부 정보 |
| ---------- | ---------- |
| 속성 | 인바운드 NAT 규칙의 이름 |
| 프런트 엔드 IP 주소 | 프런트 엔드 IP 주소를 선택 합니다. </br> 인바운드 NAT 규칙을 연결 하려는 부하 분산 장치의 프런트 엔드 IP 주소입니다. |
| IP 버전 | 옵션은 IPv4 및 IPv6입니다. |
| 서비스 | Azure Load Balancer에서 실행 되는 서비스의 유형입니다. </br> 여기에서 선택 하 여 포트 정보를 적절 하 게 업데이트 합니다. |
| 프로토콜 | Azure Load Balancer는 계층 4 네트워크 부하 분산 장치입니다. </br> 옵션은 TCP 또는 UDP입니다. |
| 유휴 제한 시간(분) | 클라이언트에 의존 하 여 연결 유지 메시지를 보내지 않고 TCP 또는 HTTP 연결을 열어 둡니다. |
| TCP 재설정 | 연결이 유휴 상태일 때 보다 예측 가능한 응용 프로그램 동작을 만들기 위해 TCP 재설정을 보낼 수 Load Balancer. </br> [TCP 다시 설정](load-balancer-tcp-reset.md) 에 대 한 자세한 정보 |
| 포트 | 이 설정은이 인바운드 NAT 규칙에 따라 트래픽을 분산 하려는 프런트 엔드 IP와 연결 된 포트입니다. |
| 대상 가상 머신 | 이 규칙을 연결할 백 엔드 풀의 가상 컴퓨터 부분입니다. |
| 포트 매핑 | 이 설정은 응용 프로그램 기본 설정에 따라 기본값이 나 사용자 지정할 수 있습니다. |

:::image type="content" source="./media/manage/inbound-nat-rule.png" alt-text="인바운드 NAT 규칙을 추가 합니다." border="true":::

## <a name="outbound-rules"></a>아웃바운드 규칙

부하 분산 디바이스 아웃바운드 규칙은 백 엔드 풀의 VM에 대한 아웃바운드 SNAT를 구성합니다.

부하 분산 장치에 아웃 바운드 규칙을 추가 하려면 Azure Portal의 부하 분산 장치로 이동 하 여 **아웃 바운드 규칙**을 선택한 다음 **+ 추가**를 선택 합니다.

| Setting | 세부 정보 |
| ------- | ------ |
| 속성 | 아웃 바운드 규칙의 이름입니다. |
| 프런트 엔드 IP 주소 | 프런트 엔드 IP 주소를 선택 합니다. </br> 아웃 바운드 규칙을 연결 하려는 부하 분산 장치의 프런트 엔드 IP 주소입니다. |
| 프로토콜 | Azure Load Balancer는 계층 4 네트워크 부하 분산 장치입니다. </br> 옵션은 **모두**, **TCP**또는 **UDP**입니다. |
| 유휴 제한 시간(분) | 클라이언트에 의존 하 여 연결 유지 메시지를 보내지 않고 **TCP** 또는 **HTTP** 연결을 열어 둡니다. |
| TCP 재설정 | 부하 분산 장치는 연결이 유휴 상태일 때 보다 예측 가능한 응용 프로그램 동작을 만들기 위해 **TCP 재설정** 을 보낼 수 있습니다. </br> [TCP 다시 설정](load-balancer-tcp-reset.md) 에 대 한 자세한 정보 |
| 백 엔드 풀 | 이 아웃 바운드 규칙을 적용 하려는 백 엔드 풀입니다. |

| Setting | 세부 정보 |
| ------- | ------ |
| 포트 할당 | **아웃 바운드 포트 수를 수동으로 선택 하는**것이 좋습니다.|

### <a name="outbound-ports"></a>아웃바운드 포트

| Setting | 세부 정보 |
| ------- | ------ |
| 선택 기준 | **인스턴스당 포트 수** 선택 |
| 인스턴스당 포트 수 | **1만**을 입력 합니다. |
