# <a name="integration-of-azure-devtest-labs-and-azure-devops"></a>Azure DevTest 랩 및 Azure DevOps 통합
DevOps는 소프트웨어 개발(Dev)을 시스템의 운영(Ops)과 통합하는 소프트웨어 개발 방법론입니다. 이 시스템은 비즈니스 목표에 맞춰 새로운 기능, 업데이트 및 수정 사항을 제공할 수 있습니다. 이 방법론은 목표, 사용 패턴 및 고객 피드백을 기반으로 새로운 기능을 디자인하는 데서 부터 모든 것을 포괄합니다. 문제가 발생할 때 시스템을 수정, 복구 및 강화합니다. 이 방법론의 쉽게 식별되는 구성 요소는 연속 통합(CI)/연속 전달(CD) 파이프라인입니다. CI/CD 파이프라인은 커밋의 정보, 코드 및 리소스를 빌드, 테스트 및 배포를 포함하는 일련의 단계를 통해 시스템을 생성합니다. 이 문서에서는 엔터프라이즈 환경에서 파이프라인 내에서 랩을 효과적으로 사용하는 다양한 방법에 대해 중점합니다. 

## <a name="benefits-of-using-labs-in-devops-workflow"></a>DevOps 워크플로우에서 랩 사용의 이점 

### <a name="focused-access"></a>집중 액세스 
랩을 구성 요소로 사용하면 특정 생태계가 제한된 그룹의 사람들과 연결할 수 있습니다. 일반적으로 공통 영역이나 특정 기능에서 작업하는 팀이나 그룹에는 해당 팀에 할당된 랩이 있습니다.   

### <a name="infrastructure-replication-in-the-cloud"></a>클라우드의 인프라 복제 
개발자는 소스 코드 및 도구가 포함된 개발 상자를 포함하는 개발 에코시스템을 신속하게 설정할 수 있습니다. 개발자는 프로덕션 구성과 거의 동일한 환경을 만들 수도 있습니다. 그것은 빠른 내부 루프 개발에 도움이됩니다. 

### <a name="pre-production"></a>사전 프로덕션 
CI/CD 파이프라인에 랩을 사용하면 비동기 테스트를 위해 여러 가지 사전 프로덕션 환경이나 컴퓨터를 동시에 실행하는 것이 더 쉬워집니다. 빌드 에이전트와 같은 다양한 지원 인프라를 랩 내에서 배포하고 관리할 수 있습니다. 

## <a name="devops-with-devtest-labs"></a>데브테스트 랩이 있는 데브옵스 

### <a name="development--operation"></a>개발 /운영 
랩은 기능 영역에서 작업하는 팀에 초점을 맞추어야 합니다. 이 공통 포커스를 사용하면 도구, 스크립트 또는 리소스 관리자 템플릿과 같은 지역별 리소스를 공유할 수 있습니다. 그것은 작은 그룹에 부정적인 영향을 제한 하면서 빠른 변경 수 있습니다. 이러한 공유 리소스를 통해 개발자는 필요한 모든 코드, 도구 및 구성을 사용하여 개발 VM을 만들 수 있습니다. 동적으로 만들거나 사용자 지정을 통해 기본 이미지를 만드는 시스템을 가질 수 있습니다. 개발자는 VM을 만들 수 있지만 필요한 템플릿을 기반으로 DevTest Labs 환경을 만들어 랩에서 적절한 Azure 리소스를 만들 수도 있습니다. 다른 사람에게 영향을 주지 않고 랩 환경에 대해 변경되거나 파괴적인 작업을 수행할 수 있습니다. 제품이 고객의 컴퓨터에 설치된 독립 실행형 시스템인 시나리오를 고려합니다. 이 시나리오에서 DevTest Labs는 아티팩트를 사용하여 추가 소프트웨어를 설치하고 코드의 내부 루프 테스트를 더 빠르게 하기 위해 고객 구성을 미리 빌드하는 것을 포함하는 VM 생성을 개선했습니다. 
  
## <a name="cicd-pipeline"></a>CI/CD 파이프라인 
CI/CD 파이프라인은 개발자의 끌어오기 요청에서 코드를 이동하고, 기존 코드와 통합하고, 프로덕션 에코시스템에 배포하는 DevOps의 중요한 구성 요소 중 하나입니다. 모든 리소스가 랩 내에 있을 필요는 없습니다. 예를 들어 Jenkins 호스트는 랩 외부에 보다 지속적인 리소스로 설정할 수 있습니다. 다음은 랩을 파이프라인에 통합하는 몇 가지 구체적인 예입니다. 

### <a name="build"></a>빌드 
빌드 파이프라인은 릴리스 파이프라인에 전달되도록 함께 테스트할 구성 요소 패키지를 만드는 데 중점을 두었습니다. 랩은 빌드 에이전트 및 기타 지원 리소스의 위치로 빌드 파이프라인의 일부가 될 수 있습니다. 인프라를 동적으로 구축할 수 있으면 더 많은 제어를 할 수 있습니다. 랩에 여러 환경을 가질 수 있는 기능을 사용하면 빌드 ID를 환경 정보의 일부로 사용하여 특정 빌드에 대한 리소스를 고유하게 식별하는 동안 각 빌드를 비동기적으로 실행할 수 있습니다.   

빌드 에이전트의 경우 랩에서 액세스를 제한할 수 있어 보안이 향상되고 우발적인 손상이 발생할 가능성이 줄어듭니다.  

### <a name="test"></a>테스트 
DevTest Labs를 사용하면 CI/CD 파이프라인을 사용하여 자동화 및 수동 테스트에 사용할 수 있는 Azure 리소스(VM, 환경)의 생성을 자동화할 수 있습니다. VM은 빌드 프로세스의 정보를 사용하여 테스트에 필요한 다양한 사용자 지정 구성을 만드는 아티팩트 또는 수식을 사용하여 만들어집니다.   

### <a name="release"></a>해제 
DevTest Labs는 일반적으로 코드를 배포하기 전에 릴리스 섹션에서 확인에 사용됩니다. 빌드 섹션에서 테스트하는 것과 유사합니다. 프로덕션 리소스는 DevTest 랩 내에 배포해서는 안 됩니다. 

### <a name="customization"></a>사용자 지정 
Azure DevOps에는 특정 랩 내에서 VM 및 환경을 조작할 수 있는 기존 작업이 있습니다. Azure DevOps 서비스는 CI/CD 파이프라인을 관리하는 한 가지 방법이지만 랩을 REST API 호출, PowerShell 스크립트 실행 또는 Azure CLI 사용 기능을 지원하는 모든 시스템에 통합할 수 있습니다. 

일부 CI/CD 파이프라인 관리자에는 Azure 및 DevTest Labs 내에서 리소스를 관리할 수 있는 기존 오픈 소스 플러그인이 있습니다. 파이프라인의 특정 요구 사항에 맞게 일부 사용자 지정 스크립팅을 사용해야 할 수 있습니다.  이 점을 염두에 두고 작업을 수행할 때 서비스 주체가 적절한 역할과 함께 랩에 액세스하는 데 사용됩니다. 서비스 주체는 일반적으로 랩에 대한 기여자 역할 액세스가 필요합니다. 자세한 내용은 [Azure DevOps 지속적인 통합 및 배달 파이프라인에 Azure DevTests 랩 통합을 참조하세요.](devtest-lab-integrate-ci-cd-vsts.md)
