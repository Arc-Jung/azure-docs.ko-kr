# <a name="what-is-a-primary-refresh-token"></a>기본 새로 고침 토큰이란 무엇입니까?

PRT(기본 새로 고침 토큰)는 Windows 10, iOS 및 Android 장치에서 Azure AD 인증의 주요 아티팩트입니다. Microsoft 의 첫 번째 파티 토큰 브로커에게 특별히 발급된 JSON 웹 토큰(JWT)은 해당 장치에서 사용되는 응용 프로그램에서 단일 사인온(SSO)을 활성화합니다. 이 문서에서는 PRT가 Windows 10 장치에서 발급, 사용 및 보호되는 방법에 대한 자세한 내용을 제공합니다.

이 문서에서는 Azure AD에서 사용할 수 있는 다른 장치 상태와 Windows 10에서 단일 사인온이 작동하는 방식을 이미 이해하고 있다고 가정합니다. Azure AD의 장치에 대한 자세한 내용은 [Azure Active Directory의 장치 관리란 무엇입니까?](overview.md)

## <a name="key-terminology-and-components"></a>주요 용어 및 구성 요소

다음 Windows 구성 요소는 PRT를 요청하고 사용하는 데 중요한 역할을 합니다.

* **클라우드 인증** 공급자(CloudAP): CloudAP는 Windows 10 장치에 로그인하는 사용자를 확인하는 Windows 로그인용 최신 인증 공급자입니다. CloudAP는 ID 공급자가 해당 ID 공급자의 자격 증명을 사용하여 Windows에 대한 인증을 사용하도록 빌드할 수 있는 플러그인 프레임워크를 제공합니다.
* **웹 계정** 관리자(WAM): WAM은 Windows 10 장치의 기본 토큰 브로커입니다. 또한 WAM은 ID 공급자가 해당 ID 공급자에 의존하는 응용 프로그램에 SSO를 빌드하고 사용하도록 설정할 수 있는 플러그인 프레임워크를 제공합니다.
* **Azure AD CloudAP 플러그인**: Windows 로그인 중에 Azure AD를 사용 하 고 사용자 자격 증명을 확인 하는 CloudAP 프레임 워크에 내장 된 Azure AD 특정 플러그인입니다.
* **Azure AD WAM 플러그인**: SSO가 인증을 위해 Azure AD를 사용하는 응용 프로그램에 SSO를 사용할 수 있도록 WAM 프레임워크에 빌드된 Azure AD 특정 플러그인입니다.
* **Dsreg**: 모든 장치 상태에 대한 장치 등록 프로세스를 처리하는 Windows 10의 Azure AD 특정 구성 요소입니다.
* **신뢰할 수 있는 플랫폼** 모듈(TPM): TPM은 장치보안에 내장된 하드웨어 구성 요소로, 사용자 및 장치 보안에 대한 하드웨어 기반 보안 기능을 제공합니다. 자세한 내용은 신뢰할 수 [있는 플랫폼 모듈 기술 개요](/windows/security/information-protection/tpm/trusted-platform-module-overview)문서에서 찾을 수 있습니다.

## <a name="what-does-the-prt-contain"></a>PRT에는 무엇이 포함되어 있습니까?

PRT에는 일반적으로 Azure AD 새로 고침 토큰에 포함된 클레임이 포함되어 있습니다. 또한 PRT에 포함된 일부 장치별 클레임이 있습니다. 다음과 같습니다.

* **장치 ID**: PRT는 특정 장치에서 사용자에게 발급됩니다. 장치 ID `deviceID` 클레임은 PRT가 사용자에게 발급된 장치를 결정합니다. 이 클레임은 나중에 PRT를 통해 얻은 토큰에 발행됩니다. 장치 ID 클레임은 장치 상태 또는 규정 준수에 따라 조건부 액세스에 대한 승인을 결정하는 데 사용됩니다.
* **세션 키**: 세션 키는 PRT의 일부로 발급 된 Azure AD 인증 서비스에서 생성 된 암호화 된 대칭 키입니다. 세션 키는 PRT가 다른 응용 프로그램에 대한 토큰을 얻는 데 사용될 때 소유 증명역할을 합니다.

### <a name="can-i-see-whats-in-a-prt"></a>PRT에 무엇이 있는지 볼 수 있습니까?

PRT는 Azure AD에서 보낸 불투명 한 Blob이며 그 내용은 클라이언트 구성 요소에 알려지지 않습니다. PRT 내부의 내용을 볼 수 없습니다.

## <a name="how-is-a-prt-issued"></a>PRT는 어떻게 발급되어 있습니까?

장치 등록은 Azure AD에서 장치 기반 인증을 위한 필수 구성 조건입니다. PRT는 등록된 장치에서만 사용자에게 발급됩니다. 장치 등록에 대한 자세한 내용은 비즈니스 [및 장치 등록에 대한 Windows Hello](/windows/security/identity-protection/hello-for-business/hello-how-it-works-device-registration)문서를 참조하십시오. 장치 등록 중에 dsreg 구성 요소는 두 세트의 암호화 키 쌍을 생성합니다.

* 장치 키(dkpub/dkpriv)
* 전송 키(tkpub/tkpriv)

개인 키는 디바이스에 유효하고 작동하는 TPM이 있는 경우 장치의 TPM에 바인딩되며, 공개 키는 장치 등록 프로세스 중에 Azure AD로 전송됩니다. 이러한 키는 PRT 요청 중에 장치 상태를 확인하는 데 사용됩니다.

PRT는 두 가지 시나리오에서 Windows 10 장치에서 사용자 인증 중에 발급됩니다.

* **Azure AD 조인** 또는 **하이브리드 Azure AD 조인:** 사용자가 조직 자격 증명으로 로그인할 때 Windows 로그온 중에 PRT가 발급됩니다. PRT는 모든 Windows와 함께 발급 됩니다 10 지원 되는 자격 증명, 예를 들어, 암호 및 비즈니스에 대 한 Windows 안녕하세요. 이 시나리오에서 Azure AD CloudAP 플러그인은 PRT의 기본 권한입니다.
* **Azure AD 등록 장치**: 사용자가 Windows 10 장치에 보조 작업 계정을 추가하면 PRT가 발급됩니다. 사용자는 두 가지 방법으로 Windows 10에 계정을 추가할 수 있습니다.  
   * 앱에 로그인한 후 **이 장치의 모든 곳에서 이 계정 사용** 프롬프트(예: Outlook)를 통해 계정 추가
   * **설정** > **계정에서** > 계정 추가**액세스 작업 또는 학교** > **연결**

Azure AD 등록 된 장치 시나리오에서 Windows 로그온이 이 Azure AD 계정에서 발생 하지 않기 때문에 Azure AD WAM 플러그인은 PRT에 대 한 기본 권한입니다.

> [!NOTE]
> 타사 ID 공급자는 Windows 10 장치에서 PRT 발급을 활성화하기 위해 WS-Trust 프로토콜을 지원해야 합니다. WS-Trust가 없으면 하이브리드 Azure AD 조인 또는 Azure AD 조인 장치의 사용자에게 PRT를 발급할 수 없습니다.

## <a name="what-is-the-lifetime-of-a-prt"></a>PRT의 수명은 무엇입니까?

PRT가 발급되면 14일 동안 유효하며 사용자가 장치를 적극적으로 사용하는 한 지속적으로 갱신됩니다.  

## <a name="how-is-a-prt-used"></a>PRT는 어떻게 사용합니까?

PRT는 Windows의 두 가지 주요 구성 요소에서 사용됩니다.

* **Azure AD CloudAP 플러그인**: Windows 로그인 중에 Azure AD CloudAP 플러그인은 사용자가 제공한 자격 증명을 사용하여 Azure AD에서 PRT를 요청합니다. 또한 사용자가 인터넷 연결에 액세스할 수 없을 때 캐시된 로그인을 사용하도록 PRT를 캐시합니다.
* **Azure AD WAM 플러그인**: 사용자가 응용 프로그램에 액세스하려고 할 때 Azure AD WAM 플러그인은 PRT를 사용하여 Windows 10에서 SSO를 활성화합니다. Azure AD WAM 플러그인은 PRT를 사용하여 토큰 요청에 대해 WAM을 사용하는 응용 프로그램에 대한 새로 고침 및 액세스 토큰을 요청합니다. 또한 브라우저 요청에 PRT를 삽입하여 브라우저에서 SSO를 사용할 수 있습니다. 윈도우에서 브라우저 SSO 10 마이크로 소프트 에지에서 지원 (기본적으로) 크롬 (윈도우를 통해 10 계정 또는 사무실 온라인 확장).

## <a name="how-is-a-prt-renewed"></a>PRT는 어떻게 갱신합니까?

PRT는 두 가지 방법으로 갱신됩니다.

* **Azure AD CloudAP 플러그인매4시간마다:** CloudAP 플러그인은 Windows 로그인 중에 4시간마다 PRT를 갱신합니다. 사용자가 해당 시간 동안 인터넷에 연결되어 있지 않은 경우, CloudAP 플러그인은 장치가 인터넷에 연결 된 후 PRT를 갱신합니다.
* **앱 토큰 요청 중 Azure AD WAM 플러그인:** WAM 플러그인은 응용 프로그램에 대한 자동 토큰 요청을 활성화하여 Windows 10 장치에서 SSO를 활성화합니다. WAM 플러그인은 두 가지 방법으로 이러한 토큰 요청 중에 PRT를 갱신 할 수 있습니다 :
   * 앱이 WAM에 액세스 토큰을 자동으로 요청하지만 해당 앱에 사용할 수 있는 새로 고침 토큰은 없습니다. 이 경우 WAM은 PRT를 사용하여 앱에 대한 토큰을 요청하고 응답에서 새 PRT를 다시 가져옵니다.
   * 앱이 액세스 토큰에 대해 WAM을 요청하지만 PRT가 유효하지 않거나 Azure AD에 추가 권한 부여(예: Azure 다단계 인증)가 필요합니다. 이 시나리오에서 WAM은 사용자가 다시 인증하거나 추가 확인을 제공하도록 요구하는 대화형 로그온을 시작하고 성공적인 인증시 새 PRT가 발급됩니다.

### <a name="key-considerations"></a>주요 고려 사항

* PRT는 네이티브 앱 인증 중에만 발급되고 갱신됩니다. PRT는 브라우저 세션 중에 갱신또는 발급되지 않습니다.
* Azure AD 조인 및 하이브리드 Azure AD 조인 장치에서 CloudAP 플러그인은 PRT의 기본 권한입니다. WAM 기반 토큰 요청 중에 PRT가 갱신되면 PRT가 CloudAP 플러그인으로 다시 전송되어 PRT를 수락하기 전에 Azure AD를 사용하여 PRT의 유효성을 확인합니다.

## <a name="how-is-the-prt-protected"></a>PRT는 어떻게 보호합니까?

PRT는 사용자가 로그인한 장치에 바인딩하여 보호됩니다. Azure AD 및 Windows 10은 다음 방법을 통해 PRT 보호를 활성화합니다.

* **첫 번째 로그인 중**: 첫 번째 로그인 중에 장치 등록 중에 암호화된 장치 키를 사용하여 요청에 서명하여 PRT가 발행됩니다. 유효하고 작동하는 TPM이 있는 장치에서장치 키는 악의적인 액세스를 방지하는 TPM에 의해 보호됩니다. 해당 장치 키 서명의 유효성을 검사할 수 없는 경우 PRT가 발급되지 않습니다.
* **토큰 요청 및 갱신 중**: PRT가 발급되면 Azure AD는 장치에 암호화된 세션 키도 발급합니다. 장치 등록의 일부로 생성되고 Azure AD로 전송되는 대중 전송 키(tkpub)로 암호화됩니다. 이 세션 키는 TPM에서 보호한 개인 전송 키(tkpriv)에서만 해독할 수 있습니다. 세션 키는 Azure AD로 전송된 모든 요청에 대한 POP(소유 증명) 키입니다.  세션 키도 TPM에 의해 보호되며 다른 OS 구성 요소는 액세스할 수 없습니다. 토큰 요청 또는 PRT 갱신 요청은 TPM을 통해 이 세션 키에 의해 안전하게 서명되므로 변조할 수 없습니다. Azure AD는 해당 세션 키에 의해 서명되지 않은 장치의 요청을 무효화합니다.

TPM으로 이러한 키를 보호하면 공격자가 장치를 물리적으로 소유하고 있어도 TPM에 액세스할 수 없으므로 악의적인 행위자는 키를 훔치거나 다른 곳에서 PRT를 재생할 수 없습니다.  따라서 TPM을 사용하면 Azure AD 조인, 하이브리드 Azure AD 조인 및 자격 증명 도용에 대해 Azure AD 등록된 장치의 보안이 크게 향상됩니다. 성능 및 안정성을 위해 TPM 2.0은 Windows 10의 모든 Azure AD 장치 등록 시나리오에 권장되는 버전입니다.

### <a name="how-are-app-tokens-and-browser-cookies-protected"></a>앱 토큰 및 브라우저 쿠키는 어떻게 보호되나요?

**앱 토큰**: 앱이 WAM을 통해 토큰을 요청할 때 Azure AD는 새로 고침 토큰과 액세스 토큰을 발행합니다. 그러나 WAM은 액세스 토큰만 앱에 반환하고 사용자의 데이터 보호 응용 프로그램 프로그래밍 인터페이스(DPAPI) 키로 암호화하여 캐시에서 새로 고침 토큰을 보호합니다. WAM은 세션 키로 요청에 서명하여 새로 고침 토큰을 안전하게 사용하여 추가 액세스 토큰을 발급합니다. DPAPI 키는 Azure AD 자체의 Azure AD 기반 대칭 키에 의해 보호됩니다. 장치가 DPAPI 키로 사용자 프로필을 해독해야 하는 경우 Azure AD는 세션 키로 암호화된 DPAPI 키를 제공하며, CloudAP 플러그인은 TPM에 해독을 요청합니다. 이 기능은 새로 고침 토큰 을 보호하는 일관성을 보장하고 응용 프로그램이 자체 보호 메커니즘을 구현하는 것을 방지합니다.  

**브라우저 쿠키**: 윈도우에서 10, Azure AD는 기본적으로 인터넷 익스플로러와 마이크로 소프트 에지또는 윈도우를 통해 구글 크롬에서 브라우저 SSO를 지원합니다 10 계정 확장. 보안은 쿠키를 보호할 뿐만 아니라 쿠키가 전송되는 끝점을 보호하기 위해 만들어졌습니다. 브라우저 쿠키는 세션 키를 활용하여 쿠키에 서명하고 보호함으로써 PRT와 동일한 방식으로 보호됩니다.

사용자가 브라우저 상호 작용을 시작하면 브라우저(또는 확장)가 COM 네이티브 클라이언트 호스트를 호출합니다. 네이티브 클라이언트 호스트는 페이지가 허용된 도메인 중 하나에서 온 지 확인합니다. 브라우저는 nonce를 포함하여 네이티브 클라이언트 호스트에 다른 매개 변수를 보낼 수 있지만 네이티브 클라이언트 호스트는 호스트 이름의 유효성 검사를 보장합니다. 네이티브 클라이언트 호스트는 CloudAP 플러그인에서 PRT 쿠키를 요청하여 TPM으로 보호된 세션 키를 만들고 서명합니다. PRT 쿠키는 세션 키에 의해 서명되어 있으므로 변조할 수 없습니다. 이 PRT 쿠키는 Azure AD의 요청 헤더에 포함되어 있어 원래 장치의 유효성을 검사합니다. Chrome 브라우저를 사용하는 경우 네이티브 클라이언트 호스트의 매니페스트에 명시적으로 정의된 확장만 호출하여 임의의 확장이 이러한 요청을 하지 못하도록 할 수 있습니다. Azure AD에서 PRT 쿠키의 유효성을 검사하면 브라우저에 세션 쿠키를 발급합니다. 이 세션 쿠키에는 PRT에서 발급된 세션 키도 포함되어 있습니다. 후속 요청 중에 세션 키는 쿠키를 장치에 효과적으로 바인딩하고 다른 곳에서 재생을 방지합니다.

## <a name="when-does-a-prt-get-an-mfa-claim"></a>PRT는 언제 MFA 클레임을 받을 수 있습니까?

PRT는 특정 시나리오에서 MFA(다단계 인증) 클레임을 얻을 수 있습니다. MFA 기반 PRT를 사용하여 응용 프로그램에 대한 토큰을 요청하는 경우 MFA 클레임은 해당 앱 토큰으로 전송됩니다. 이 기능은 필요한 모든 앱에 대해 MFA 챌린지를 방지하여 사용자에게 원활한 환경을 제공합니다. PRT는 다음과 같은 방법으로 MFA 클레임을 얻을 수 있습니다.

* **비즈니스용 Windows Hello로 로그인**: 비즈니스용 Windows Hello는 암호를 대체하고 암호화 키를 사용하여 강력한 이중 인증을 제공합니다. 비즈니스용 Windows Hello는 장치의 사용자에 따라 다르며 자체에서 MFA를 프로비전해야 합니다. 사용자가 비즈니스용 Windows Hello로 로그인하면 사용자의 PRT에 MFA 클레임이 표시됩니다. 이 시나리오는 스마트 카드 인증ADFS에서 MFA 클레임을 생성하는 경우 스마트 카드로 로그인하는 사용자에게도 적용됩니다.
   * 비즈니스용 Windows Hello는 다단계 인증으로 간주되므로 PRT 자체가 새로 고쳐질 때 MFA 클레임이 업데이트되므로 사용자가 비즈니스용 WIndows Hello로 로그인할 때 MFA 기간이 계속 연장됩니다.
* **WAM 대화형 로그인 중 MFA**: WAM을 통한 토큰 요청 중에 사용자가 앱에 액세스하기 위해 MFA를 수행해야 하는 경우 이 상호 작용 중에 갱신되는 PRT에는 MFA 클레임이 각인됩니다.
   * 이 경우 MFA 클레임은 지속적으로 업데이트되지 않으므로 MFA 기간은 디렉터리에서 설정된 수명을 기반으로 합니다.
   * 이전 기존 PRT 및 RT가 앱에 액세스하는 데 사용되는 경우 PRT 및 RT는 첫 번째 인증 증명으로 간주됩니다. 두 번째 증명과 각인된 MFA 클레임이 있는 새 AT가 필요합니다. 이것은 또한 새로운 PRT 및 RT를 발행합니다.
* **장치 등록 중 MFA**: 관리자가 [장치를 등록하기 위해 MFA를 요구하도록](device-management-azure-portal.md#configure-device-settings)Azure AD에서 장치 설정을 구성한 경우 사용자는 등록을 완료하기 위해 MFA를 수행해야 합니다. 이 과정에서 사용자에게 발급되는 PRT에는 등록 중에 획득한 MFA 클레임이 있습니다. 이 기능은 조인 작업을 수행한 사용자에게만 적용되며 해당 장치에 로그인한 다른 사용자에게만 적용됩니다.
   * WAM 대화형 로그인과 마찬가지로 MFA 클레임은 지속적으로 업데이트되지 않으므로 MFA 기간은 디렉터리에서 설정된 수명을 기준으로 합니다.

Windows 10은 각 자격 증명에 대해 분할된 PRT 목록을 유지 관리합니다. 그래서, 비즈니스에 대 한 Windows 안녕하세요의 각각에 대 한 PRT가 있다, 암호, 또는 스마트 카드. 이 분할은 MFA 클레임이 사용된 자격 증명에 따라 격리되고 토큰 요청 중에 혼합되지 않도록 합니다.

## <a name="how-is-a-prt-invalidated"></a>PRT는 어떻게 무효화합니까?

PRT는 다음 시나리오에서 무효화됩니다.

* **잘못된 사용자**: Azure AD에서 사용자가 삭제되거나 비활성화된 경우 해당 PRT가 무효화되며 응용 프로그램에 대한 토큰을 가져오는 데 사용할 수 없습니다. 삭제되었거나 비활성화된 사용자가 이전에 장치에 이미 로그인한 경우 CloudAP가 잘못된 상태를 인식할 때까지 캐시된 로그인이 로그인됩니다. CloudAP에서 사용자가 유효하지 않은 것으로 확인되면 후속 로그온이 차단됩니다. 잘못된 사용자는 자격 증명이 캐시되지 않은 새 장치에 로그인하지 못하도록 자동으로 차단됩니다.
* **잘못된 장치**: Azure AD에서 장치가 삭제되거나 비활성화된 경우 해당 장치에서 얻은 PRT가 무효화되며 다른 응용 프로그램에 대한 토큰을 가져오는 데 사용할 수 없습니다. 사용자가 이미 잘못된 장치에 로그인한 경우 계속 로그인할 수 있습니다. 그러나 장치의 모든 토큰은 무효화되고 사용자는 해당 장치의 리소스에 대한 SSO를 갖지 않습니다.
* **암호 변경**: 사용자가 암호를 변경하면 이전 암호로 얻은 PRT가 Azure AD에 의해 무효화됩니다. 암호 변경으로 인해 사용자가 새 PRT를 얻게 됩니다. 이 무효화는 두 가지 방법으로 발생할 수 있습니다.
   * 사용자가 새 암호로 Windows에 로그인하는 경우 CloudAP는 이전 PRT를 폐기하고 Azure AD에 새 암호로 새 PRT를 발급하도록 요청합니다. 사용자가 인터넷에 연결되어 있지 않으면 새 암호의 유효성을 검사할 수 없으며 Windows에서 사용자에게 이전 암호를 입력하도록 요구할 수 있습니다.
   * 사용자가 이전 암호로 로그인했거나 Windows에 로그인한 후 암호를 변경한 경우 이전 PRT는 WAM 기반 토큰 요청에 사용됩니다. 이 시나리오에서는 WAM 토큰 요청 중에 다시 인증 하라는 메시지가 표시 되고 새 PRT가 발급됩니다.
* **TPM 문제**: 경우에 따라 장치의 TPM이 흔들리거나 실패하여 TPM에 의해 보호된 키에 액세스할 수 없게 될 수 있습니다. 이 경우, 장치는 암호화 키의 소유를 증명할 수 없기 때문에 PRT를 얻거나 기존 PRT를 사용하여 토큰을 요청할 수 없습니다. 따라서 기존 PRT는 Azure AD에 의해 무효화됩니다. Windows 10에서 오류가 감지되면 새 암호화 키로 장치를 다시 등록하는 복구 흐름이 시작됩니다. 하이브리드 Azure Ad 조인을 사용하면 초기 등록과 마찬가지로 사용자 입력 없이 복구가 자동으로 수행됩니다. Azure AD 조인 또는 Azure AD 등록 장치의 경우 장치에 관리자 권한이 있는 사용자가 복구를 수행해야 합니다. 이 시나리오에서는 복구 흐름은 사용자가 성공적으로 장치를 복구 하도록 안내 하는 Windows 프롬프트에 의해 시작 됩니다.

> [!NOTE]

| 단계 | 설명 |
| :---: | --- |
| A | 사용자가 로그인 UI에 암호를 입력합니다. LogonUI는 인증 버퍼의 자격 증명을 LSA에 전달하며, 이 자격 증명은 내부적으로 CloudAP에 전달합니다. CloudAP는 이 요청을 CloudAP 플러그인으로 전달합니다. |
| b | CloudAP 플러그인은 사용자의 ID 공급자를 식별하기 위해 영역 검색 요청을 시작합니다. 사용자의 테넌트에 페더레이션 공급자 설정이 있는 경우 Azure AD는 페더레이션 공급자의 메타데이터 교환 끝점(MEX) 끝점을 반환합니다. 그렇지 않은 경우 Azure AD는 사용자가 Azure AD로 인증할 수 있음을 나타내는 관리되는 사용자를 반환합니다. |
| C | 사용자가 관리되는 경우 CloudAP는 Azure AD에서 nonce를 가져옵니다. 사용자가 페더레이션된 경우 CloudAP 플러그인은 사용자의 자격 증명을 사용하여 페더레이션 공급자로부터 SAML 토큰을 요청합니다. SAML 토큰이 수신되면 Azure AD에서 nonce를 요청합니다. |
| D | CloudAP 플러그인은 사용자의 자격 증명, nonce 및 브로커 범위로 인증 요청을 구성하고 장치 키(dkpriv)로 요청에 서명하고 Azure AD로 보냅니다. 페더레이션 환경에서 CloudAP 플러그인은 사용자의 자격 증명 대신 페더레이션 공급자가 반환하는 SAML 토큰을 사용합니다. |
| E | Azure AD는 사용자 자격 증명, nonce 및 장치 서명의 유효성을 검사하고 장치가 테넌트에서 유효한지 확인하고 암호화된 PRT를 발급합니다. PRT와 함께 Azure AD는 전송 키(tkpub)를 사용하여 Azure AD에서 암호화한 세션 키라는 대칭 키도 발급합니다. 또한 세션 키는 PRT에도 포함됩니다. 이 세션 키는 PRT에 대한 후속 요청에 대한 소유 증명(PoP) 키 역할을 합니다. |
| F | CloudAP 플러그인은 암호화된 PRT 및 세션 키를 CloudAP에 전달합니다. CloudAP는 전송 키(tkpriv)를 사용하여 세션 키의 암호를 해독하고 TPM자체 키를 사용하여 세션 키를 다시 암호화하도록 TPM에 요청합니다. CloudAP는 암호화된 세션 키를 PRT와 함께 캐시에 저장합니다. |

| E | CloudAP 플러그인은 사용자의 자격 증명, nonce 및 기존 PRT를 사용하여 인증 요청을 구성하고 세션 키로 요청에 서명하고 Azure AD로 보냅니다. 페더레이션 환경에서 CloudAP 플러그인은 사용자의 자격 증명 대신 페더레이션 공급자가 반환하는 SAML 토큰을 사용합니다. |
| F | Azure AD는 PRT에 포함된 세션 키와 비교하여 세션 키 서명의 유효성을 검사하고 nonce의 유효성을 검사하고 장치가 테넌트에서 유효한지 확인하고 새 PRT를 발급합니다. 앞에서 보았듯이 PRT는 전송 키(tkpub)에 의해 암호화된 세션 키와 다시 함께 제공됩니다. |
| G | CloudAP 플러그인은 암호화된 PRT 및 세션 키를 CloudAP에 전달합니다. CloudAP는 전송 키(tkpriv)를 사용하여 세션 키를 해독하고 TPM 자체 키를 사용하여 세션 키를 다시 암호화하도록 TPM에 요청합니다. CloudAP는 암호화된 세션 키를 PRT와 함께 캐시에 저장합니다. |

### <a name="prt-usage-during-app-token-requests"></a>앱 토큰 요청 중 PRT 사용

![앱 토큰 요청 중 PRT 사용](./media/concept-primary-refresh-token/prt-usage-app-token-requests.png)

| 단계 | 설명 |
| :---: | --- |
| A | 응용 프로그램(예: Outlook, OneNote 등)은 WAM에 토큰 요청을 시작합니다. WAM은 차례로 Azure AD WAM 플러그인에 토큰 요청을 서비스하도록 요청합니다. |
| b | 응용 프로그램에 대한 새로 고침 토큰을 이미 사용할 수 있는 경우 Azure AD WAM 플러그인은 이를 사용하여 액세스 토큰을 요청합니다. 장치 바인딩 증명을 제공하기 위해 WAM 플러그인은 세션 키로 요청에 서명합니다. Azure AD는 세션 키의 유효성을 검사하고 세션 키로 암호화된 앱에 대한 액세스 토큰 및 새 새로 고침 토큰을 발행합니다. WAM 플러그인은 클라우드 AP 플러그인을 요청하여 토큰을 해독하도록 요청하며, 차례로 TPM이 세션 키를 사용하여 해독하도록 요청하여 WAM 플러그인이 두 토큰을 모두 가져옵니다. 다음으로 WAM 플러그인은 응용 프로그램에 대한 액세스 토큰만 제공하며 새로 고침 토큰을 DPAPI로 다시 암호화하고 자체 캐시에 저장합니다.  |
| C |  응용 프로그램에 대한 새로 고침 토큰을 사용할 수 없는 경우 Azure AD WAM 플러그인은 PRT를 사용하여 액세스 토큰을 요청합니다. 소유 증명을 제공하기 위해 WAM 플러그인은 세션 키와 함께 PRT를 포함하는 요청에 서명합니다. Azure AD는 PRT에 포함된 세션 키와 비교하여 세션 키 서명의 유효성을 검사하고 장치가 유효한지 확인하고 응용 프로그램에 대한 액세스 토큰 및 새로 고침 토큰을 발급합니다. 또한 Azure AD는 세션 키로 암호화된 새 PRT(새로 고침 주기에 따라)를 발행할 수 있습니다. |
| D | WAM 플러그인은 클라우드 AP 플러그인을 요청하여 토큰을 해독하도록 요청하며, 차례로 TPM이 세션 키를 사용하여 해독하도록 요청하여 WAM 플러그인이 두 토큰을 모두 가져옵니다. 다음으로 WAM 플러그인은 응용 프로그램에 대한 액세스 토큰만 제공하며 새로 고침 토큰을 DPAPI로 다시 암호화하고 자체 캐시에 저장합니다. WAM 플러그인은이 응용 프로그램에 대 한 앞으로 새로 고침 토큰을 사용 합니다. WAM 플러그인은 또한 클라우드 AP 플러그인에 새로운 PRT를 다시 제공, 자체 캐시에서 업데이트 하기 전에 Azure AD와 PRT의 유효성을 검사. 클라우드 AP 플러그인은 앞으로 새로운 PRT를 사용합니다. |
| E | WAM은 새로 발급된 액세스 토큰을 WAM에 제공하며, 이 토큰은 호출 응용 프로그램에 다시 제공합니다.|

### <a name="browser-sso-using-prt"></a>PRT를 사용하는 브라우저 SSO

![PRT를 사용하는 브라우저 SSO](./media/concept-primary-refresh-token/browser-sso-using-prt.png)

| 단계 | 설명 |
| :---: | --- |
| A | 사용자는 자격 증명을 사용하여 Windows에 로그인하여 PRT를 가져옵니다. 사용자가 브라우저를 열면 브라우저(또는 확장)가 레지스트리에서 URL을 로드합니다. |
| b | 사용자가 Azure AD 로그인 URL을 열면 브라우저 또는 확장 프로그램이 레지스트리에서 가져온 URL의 유효성을 검사합니다. 일치하는 경우 브라우저는 토큰을 얻기 위해 네이티브 클라이언트 호스트를 호출합니다. |
| C | 네이티브 클라이언트 호스트는 URL이 Microsoft ID 공급자(Microsoft 계정 또는 Azure AD)에 속하는지 확인하고 URL에서 보낸 nonce를 추출하고 CloudAP 플러그인을 호출하여 PRT 쿠키를 가져옵니다. |
| D | CloudAP 플러그인은 PRT 쿠키를 만들고 TPM 바인딩 세션 키로 로그인한 후 네이티브 클라이언트 호스트로 다시 보냅니다. 쿠키는 세션 키에 의해 서명되기 때문에 변조할 수 없습니다. |
| E | 네이티브 클라이언트 호스트는 이 PRT 쿠키를 브라우저로 반환하며, 여기에는 x-ms-RefreshTokenCredential이라는 요청 헤더의 일부로 포함되고 Azure AD에서 토큰을 요청합니다. |
| F | Azure AD는 PRT 쿠키의 세션 키 서명을 확인하고, nonce의 유효성을 검사하고, 테넌트에서 장치가 유효한지 확인하고, 웹 페이지에 대한 ID 토큰과 브라우저에 대한 암호화된 세션 쿠키를 발급합니다. |
