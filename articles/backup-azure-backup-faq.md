# Azure 백업-FAQ
다음은 Azure 백업에 대 한 자주 묻는 질문의 목록입니다. Azure 백업에 대 한 추가 질문이 있으면 이동은 [토론 포럼](https://social.msdn.microsoft.com/forums/azure/en-US/home?forum=windowsazureonlinebackup) 여 질문을 게시 합니다. 커뮤니티의 구성원이 답변 도움이 됩니다. 자주 묻는 질문을 하는 경우 빠르고 쉽게 찾을 수 있도록이 문서에 추가 합니다 하겠습니다.

**2 분기 합니다. 최신 Azure 백업 에이전트를 어디서 다운로드할 수 있습니까?** <br/> A2 입력 합니다. 최신 에이전트에서 다운로드할 수 있습니다 [여기](http://aka.ms/azurebackup_agent). 이 Windows Server, SCDPM 서버 또는 Windows 클라이언트에 설치할 수 있습니다.

**Q 3입니다. SCDPM server 버전에 사용할 수 있습니까?** <br/> A3 합니다. 설치 하는 것이 좋습니다.는 [최신](http://aka.ms/azurebackup_agent) SCDPM의 최신 업데이트 롤업에 Azure 백업 에이전트 (UR5 2015 년 2 월을 기준으로)

**Q 4입니다. Azure 백업 에이전트를 구성할 때 "자격 증명 모음 자격 증명"을 입력 하 라는 메시지가 있습니다. 가 있는 모든 만료 날짜에 연결 된 보관 자격 증명?** <br/> A4 합니다. 예, 자격 증명 모음 자격 증명 48hours 지나면 만료 됩니다. 파일 만료 되 면 Azure 포털에 로그인 하 고 해당 백업 자격 증명 모음에서 자격 증명 모음 자격 증명 파일을 다운로드 합니다.

**Q5 합니다. 이 수가 각 Azure 구독에 만들 수 있는 백업 자격 증명 모음에 모든 제한이 있습니까?** <br/> A5 합니다. 예. 2015 년 3 월을 기준으로 구독 당 25 자격 증명 모음을 만들 수 있습니다. 그런 다음 더 많은 자격 증명 모음에 필요한 경우에 새 구독을 만듭니다.

**Q6 합니다. 찾아야 합니까 자격 증명 모음에 청구 엔터티로?** <br/> A6 합니다. 각 자격 증명 모음에 대 한 자세한 자재 명세서를 획득할 수 이지만, 대금 청구 엔터티로 Azure 구독을 고려 하는 것이 좋습니다. 이 모든 서비스에서 일치 하 고 관리 하기가 쉽습니다.

**Q7 합니다. 각 자격 증명 모음에 대해 등록할 수 있는 서버/컴퓨터의 수에 제한이 있습니까?** <br/> A7 합니다. 예, 자격 증명 모음 당 50 컴퓨터까지 등록할 수 있습니다. 더 많은 컴퓨터를 등록 해야할 경우 새 자격 증명 모음을 만듭니다.

**Q8 합니다. 이 Windows 서버/클라이언트 또는 SCDPM 서버에서 백업할 수 있는 데이터의 양에 제한이 있습니까?** <br/> A8 합니다. 번호

**Q9 합니다. 다른 데이터 센터로 내 서버를 등록 하는 방법**<br/> A9 합니다. 일반적으로 백업 데이터가 등록 되는 백업 서비스의 데이터 센터에 전송 됩니다. 데이터 센터를 변경 하는 가장 쉬운 방법은 하 에이전트를 제거 하 고 에이전트를 다시 설치 하 고 새 데이터 센터에 등록 됩니다.

**Q10 합니다. Azure에 데이터를 백업 하는 Windows 서버 이름을 바꿀 있습니까 있으면 어떻게 될까요?** <br/> A10 합니다. 현재 구성 된 모든 백업이 중지 됩니다. 백업 자격 증명 모음에 서버를 등록 해야 하 고 것으로 간주 됩니다 새 서버 복구 서비스에 의해 등록 후에 발생 하는 첫번째 백업 작업의 마지막 백업 이후에 변경 된 사항만 하는 대신 백업에 포함 된 데이터의 모든 전체 백업이 되도록 합니다. 그러나 복구 작업을 수행 해야하는 경우 복구를 사용 하 여 다른 서버 복구 옵션에서 백업 된 데이터를 복구할 수 있습니다. 자세한 내용은 서버 이름바꾸기를 참조 하십시오.

**Q11 합니다. 파일 및 폴더에서 어떤 종류의 드라이브를 백업할 수 있습니까?** <br/> A 11입니다. 다음 집합 드라이브/볼륨을 백업 될 수 있습니다.

+ 이동식 미디어: 되도록 고정 드라이브를 보고 해야 백업 항목 원본을 사용 합니다.

+ 읽기 전용 볼륨: 볼륨은 볼륨 섀도 복사본 서비스 (VSS) 함수에 대 한 쓰기 가능 해야 합니다.

+ 오프 라인 볼륨: 볼륨은 함수에 VSS에 대 한 온라인 상태 여야 합니다.

+ 네트워크 공유: 볼륨 백업할 온라인 백업을 사용 하 여 서버에 로컬 이어야 합니다.

**Q12 합니다. 어떤 파일 및 폴더 형식을 백업할 수 있습니까 내 서버에서?**<br/> A 12입니다. 다음과 같은 형식이 지원 됩니다.

+ 암호화

+ 압축

+ 스파스

+ 압축 + 스파스

+ 하드 링크: 지원, 생략 하지

+ 지원, 생략 하지 재분석 지점:

+ 암호화 + 압축: Not 지원, 생략

+ 암호화 + 스파스: Not 지원, 생략

+ 압축 된 스트림: 지원 하지 않거나, 생략

+ 스파스 스트림: 지원 하지 않거나, 생략

**Q13 합니다. 캐시 폴더에 대 한 최소 크기 요구 사항은 무엇입니까?** <br/> A13 합니다. 캐시 폴더의 크기를 백업 하는 데이터의 양에 따라 결정 됩니다. 일반적 캐시 폴더에 대 한 데이터 저장에 필요한 공간의 10 ~ 15%를 할당 되는 예상 해야 합니다.

**Q14 합니다. 특정 서버 조직 내에서 다른 서버에서 복구 된 데이터로 격리할 수 있습니까? **<br/> A14 합니다. 모든 등록 된 서버는 동일한 자격 증명 모음을 사용 하 여 동일한 암호를 사용 하는 다른 서버에서 백업 된 데이터를 복구할 수 없게 됩니다. 조직에서 특정 서버에 복구에만 발생 하도록 할 서버에 있으면 해당 서버에 대해 지정 된 별도 암호를 사용 해야 합니다. 예 인사 서버 하나 암호화 암호, 회계 서버와 저장소 서버는 제 3를 사용할 수 있습니다.

**Q15 합니다. "데이터 마이그레이션할 수 있습니까" 내 백업 구독 간에?** <br/> A 15: 아니요

**Q16: 마이그레이션할 수 있습니까 "" 구독 간에 내 백업 자격 증명 모음?** <br/> A16: 아니요. 자격 증명 모음 구독 수준에서 만들어지고 생성 되 면 다른 구독을 할당할 수 없습니다.

## 백업 및 보존
**Q 1입니다. 백업 중인 각 데이터 원본의 크기에 제한이 있네요?** <br/> A1 합니다. 2015 년 3 월을 기준으로 각 데이터 원본이 1.7 t B 보다 작거나 여야 합니다. 데이터 소스가

+ 파일/폴더 볼륨

+ SQL DB

+ Sharepoint 팜

+ Exchange server

+ Hyper-v VM

**2 분기 합니다. 하루에 백업을 예약할 수 있는 횟수에 제한이 있는?**<br/> A2 입력 합니다. 예, Azure Backup을 통해 Windows 서버/클라이언트를 통해 하루에 3 백업 복사본 및 SCDPM 통해 하루 2 백업 복사본.

**Q 3입니다. DPM의 간에 차이가 및 백업 일정 정책 Azure Backup의 (즉, DPM 없이 Windows 서버에서)?** <br/> A3 합니다. 예. DPM을 사용 하 여, 매일, 매주, 매월, 매년 (DPM) 없이 Windows 서버에서 작업 하는 동안 일정을 지정할 수 있습니다, 주별 일정 매일만 지정할 수 있습니다.

**Q 4입니다. 그 DPM의 Azure Backup의 (즉, DPM 없이 Windows 서버에서) 사이의 차이 보존 정책을?**<br/> A4 합니다. 아니요, 동일한 기능을 갖고 있습니다. 지정할 수 있습니다 매일, 매주, 매월 및 매년 보존 정책.

**Q5 합니다. 내 보존 정책 – 선택적으로, 즉 구성 매주 및 매일 하지만 하지 매년, 매월을 구성할 수 있습니까?**<br/> A5 합니다. 전체 집합을 가장 잘 규정 준수/보존 요구 사항을 정의 하는 정책을 세워야 노브를 해야 합니다.

**Q6 합니다. "오후 6 시에" 백업 예약 및 "보존 정책" 다른 시간에 지정 수 있습니까?**<br/> A6 합니다. 아니요. 보존 정책은 백업 지점에만 적용할 수 있습니다. 에 아래 그림을 보존 정책을 지정 되 오전 12 시와 오후 6 시에 수행 된 백업을 수행 합니다. <br/>

![백업 일정 및 보존][1] <br/>

**Q7 합니다. 예약 된 보존 정책에 대 한 전송 되는 증분 복사?** <br/> A7 합니다. 아니요, 증분 복사 백업 일정 페이지에 언급 된 시간에 따라 전송 됩니다. 보유할 수 있는 지점은 보존 정책에 따라 결정 됩니다.

**Q8 합니다. 백업 오랜 시간 동안 유지 되는지, 데 (예: 가장 오래 된 지점) 데이터를 복구 하는데 상당한 시간이?** <br/> A8 합니다. 아니요 – 복구 하는데 걸리는 시간 가장 오래 된 또는 최신 지점은 하 나와 동일 합니다. 각 복구 지점 전체 지점 처럼 동작합니다.

**Q9 합니다. 각 복구 지점이 전체 포인트와 같은 경우 그 영향 을지 않습니다 총 청구 가능 백업 저장소?**<br/> A9 합니다. 일반적인 장기 보존 지점 제품 전체 지점으로 백업 데이터를 저장합니다. 그러나 이러한은 저장소 비효율적인 있지만 쉽고 빠르게 복원할 수입니다. 증분 복사본은 효율적인 저장소 하지만 필요 하면 복구 시간에 영향을 주는 데이터의 체인을 복원할 수 있습니다. Azure 백업 고유 저장소 아키텍처 최적으로 빠른 복원에 대 한 데이터를 저장 하 고 낮은 저장소 비용이 발생 하 여의 장점 모두 제공 합니다. 이 방법을 사용 하면 프로그램 (송 / 수신) 대역폭을 효율적으로 사용 하는, 저장소 최소한으로 유지 됩니다 고 복구 하는데 걸리는 시간 최소한으로 유지 됩니다.

**Q10 합니다. 이 만들 수 있는 복구 지점의 수에 제한이 있습니까?**<br/> A10 합니다. 2015 년 4 월을 기준으로 366 복구 지점까지 가질 수 있습니다. 모든 순열 도착 하는 보다 작은 366은 하는 숫자를 사용할 수 있습니다. 보존 가리키는 예: –는 아래 그림 354에 추가 합니다. <br/>

![보존 화면][2]

**Q11 합니다. Microsoft 제한을 366에서 향상 되 면 됩니다 해야 에이전트를 업그레이드 하거나 다시 설정 하 여 초기 백업을?** <br/> A 11입니다. 아니요. 서비스, 변경 사항을 적용할 우리가 일단 우리의 소셜 미디어 콘센트 (블로그, Azure 공지 포털 등)를 통해 알림이 표시 됩니다. 필요에 따라, 있습니다 다음 해야만 보존 정책을 변경 합니다.

**Q12 합니다. 데이터의 양은 있습니까 백업 데이터의 양을 같지 않은 백업에 전송 되는 이유**<br/> A 12입니다. 백업 된 모든 데이터의 압축 및 전송 하기 전에 암호화 됩니다. 백업 중인 데이터의 유형에 따라 30-40% 압축 이점을 기대할 수 있습니다.

## 복구
**Q 1입니다. Azure에 백업 하는 데이터에 대해 수행할 수 어떻게 많은 복구**<br/> A1 합니다. Azure 백업에서 복구의 수에 제한이 있습니다.

**2 분기 합니다. 복구 하는 동안 Azure 데이터 센터에서 송신 트래픽에 요금을 지불 해야 합니까?**<br/> A2 입력 합니다. 아니요. 프로그램 복구는 무료 및 송신 트래픽에 대 한 요금이 부과 되지 않습니다.

## 보안
**Q 1입니다. 암호화는 Azure에 데이터를 전송 하 시겠습니까?** <br/> A1 합니다. 예. A e s 256을 사용 하 여 온-프레미스 서버/클라이언트/SCDPM 컴퓨터에서 데이터를 암호화 하 고 데이터 보안 HTTPS 연결을 통해 전송 됩니다.

**2 분기 합니다. 암호화 된 상태로 Azure에 백업 데이터가 포함 되어 있습니까?**<br/> A2 입력 합니다. 예. Azure에 전송 되는 데이터 (휴지 상태) 암호화 상태를 유지 합니다. Microsoft는 언제 든 지 백업 데이터를 해독 하지 않습니다.

**Q 3입니다. 백업 데이터를 암호화 하는데 사용 되는 암호화 키의 최소 길이 무엇입니까?** <br/> A3 합니다. 암호화 키에는 적어도 16 자 여야 합니다.

**Q 4입니다. 암호화 키를 잃어버리기 합니까 경우 어떻게 됩니까? 데이터는 복구할 수 있습니다 (또는) Microsoft의 데이터를 복구할 수 있습니까?** <br/> A4 합니다. 백업 데이터를 암호화 하는데 사용 되는 키가 고객 사업장에만 존재 합니다. Microsoft은 Azure의 복사본을 유지 하지 않습니다 있고 모든 키에 액세스할 수 없습니다. 고객 misplaces 키, Microsoft 백업 데이터를 복구할 수 없습니다.
