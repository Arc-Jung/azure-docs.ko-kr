# <a name="subnet-extension"></a>서브넷 확장
퍼블릭 클라우드로의 워크로드 마이그레이션에는 신중한 계획과 조정이 필요합니다. 주요 고려 사항 중 하나는 IP 주소를 유지하는 기능일 수 있습니다. 특히 응용 프로그램에 IP 주소 종속성이 있거나 특정 IP 주소를 사용하기 위한 규정 준수 요구 사항이 있는 경우 특히 중요할 수 있습니다. Azure 가상 네트워크는 원하는 IP 주소 범위를 사용하여 VNet 및 서브넷을 만들 수 있도록 하여 이 문제를 해결합니다.

위의 요구 사항이 일부 응용 프로그램을 온-프레미스에 유지하기 위한 추가 요구 사항과 결합될 경우 마이그레이션은 다소 어려울 수 있습니다. 이러한 상황에서는 양쪽의 IP 주소에 다시 번호를 매기지 않고 응용 프로그램을 Azure와 온-프레미스 간에 분할해야 합니다. 또한 응용 프로그램이 동일한 네트워크에 있는 것처럼 통신할 수 있도록 허용해야 합니다.

위의 문제에 대한 한 가지 해결책은 서브넷 확장입니다. 네트워크를 확장하면 응용 프로그램이 서로 다른 물리적 위치에 있을 때 동일한 브로드캐스트 도메인을 통해 대화할 수 있으므로 네트워크 토폴로지의 재설계를 수행할 필요가 없습니다. 

네트워크를 확장하는 것이 일반적인 방법은 아니지만 아래 사용 사례에서는 필요할 수 있습니다.

- **단계별 마이그레이션**: 가장 일반적인 시나리오는 마이그레이션을 단계적으로 수행하려는 것입니다. 몇 가지 응용 프로그램을 먼저 가져오고 시간이 지남에 따라 나머지 응용 프로그램을 Azure로 마이그레이션하려고 합니다.
- **대기 시간**: 대기 시간이 짧은 요구 사항은 일부 응용 프로그램을 온-프레미스에 보관하여 데이터 센터에 최대한 가깝게 유지해야 하는 또 다른 이유가 될 수 있습니다.
- **규정 준수**: 또 다른 사용 사례는 일부 응용 프로그램을 온-프레미스에 유지하기 위한 규정 준수 요구 사항이 있을 수 있다는 것입니다.
 
> [!NOTE] 
> 필요한 경우가 아니면 서브넷을 확장해서는 안 됩니다. 서브넷을 확장하는 경우 중간 단계로 만들어야 합니다. 시간이 지남에 따라 온-프레미스 네트워크에서 응용 프로그램 번호를 다시 시도하고 Azure로 마이그레이션해야 합니다.

다음 섹션에서는 서브넷을 Azure로 확장하는 방법에 대해 설명합니다.


## <a name="extend-your-subnet-to-azure"></a>서브넷을 Azure로 확장
 계층 3 오버레이 네트워크 기반 솔루션을 사용하여 온-프레미스 서브넷을 Azure로 확장할 수 있습니다. 대부분의 솔루션은 VXLAN과 같은 오버레이 기술을 사용하여 계층 3 오버레이 네트워크를 사용하여 계층 2 네트워크를 확장합니다. 아래 다이어그램은 일반화된 솔루션을 보여 주며 있습니다. 이 솔루션에서는 Azure 및 온-프레미스라는 양쪽에 동일한 서브넷이 존재합니다. 

![서브넷 확장 예제](./media/subnet-extension/subnet-extension.png)

서브넷의 IP 주소는 Azure 및 온-프레미스의 VM에 할당됩니다. Azure와 온-프레미스 모두에 NVA가 네트워크에 삽입되어 있습니다. Azure의 VM이 온-프레미스 네트워크에서 VM과 통신하려고 하면 Azure NVA는 패킷을 캡처하고 캡슐화한 다음 VPN/익스프레스 라우팅을 통해 온-프레미스 네트워크로 보냅니다. 온-프레미스 NVA는 패킷을 수신하고 캡슐화하여 네트워크의 의도한 받는 사람에게 전달합니다. 반환 트래픽은 유사한 경로와 논리를 사용합니다.

위의 예에서 Azure NVA 및 온-프레미스 NVA통신 및 서로 뒤에 있는 IP 주소에 대해 알아봅니다. 더 복잡한 네트워크에는 NAS와 그 뒤에 있는 IP 주소 간의 매핑을 유지하는 매핑 서비스가 있을 수도 있습니다. NVA가 패킷을 수신하면 매핑 서비스를 쿼리하여 그 뒤에 대상 IP 주소가 있는 NVA의 주소를 찾습니다.

다음 섹션에서는 Azure에서 테스트한 서브넷 확장 솔루션에 대한 자세한 내용을 확인할 수 있습니다.
