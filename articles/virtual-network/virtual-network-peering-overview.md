# VNet 피어링

VNet 피어링은 Azure 백본 네트워크를 통해 동일한 지역에 있는 두 개의 가상 네트워크를 연결하는 메커니즘입니다. 두 가상 네트워크가 피어링되면 모든 연결에 대해 하나처럼 표시됩니다. 여전히 별도 리소스로 관리할 수는 있지만 이러한 VNet의 가상 컴퓨터는 개인 IP 주소를 사용하여 직접 서로 통신할 수 있습니다. 동일한 VNet에 있는 VM 간의 트래픽과 마찬가지로 피어링된 VNet에 있는 가상 컴퓨터 간의 트래픽은 Azure 인프라를 통해 라우팅됩니다. VNet 피어링을 사용하는 이점은 다음과 같습니다.

- 다른 VNet에 있는 리소스 간에 짧은 대기 시간, 높은 대역폭 연결
- 피어링된 VNet\(전송\)에서 네트워크 가상 어플라이언스, VPN 게이트웨이와 같은 리소스 사용 가능
- 기존 VNet에 Resource Manager VNet을 연결하고 이러한 VNet에 있는 리소스 간의 전체 연결 가능

VNet 피어링의 요구 사항 및 주요 측면은 다음과 같습니다.

- 피어링된 두 개의 가상 네트워크는 동일한 Azure 지역에 있어야 합니다.
- 피어링된 VNet에는 겹치지 않는 IP 주소 공간이 있어야 합니다.
- VNet 피어링은 두 가상 네트워크 간을 연결하지만 파생된 전이적 관계가 없습니다. 즉, VNet A가 VNet B와 피어링되고 VNet B가 VNet C로 피어링되더라도 VNet A가 VNet C와 피어링되었다고 해석하지 않습니다.
- 해당하는 구독의 권한있는 사용자가 피어링을 허용하면 서로 다른 두 구독에 있는 가상 네트워크 간에 피어링을 설정할 수 있습니다.
- 하나의 Resource Manager VNet은 다른 Resource Manager VNet 또는 기존 VNet과 피어링될 수 있지만 두 기존 VNet은 서로 피어링될 수 없습니다.
- 피어링된 VNet에 있는 가상 컴퓨터 간의 통신에 대역폭 제한이 추가되지 않더라도 VM 크기에 따른 대역폭 상한이 여전히 적용됩니다.

## 연결 
두 개의 VNet이 피어링되면 VNet에 있는 가상 컴퓨터\(웹/작업자 역할\)는 피어링된 VNet의 다른 가상 컴퓨터와 직접 연결할 수 있습니다. 전체 IP 수준 연결이 생성됩니다. 피어링된 VNet에 있는 두 가상 컴퓨터 간의 왕복 네트워크 대기 시간은 로컬 VNet내에서와 동일합니다. 네트워크 처리량은 크기에 비례하여 가상 컴퓨터에 허용된 대역폭에 따라 달라 집니다. 허용되는 대역폭에 대한 추가 제한 사항은 없습니다. 피어링된 VNet에 있는 가상 컴퓨터 간의 트래픽은 게이트웨이가 아닌 Azure의 백 엔드 인프라를 통해 직접 라우팅됩니다.

VNet의 가상 컴퓨터는 피어링된 VNet에 있는 ILB\(내부 부하 분산 끝점\)에 액세스할 수 있습니다. 네트워크 보안 그룹은 다른 VNet에 대한 액세스를 차단하는 VNet 또는 원하는 경우 서브넷에 적용될 수 있습니다. 사용자가 피어링을 구성하는 경우 VNet 간의 네트워크 보안 그룹 규칙을 열거나 닫도록 선택합니다. 사용자가 피어링된 VNet 간의 전체 연결을 열도록 선택하는 경우\(기본 옵션\) 특정 서브넷 또는 가상 컴퓨터에 NSG를 사용하여 특정 액세스를 차단하거나 거부할 수 있습니다.

Azure에서 제공한 가상 컴퓨터에 대한 내부 DNS 이름 확인은 피어링된 VNet에 작동하지 않습니다. 가상 컴퓨터는 로컬 가상 네트워크 내에서만 확인할 수 있는 내부 DNS 이름을 갖게 됩니다. 그러나 사용자는 피어링된 VNet에서 실행 중인 가상 컴퓨터를 가상 네트워크에 대한 DNS 서버로 구성할 수 있습니다.

## 서비스 체이닝
아래 다이어그램처럼 피어링된 VNet에서 가상 컴퓨터를 다음 홉으로 가리키는 사용자 정의 경로 테이블을 구성할 수 있습니다. 이렇게 하면 사용자 정의 경로 테이블을 통해 하나의 VNet에서 피어링된 VNet에서 실행되는 가상 어플라이언스로 트래픽을 보낼 수 있는 서비스 체이닝을 설정할 수 있습니다. 사용자는 허브 및 스포크 유형의 환경을 효율적으로 만들 수 있습니다. 여기서 허브는 네트워크 가상 어플라이언스와 같은 인프라 구성 요소를 호스팅할 수 있고 모든 스포크 VNet은 해당 요소와 피어링하고 허브 VNet에서 실행되는 어플라이언스에 대한 트래픽의 하위 집합을 직접 전달할 수 있습니다. 즉, VNet 피어링을 사용하면 ‘사용자 정의 경로 테이블’에 대한 다음 홉 IP 주소는 피어링된 VNet에 있는 가상 컴퓨터의 IP 주소가 될 수 있습니다.

## 게이트웨이 및 온-프레미스 연결
각 가상 네트워크는 다른 VNet과 피어링되었는지 여부와 상관없이 고유한 게이트웨이를 가지고 있으며 이를 온-프레미스에 연결하는 데 사용할 수 있습니다. 사용자는 VNet이 피어링된 경우에도 게이트웨이를 사용하여 VNet 간 연결을 구성할 수도 있습니다\(링크 제공\). VNet 간 연결을 위한 옵션을 모두 구성한 경우 VNet 간의 트래픽은 피어링 구성\(즉, Azure 백본\)을 통해 이동합니다.

VNet이 피어링되면 사용자도 피어링된 VNet에 있는 게이트웨이를 온-프레미스에 대한 전송 지점으로 사용하도록 구성할 수 있습니다. 이 경우에 원격 게이트웨이를 사용하는 VNet에는 자체적인 게이트웨이가 있을 수 없습니다. 즉, 하나의 VNet에는 하나의 게이트웨이만이 있을 수 있으며 아래 그림에 표시된 것처럼 피어링된 VNet에는 로컬 게이트웨이 또는 원격 게이트웨이 중 하나가 있을 수 있습니다. 게이트웨이 전송은 Resource Manager 및 기존 VNet 간에는 지원되지 않습니다. 게이트웨이 전송이 작동하기 위해 피어링 관계인 두 VNet은 Resource Manager VNet이어야 합니다. 단일 ER 회로를 공유하는 VNet이 피어링된 경우 둘 사이의 트래픽은 피어링 관계\(즉, Azure 백본 네트워크\)를 거치게 됩니다. 사용자는 각 VNet에서 로컬 게이트웨이를 사용하여 온-프레미스 회로에 연결하거나 공유 게이트웨이를 사용하고 온-프레미스 연결에 대한 전송을 구성할 수 있습니다.

## 프로비전
VNet 피어링은 권한 있는 작업입니다. 가상 네트워크 네임스페이스의 별도 함수입니다. 사용자는 특정 권한을 지정하여 피어링하는 권한을 부여할 수 있습니다. VNet에 대한 읽기/쓰기 액세스 권한이 있는 사용자는 이를 자동으로 상속합니다. 피어링 기능의 관리자 또는 권한있는 사용자 중 하나인 사용자는 다른 VNet에 피어링 작업을 시작할 수 있습니다. 다른 쪽에 피어링에 대해 동일한 요청이 있고 다른 요구 사항을 충족하는 경우 피어링이 설정됩니다.

두 가상 네트워크 간의 VNet 피어링을 설정하는 방법에 대해 자세히 알아보려면 방법 문서를 참조하세요.

## 제한
단일 가상 네트워크에 허용되는 피어링의 수가 제한되어 있습니다. 자세한 내용은 [Azure 네트워킹 제한](../azure-subscription-service-limits.md#networking-limits)을 참조하세요.

## 가격
VNet 피어링은 검토 기간 동안 요금이 청구되지 않습니다. 일반 공급을 위해 출시되면 피어링을 활용하는 송/수신 트래픽에 명목 요금이 부과됩니다. 자세한 내용은 [가격 책정 페이지](https://azure.microsoft.com/pricing/details/virtual-network)를 참조하세요.
