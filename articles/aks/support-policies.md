# <a name="support-policies-for-azure-kubernetes-service"></a>Azure Kubernetes Service에 대 한 지원 정책

이 문서에서는 기술 지원 정책 및 제한 사항에 대 한 Azure Kubernetes Service (AKS)에 대 한 정보를 제공 합니다. 이 문서에는 작업자 노드 관리, 관리 되는 제어 평면 구성 요소, 타사 오픈 소스 구성 요소 및 보안 또는 패치 관리도 자세히 설명합니다.

## <a name="service-updates-and-releases"></a>서비스 업데이트 및 릴리스

* 릴리스 정보 참조 [AKS에 릴리스](https://github.com/Azure/AKS/releases)합니다.
* 미리 보기의 기능에 대 한 자세한 내용은 [기능 및 관련된 프로젝트 AKS 미리 보기](https://github.com/Azure/AKS/blob/master/previews.md)합니다.

## <a name="managed-features-in-aks"></a>AKS에서 기능 관리

기본 인프라 서비스 (IaaS) 클라우드 구성 요소를, 계산 및 네트워킹 구성 요소, 등으로 사용자 액세스 하위 수준 컨트롤 및 사용자 지정 옵션을 제공 합니다. 반면, AKS는 고객에 게 필요한 기능과 구성의 공통 집합을 제공 하는 턴키 Kubernetes 배포를 제공 합니다. 사용자 지정, 배포 및 기타 옵션 AKS 고객에 게 제한 됩니다. 이러한 고객에 걱정 하거나 Kubernetes 클러스터를 직접 관리할 필요가 없습니다.

AKS를 통해 고객 가져옵니다 완전 관리형 *제어 평면*합니다. 제어 평면 모든 구성 요소 및 고객이 작동 하 여 최종 사용자에 게 Kubernetes 클러스터를 제공 해야 하는 서비스를 포함 합니다. Kubernetes에 대 한 구성 요소를 모두 유지 관리 되며 Microsoft에서 운영 됩니다.

Microsoft는 관리 하 고 컨트롤 창을 통해 다음 구성 요소를 모니터링 합니다.

* Kubelet 또는 Kubernetes API 서버
* Etcd 또는 서비스 품질 (QoS), 확장성 및 런타임 제공 호환 키-값 저장소
* DNS 서비스 (예를 들어, kube-dns 또는 CoreDNS)
* Kubernetes 프록시 또는 네트워킹

AKS는 클러스터를 완전히 관리 되는 솔루션을 하지 않습니다. 작업자 노드 등의 일부 구성 요소 *공동 책임*, 여기서 사용자 AKS 클러스터를 유지 관리할 수 있어야 합니다. 예를 들어, 사용자 입력을 작업자 노드의 운영 체제 (OS) 보안 패치를 적용 하려면 필요 합니다.

서비스가 *관리 되는* 에 Microsoft와 AKS 팀 배포, 작동, 서비스 가용성 및 기능을 담당 합니다. 고객은 이러한 관리 되는 구성 요소를 변경할 수 없습니다. Microsoft는 일관적이 고 확장성이 뛰어난 사용자 환경을 보장 하려면 사용자 지정을 제한 합니다. 완전히 사용자 지정 가능한 솔루션을 참조 하세요 [AKS 엔진](https://github.com/Azure/aks-engine)합니다.

> [!NOTE]
> AKS 작업자 노드 Azure portal에서 Azure IaaS 리소스를 정기적으로 나타납니다. 되지만 이러한 가상 머신을 사용자 지정 Azure 리소스 그룹에 배포 됩니다 (MC 접두사로\\*). AKS 작업자 노드를 변경 하는 것이 가능 합니다. 그러나 예를 들어, AKS 작업자 노드의 일반 가상 컴퓨터를 변경 하는 방법을 변경 하려면 SSH (보안 셸)를 사용할 수 있습니다 (기본 OS 이미지를 변경할 수 없는 및 변경 내용 업데이트를 통해 유지 되지 수도 다시 부팅) 및 AKS를 다른 Azure 리소스를 연결할 수 있습니다 작업자 노드입니다. 변경을 수행 하면 되지만 *대역 외 관리 및 사용자 지정* AKS 클러스터 지원 될 수 있습니다. Microsoft 지원 변경할 수 있도록 안내 하는 경우가 아니면 작업자 노드를 변경 하지 마십시오.

## <a name="shared-responsibility"></a>공동 책임

클러스터를 만들 때 고객이 만든 AKS Kubernetes 작업자 노드를 정의 합니다. 고객 워크 로드는 이러한 노드에서 실행 됩니다. 고객 소유 및 보거나 수정할 수 있는 작업자 노드.

고객 클러스터 노드 전용 코드를 실행 하 고 중요 한 데이터를 저장 하기 때문에 Microsoft 지원만 제한 된 방식으로를 액세스할 수 있습니다. Microsoft 지원에서 명령을 실행, 또는 express 고객 권한 또는 지원 하지 않고 이러한 노드에 대 한 로그를 보려면 로그인 할 수 없습니다.

작업자 노드는 중요 한 이기 때문에 Microsoft는 해당 배경 관리 제한에 주의 합니다. 대부분의 경우 워크 로드 Kubernetes 마스터 노드, etcd, 및 기타 Microsoft에서 관리 하는 구성 요소 실패 하는 경우에 실행을 계속 됩니다. 부주의 수정 된 작업자 노드는 데이터 및 워크 로드의 손실 될 수 있습니다 및 클러스터 못했고 렌더링할 수 있습니다.

## <a name="aks-support-coverage"></a>AKS 검사를 지원합니다.

Microsoft는 다음에 대 한 기술 지원을 제공 합니다.

* Kubernetes 서비스를 제공 하 고 지 원하는, API 서버와 같은 모든 Kubernetes 구성 요소에 연결 합니다.
* 관리, 가동 시간, QoS 및 Kubernetes 작업 평면 서비스 (Kubernetes 마스터 노드, API 서버, etcd, kube-예를 들어 dns)를 제어 합니다.
* Etcd 합니다. 지원 계획 및 클러스터 상태 복원이 재해에 대 일 분 마다 모든 etcd 데이터의 투명 한 자동화 된 백업에 포함 됩니다. 이러한 백업은 고객 또는 사용자에 게 직접 사용할 수 없습니다. 데이터 안정성과 일관성 확인합니다.
* Kubernetes 용 Azure 클라우드 공급자 드라이버에서 모든 통합 지점입니다. 여기에 통합 부하 분산 장치, 영구적 볼륨 또는 네트워킹 (Kubernetes 및 Azure CNI) 등 다른 Azure 서비스에 포함 됩니다.
* Kubernetes API 서버, 등 etcd, kube-dns 제어 평면 구성 요소의 사용자 지정 하는 방법에 대 한 문제 또는 질문 합니다.
* 네트워킹, Azure CNI, kubenet, 또는 다른 네트워크 액세스 및 기능 같은 문제에 대 한 문제입니다. 문제는 DNS 확인, 패킷 손실, 라우팅 및에 포함 될 수 있습니다. Microsoft는 다양 한 네트워킹 시나리오를 지원합니다.
  * (기본) Kubenet 클러스터 내에서 고급 네트워킹이 Azure CNI () 및 관련 된 구성 요소
  * 다른 Azure 서비스 및 응용 프로그램에 대 한 연결
  * 수신 컨트롤러 및 수신 또는 부하 분산 장치 구성
  * 네트워크 성능 및 대기 시간

Microsoft는 다음에 대 한 기술 지원을 제공 하지 않습니다.

* Kubernetes를 사용 하는 방법에 대 한 질문입니다. 예를 들어, Microsoft 지원 사용자 지정 수신 컨트롤러를 만들기, 응용 프로그램 워크 로드를 사용 하 여 또는 타사 또는 오픈 소스 소프트웨어 패키지 또는 도구를 적용 하는 방법에 통지를 제공 하지 않습니다.
  > [!NOTE]
  > Microsoft 기술 지원 서비스는 AKS 클러스터 기능, 사용자 지정 및 조정 (예: Kubernetes 작업 문제 및 프로시저)에 대해 advise 할 수 있습니다.
* Kubernetes의 일부로 제공 되지 않는 타사 오픈 소스 프로젝트 제어 평면 또는 AKS 클러스터를 사용 하 여 배포 합니다. 이러한 프로젝트는 Istio, Helm, Envoy 또는 다른 사용자가 포함 될 수 있습니다.
  > [!NOTE]
  > Microsoft는 Helm Kured 등의 타사 오픈 소스 프로젝트에 대 한 최선의 지원을 제공할 수 있습니다. Kubernetes Azure 클라우드 공급자 또는 다른 AKS 관련 버그를 통합 하는 타사 오픈 소스 도구, 여기서 Microsoft 예제 및 Microsoft 설명서에서 응용 프로그램을 지원 합니다.
* 타사 닫힌 소스 소프트웨어입니다. 이 소프트웨어는 보안 검사 도구 및 네트워킹 장치나 소프트웨어에 포함할 수 있습니다.
* 다중 또는 다중 공급 업체 구축 된 환경이 문제를 다룹니다. 예를 들어, Microsoft는 페더레이션된 multipublic 클라우드 공급 업체 솔루션 실행과 관련 된 문제를 지원 하지 않습니다.
* 네트워크에 나열 되지 않은 사용자 지정 된 [AKS 설명서](https://docs.microsoft.com/azure/aks/)합니다.
  > [!NOTE]
  > Microsoft는 문제 및 네트워크 보안 그룹 (Nsg)와 관련 된 버그가 지원 됩니다. 예를 들어, Microsoft 지원 업데이트 하는 NSG 오류나 예기치 않은 NSG 또는 부하 분산 장치 동작을 하는 방법에 대 한 질문에 대답할 수 있습니다.

## <a name="aks-support-coverage-for-worker-nodes"></a>AKS는 작업자 노드에 대 한 검사를 지원

### <a name="microsoft-responsibilities-for-aks-worker-nodes"></a>AKS 작업자 노드에 대 한 Microsoft의 책임

Microsoft와 고객은 Kubernetes 작업자 노드에 대 한 책임을 공유 위치:

* 기본 OS 이미지 추가 (예: 모니터링 및 에이전트 네트워킹) 필요한입니다.
* 작업자 노드의 OS 패치를 자동으로 수신 합니다.
* Kubernetes 사용 하 여 문제 제어 평면 작업자 노드에서 실행 되는 구성 요소가 자동으로 수정 됩니다. 구성 요소는 다음과 같습니다.
  * kube-프록시
  * Kubernetes를 통신 경로 제공 하는 네트워킹 터널 마스터 구성 요소
  * kubelet
  * Docker 또는 모 비 데몬

> [!NOTE]
> 작업자 노드에서 제어 평면 구성 요소에서 작동 하지 않습니다. AKS 팀이 전체 작업자 노드를 다시 부팅 해야 할 수 있습니다. 고객의 현재 워크 로드 및 데이터에 해당 제한 된 액세스로 인해 AKS 팀 고객 문제를 에스컬레이션 하는 경우에 작업자 노드를 재부팅 합니다. 가능 하면 응용 프로그램에 영향을 주는에서 필요한 다시 부팅을 방지 하기 위해 AKS 팀 작동 합니다.

### <a name="customer-responsibilities-for-aks-worker-nodes"></a>AKS 작업자 노드에 대 한 고객의 책임

Microsoft OS 수준 패치를 적용 하는 작업자 노드를 자동으로 재부팅 하지 않습니다. OS 패치를 작업자 노드에 전달 됩니다 있지만 합니다 *고객* 작업자 노드의 변경 내용을 적용 하려면 다시 부팅을 담당 합니다. 공유 라이브러리, 하이브리드 반도체 드라이브 (SSHD)와 같은 디먼 및 기타 구성 요소는 시스템이 나 OS 수준에서 자동으로 패치 됩니다.

고객은 Kubernetes 업그레이드를 실행 하는 일을 담당 합니다. Azure 제어판 또는 Azure CLI를 통해 업그레이드를 실행할 수 있습니다. 이 보안 또는 Kubernetes에 향상 된 기능을 포함 하는 업데이트에 적용 됩니다.

> [!NOTE]
> AKS는 때문에 *관리 되는 서비스*, 최종 목표 제거 패치, 업데이트에 대 한 책임을 포함 하 고 로그 보다 완벽 하 고 자동화 된 방식으로 서비스 관리를 확인 하는 컬렉션입니다. 종단 간 관리를 위한 서비스의 용량 증가 하면 향후 일부 함수 (예: 노드 다시 부팅 및 자동 패치)를 생략할 수 있습니다.

### <a name="security-issues-and-patching"></a>보안 문제 및 패치

AKS의 하나 이상의 구성 요소에 보안 결함이 있으면 AKS 팀은 문제를 완화 하려면 영향을 받는 모든 클러스터 패치 됩니다. 또는 팀 사용자 업그레이드 지침은 제공 됩니다.

작업자 노드 보안 결함 영향 경우 가동 패치를 사용할 수 있는지에 대 한 AKS 팀 해당 패치를 적용 되며 변경 내용 사용자에 게 알립니다.

보안 패치를는 작업자 노드 재부팅을 필요한 경우 Microsoft는이 요구 사항의 고객에 게를 알립니다. 고객은 다시 부팅 하거나 가져오기 클러스터 패치를 업데이트 하는 일을 담당 합니다. 사용자는 AKS 지침에 따라 패치를 적용 하지 않는 경우에 해당 클러스터 보안 문제에 취약 하 게 계속 됩니다.

### <a name="node-maintenance-and-access"></a>노드 유지 관리 및 액세스

작업자 노드 책임 지는 되며 고객에 의해 소유 됩니다. 이 인해 고객은 해당 작업자 노드에 로그인 하 여 커널 업데이트 및 설치 하거나 제거 하는 패키지와 같은 잠재적으로 해로운 변경 수가 있습니다.

고객은 안전 하지 않은 변경 또는 결과 제어 평면 구성 요소는 오프 라인으로 전환 하거나 작동 하지 않을 경우 AKS이이 오류를 감지 하 고 자동으로 작업자 노드를 이전 작동 상태로 복원 합니다.

고객에 로그인 하 고 작업자 노드를 변경할 수 있습니다, 있지만 이렇게 권장 되지 않습니다 변경 못했고 클러스터를 만들 수 있으므로 합니다.

## <a name="network-ports-access-and-nsgs"></a>네트워크 포트, 액세스 및 Nsg

관리 되는 서비스로 AKS는 특정 네트워킹 및 연결 요구 사항이 있습니다. 이러한 요구 사항은 일반 IaaS 구성 요소에 대 한 요구 사항 보다 덜 효율적입니다. AKS에서 작업 (예: 443 아웃 바운드 포트를 차단 하는 방화벽 규칙을 사용 하 여), 특정 포트를 차단 하는 NSG 규칙을 사용자 지정 등의 허용 목록 Url 못했고 클러스터를 만들 수 있습니다.

> [!NOTE]
> 현재 AKS 클러스터 (예를 들어 명시적 도메인 또는 포트가 허용 목록)에서 송신을 완전히 잠글 수 있습니다를 허용 하지 않습니다. Url 및 포트의 목록은 경고 없이 변경 될 수 있습니다. Azure 지원 티켓을 만들면 업데이트 된 목록을 가져올 수 있습니다. 해당 클러스터 가용성을 저하 될 수 있습니다 감수 하는 고객에 대해서만 목록은 *언제 든 지 합니다.*

## <a name="unsupported-alpha-and-beta-kubernetes-features"></a>지원 되지 않는 알파 및 베타 Kubernetes 기능

AKS는 업스트림 Kubernetes 프로젝트 내에서 안정적인 기능만 지원합니다. 문서화이 고, 그렇지 않으면 AKS 업스트림 Kubernetes 프로젝트에서 사용할 수 있는 알파 및 베타 기능을 지원 하지 않습니다.

두 가지 시나리오에서 알파 또는 베타 기능 수를 롤아웃할 수 일반 공급 되기 전에:

* 고객은 AKS 제품, 지원 또는 엔지니어링 팀을 사용 하 여 충족 하며 이러한 새 기능을 시도 하 라는 메시지가 표시 되었습니다.
* 이러한 기능은 되었습니다 [기능 플래그를 설정한](https://github.com/Azure/AKS/blob/master/previews.md)합니다. 고객에 게 명시적으로 이러한 기능을 사용 하려면 옵트인 해야 합니다.

## <a name="preview-features-or-feature-flags"></a>미리 보기 기능 또는 기능 플래그

기능 및 확장 테스트 해야 하는 기능 및 사용자 피드백을 Microsoft 새로운 미리 보기 기능 또는 기능 플래그를 통해 기능을 해제 합니다. 시험판 또는 베타 기능으로 이러한 기능을 고려 합니다.

미리 보기 기능 또는 기능 플래그 기능 프로덕션을 위한 되지 않습니다. Api 및 동작, 버그 수정 및 기타 변경 내용에 대 한 지속적인 변화는 불안정 한 클러스터 가동 중지 시간에 발생할 수 있습니다.

공개 미리 보기에서 기능은 '최상의' 지원 포함이 기능이 미리 보기로 제공 됩니다 하지 프로덕션에 사용 하려는 및 업무 시간 동안 AKS 기술 지원 팀에서 지원 됩니다. 자세한 내용은 다음을 참조 하세요.

* [Azure 지원 FAQ](https://azure.microsoft.com/en-us/support/faq/)

> [!NOTE]
> 미리 보기 기능을 Azure에 적용 *구독* 수준입니다. 프로덕션 구독에 미리 보기 기능을 설치 하지 마세요. 프로덕션 구독에서 미리 보기 기능 기본 API 동작을 변경 하 고 일반 작업에 영향을 줄 수 있습니다.

## <a name="upstream-bugs-and-issues"></a>업스트림 버그 및 문제

업스트림 Kubernetes 프로젝트의 개발 속도 지정 합니다 버그 항상 발생 하지 않습니다. 이러한 버그 중 패치 이거나 안 AKS 시스템 내에서 해결 되었습니다. 대신, 버그 수정 (예: Kubernetes, 노드 또는 작업자 운영 체제 커널) 업스트림 프로젝트에 더 큰 패치 필요합니다. Microsoft (예: Azure 클라우드 공급자)를 소유 하는 구성 요소에 대 한 AKS 및 Azure 담당자는 커뮤니티에서 업스트림 문제를 해결 합니다.

기술 지원 문제를 하나 이상의 업스트림 버그 하 여 근본 원인이 파악 되 면 AKS 지원 및 엔지니어링 팀이 됩니다.

* 식별 하 고이 문제에 클러스터 또는 워크 로드에 영향을 줍니다 이유를 설명 하는 데 지원 세부 정보를 사용 하 여 업스트림 버그를 링크 합니다. 고객 문제를 조사 하 고 새 릴리스는 픽스를 제공 하는 경우를 볼 수 있도록 필요한 저장소에 대 한 링크를 받습니다.
* 잠재적인 해결 방법이 나 완화 방법을 제공 합니다. 문제를 완화할 수 있습니다 하는 경우는 [알려진 문제](https://github.com/Azure/AKS/issues?q=is%3Aissue+is%3Aopen+label%3Aknown-issue) AKS 리포지토리에서 정리할 수 있도록 합니다. 알려진 문제 신고를 설명합니다.
  * 문제를 업스트림 버그에 대 한 링크를 포함 합니다.
  * 해결 방법 및 업그레이드 또는 다른 정보를 솔루션의 지 속성입니다.
  * 타임 라인 포함할 문제의 업스트림 릴리스 작업 리듬에 따라 대략적인 합니다.
