# <a name="support-policies-for-azure-kubernetes-service"></a>Azure Kubernetes 서비스에 대한 지원 정책

이 문서에서는 AKS(Azure Kubernetes Service)에 대한 기술 지원 정책 및 제한 사항에 대한 자세한 내용을 제공합니다. 또한 이 문서에서는 작업자 노드 관리, 관리되는 제어 평면 구성 요소, 타사 오픈 소스 구성 요소 및 보안 또는 패치 관리에 대해서도 자세히 설명합니다.

## <a name="service-updates-and-releases"></a>서비스 업데이트 및 릴리스

* 릴리스 정보는 [AKS 릴리스 정보를](https://github.com/Azure/AKS/releases)참조하십시오.
* 미리 보기의 기능에 대한 자세한 내용은 [AKS 미리 보기 기능 및 관련 프로젝트를](https://github.com/Azure/AKS/blob/master/previews.md)참조하십시오.

## <a name="managed-features-in-aks"></a>AKS에서 관리되는 기능

컴퓨팅 또는 네트워킹 구성 요소와 같은 IaaS(클라우드 구성 요소)의 기본 인프라를 통해 사용자는 낮은 수준의 제어 및 사용자 지정 옵션에 액세스할 수 있습니다. 이와 대조적으로 AKS는 고객에게 필요한 구성및 기능의 공통 집합을 제공하는 턴키 Kubernetes 배포를 제공합니다. AKS 고객은 사용자 지정, 배포 및 기타 옵션이 제한되어 있습니다. 이러한 고객은 Kubernetes 클러스터에 대해 직접 걱정하거나 관리할 필요가 없습니다.

AKS를 사용하면 고객은 완전히 관리되는 *제어 비행기를 얻습니다.* 제어 평면에는 고객이 운영하는 데 필요한 모든 구성 요소와 서비스가 포함되어 있으며 최종 사용자에게 Kubernetes 클러스터를 제공합니다. 모든 Kubernetes 구성 요소는 Microsoft에서 유지 관리하고 운영합니다.

Microsoft는 컨트롤 창을 통해 다음 구성 요소를 관리하고 모니터링합니다.

* 쿠벨렛 또는 쿠베네테스 API 서버
* Etcd 또는 호환 되는 키-값 저장소, 서비스 품질 (QoS), 확장성 및 런타임제공
* DNS 서비스(예: 쿠베-dns 또는 CoreDNS)
* Kubernetes 프록시 또는 네트워킹

AKS는 완전히 관리되는 클러스터 솔루션이 아닙니다. 작업자 노드와 같은 일부 구성 요소는 사용자가 AKS 클러스터를 유지하는 데 도움을 주어야 하는 *공동 책임이*있습니다. 예를 들어 작업자 노드 운영 체제(OS) 보안 패치를 적용하려면 사용자 입력이 필요합니다.

서비스는 Microsoft와 AKS 팀이 서비스 가용성 및 기능을 배포, 운영 및 담당한다는 의미로 *관리됩니다.* 고객은 이러한 관리되는 구성 요소를 변경할 수 없습니다. Microsoft는 일관되고 확장 가능한 사용자 환경을 보장하기 위해 사용자 지정을 제한합니다. 완벽하게 사용자 정의 할 수있는 솔루션은 [AKS 엔진을](https://github.com/Azure/aks-engine)참조하십시오.

> [!NOTE]
> AKS 작업자 노드는 Azure 포털에 일반 Azure IaaS 리소스로 표시됩니다. 그러나 이러한 가상 컴퓨터는 사용자 지정 Azure 리소스\\그룹에 배포됩니다(MC*가 붙은 경우). AKS 작업자 노드를 변경할 수 있습니다. 예를 들어 보안 셸(SSH)을 사용하여 일반 가상 컴퓨터를 변경하는 방식으로 AKS 작업자 노드를 변경할 수 있습니다(그러나 기본 OS 이미지를 변경할 수 없으며 업데이트 또는 재부팅을 통해 변경 내용이 지속되지 않을 수 있음) 다른 Azure 리소스를 AKS에 연결할 수 있습니다. 작업자 노드. 그러나 *대역 관리 및 사용자 지정을 통해* 변경하면 AKS 클러스터를 지원할 수 없게 될 수 있습니다. Microsoft 지원에서 변경하도록 지시하지 않는 한 작업자 노드를 변경하지 마십시오.

모든 에이전트 노드의 대역 해제와 같이 위에서 정의한 대로 지원되지 않는 작업을 실행하면 클러스터가 지원되지 않습니다. AKS는 30일 이상 기간 동안 지원 지침에서 구성된 제어 비행기를 보관할 수 있는 권리를 보유합니다. AKS는 클러스터 etcd 메타데이터의 백업을 유지 관리하며 클러스터를 쉽게 재할당할 수 있습니다. 이 재할당은 클러스터를 활성 에이전트 노드로 업그레이드하거나 확장하는 것과 같이 클러스터를 다시 지원으로 가져오는 PUT 작업에 의해 시작될 수 있습니다.

## <a name="shared-responsibility"></a>공동 책임

클러스터가 생성되면 고객은 AKS가 만드는 Kubernetes 작업자 노드를 정의합니다. 고객 워크로드는 이러한 노드에서 실행됩니다. 고객은 작업자 노드를 소유하고 수정할 수 있습니다.

고객 클러스터 노드는 개인 코드를 실행하고 중요한 데이터를 저장하기 때문에 Microsoft 지원은 제한된 방법으로만 액세스할 수 있습니다. Microsoft 지원은 명시적인 고객 권한 이나 지원 없이 이러한 노드에 대 한 로그인, 명령을 실행 하거나 로그를 볼 수 없습니다.

작업자 노드는 중요하기 때문에 Microsoft는 백그라운드 관리를 제한하기 위해 세심한 주의를 기울입니다. 대부분의 경우 Kubernetes 마스터 노드 등 및 기타 Microsoft 관리 구성 요소가 실패하더라도 워크로드가 계속 실행됩니다. 부주의하게 수정된 작업자 노드는 데이터와 워크로드의 손실을 초래할 수 있으며 클러스터를 지원할 수 없게 만들 수 있습니다.

## <a name="aks-support-coverage"></a>AKS 지원 범위

Microsoft는 다음에 대한 기술 지원을 제공합니다.

* Kubernetes 서비스가 제공하는 모든 Kubernetes 구성 요소에 대한 연결(예: API 서버).
* Kubernetes 제어 평면 서비스(예: Kubernetes 마스터 노드, API 서버 등, 쿠베-dns)의 관리, 가동 시간, QoS 및 운영.
* Etcd. 지원에는 재해 계획 및 클러스터 상태 복원을 위해 30분마다 모든 etcd 데이터의 자동화되고 투명한 백업이 포함됩니다. 이러한 백업은 고객 이나 사용자가 직접 사용할 수 없습니다. 데이터 안정성과 일관성을 보장합니다.
* Kubernetes에 대한 Azure 클라우드 공급자 드라이버의 모든 통합 지점입니다. 여기에는 로드 밸러버, 영구 볼륨 또는 네트워킹(Kubernetes 및 Azure CNI)과 같은 다른 Azure 서비스에 대한 통합이 포함됩니다.
* Kubernetes API 서버 등과 같은 제어 평면 구성 요소의 사용자 지정에 대한 질문이나 문제 및 kube-dns.
* Azure CNI, kubenet 또는 기타 네트워크 액세스 및 기능 문제와 같은 네트워킹에 대한 문제입니다. 문제에는 DNS 확인, 패킷 손실, 라우팅 등이 포함될 수 있습니다. Microsoft는 다음과 같은 다양한 네트워킹 시나리오를 지원합니다.
  * 클러스터 및 관련 구성 요소 내의 Kubenet(기본) 및 고급 네트워킹(Azure CNI)
  * 다른 Azure 서비스 및 응용 프로그램에 대한 연결
  * 컨트롤러 및 인시어 또는 부하 분산기 구성
  * 네트워크 성능 및 대기 시간

Microsoft는 다음에 대한 기술 지원을 제공하지 않습니다.

* Kubernetes를 사용하는 방법에 대한 질문. 예를 들어 Microsoft 지원은 사용자 지정 받는 컨트롤러를 만들거나, 응용 프로그램 워크로드를 사용하거나, 타사 또는 오픈 소스 소프트웨어 패키지 또는 도구를 적용하는 방법에 대한 조언을 제공하지 않습니다.
  > [!NOTE]
  > Microsoft 지원은 AKS 클러스터 기능, 사용자 지정 및 튜닝(예: Kubernetes 작업 문제 및 절차)에 대해 조언할 수 있습니다.
* Kubernetes 제어 기의 일부로 제공되지 않거나 AKS 클러스터와 함께 배포되지 않는 타사 오픈 소스 프로젝트입니다. 이러한 프로젝트에는 Istio, Helm, Envoy 등이 포함될 수 있습니다.
  > [!NOTE]
  > Microsoft는 Helm 및 Kured와 같은 타사 오픈 소스 프로젝트에 대한 최상의 지원을 제공할 수 있습니다. 타사 오픈 소스 도구가 Kubernetes Azure 클라우드 공급자 또는 기타 AKS 관련 버그와 통합되는 경우 Microsoft는 Microsoft 설명서의 예제 및 응용 프로그램을 지원합니다.
* 타사 폐쇄 소스 소프트웨어. 이 소프트웨어는 보안 검색 도구 및 네트워킹 장치 또는 소프트웨어를 포함할 수 있습니다.
* 멀티클라우드 또는 멀티벤더 빌드 아웃에 대한 문제 예를 들어 Microsoft는 페더레이션된 다중 퍼블릭 클라우드 벤더 솔루션 실행과 관련된 문제를 지원하지 않습니다.
* [AKS 설명서에](https://docs.microsoft.com/azure/aks/)나열된 것 이외의 네트워크 사용자 지정.
  > [!NOTE]
  > Microsoft는 NSG(네트워크 보안 그룹)와 관련된 문제 및 버그를 지원합니다. 예를 들어 Microsoft 지원은 NSG 업데이트 실패 또는 예기치 않은 NSG 또는 로드 밸런서 동작에 대한 질문에 답할 수 있습니다.

## <a name="aks-support-coverage-for-worker-nodes"></a>작업자 노드에 대한 AKS 지원 범위

### <a name="microsoft-responsibilities-for-aks-worker-nodes"></a>AKS 작업자 노드에 대한 Microsoft 책임

Microsoft와 고객은 다음과 같은 경우 Kubernetes 작업자 노드에 대한 책임을 공유합니다.

* 기본 OS 이미지에는 모니터링 및 네트워킹 에이전트와 같은 추가가 필요합니다.
* 작업자 노드는 OS 패치를 자동으로 수신합니다.
* 작업자 노드에서 실행되는 Kubernetes 제어 평면 구성 요소의 문제는 자동으로 수정됩니다. 구성 요소는 다음과 같습니다.
  * 구베 프록시
  * Kubernetes 마스터 구성 요소에 대한 통신 경로를 제공하는 네트워킹 터널
  * kubelet
  * 도커 또는 모비 데몬

> [!NOTE]
> 작업자 노드에서 제어 평면 구성 요소가 작동하지 않는 경우 AKS 팀은 개별 구성 요소 또는 전체 작업자 노드를 재부팅해야 할 수 있습니다. 이러한 재부팅 작업은 자동화되어 있으며 일반적인 문제에 대한 자동 수정을 제공합니다. 이러한 재부팅은 긴급 유지 관리 또는 중단이 없는 한 _클러스터가_ 아닌 노드 수준에서만 발생합니다.

### <a name="customer-responsibilities-for-aks-worker-nodes"></a>AKS 작업자 노드에 대한 고객 책임

Microsoft는 OS 수준 패치를 적용하기 위해 작업자 노드를 자동으로 재부팅하지 않습니다. OS 패치는 작업자 노드에 전달되지만 *고객은* 변경 내용을 적용하기 위해 작업자 노드를 다시 부팅해야 합니다. 공유 라이브러리, SSHD(솔리드 스테이트 하이브리드 드라이브)와 같은 데몬 및 시스템 또는 OS 수준의 기타 구성 요소가 자동으로 패치됩니다.

고객은 Kubernetes 업그레이드를 실행해야 합니다. Azure 제어판 또는 Azure CLI를 통해 업그레이드를 실행할 수 있습니다. 이는 Kubernetes에 대한 보안 또는 기능 개선 사항이 포함된 업데이트에 적용됩니다.

> [!NOTE]
> AKS는 *관리되는 서비스이므로*최종 목표는 패치, 업데이트 및 로그 수집에 대한 책임을 제거하여 서비스 관리를 보다 완전하고 실습할 수 있도록 하는 것입니다. 종단 간 관리 용 서비스의 용량이 증가함에 따라 향후 릴리스에서는 일부 기능(예: 노드 재부팅 및 자동 패치)을 생략할 수 있습니다.

### <a name="security-issues-and-patching"></a>보안 문제 및 패치

AKS의 하나 이상의 구성 요소에서 보안 결함이 발견되면 AKS 팀은 영향을 받는 모든 클러스터를 패치하여 문제를 완화합니다. 또는 팀에서 사용자에게 업그레이드 지침을 제공합니다.

보안 결함이 영향을 주는 작업자 노드의 경우 가동 중지 시간 제로 패치를 사용할 수 있는 경우 AKS 팀은 해당 패치를 적용하고 변경 사항을 사용자에게 알립니다.

보안 패치에서 작업자 노드를 재부팅해야 하는 경우 Microsoft는 고객에게 이 요구 사항을 알립니다. 고객은 클러스터 패치를 받기 위해 재부팅하거나 업데이트할 책임이 있습니다. 사용자가 AKS 지침에 따라 패치를 적용하지 않으면 클러스터는 보안 문제에 계속 취약해집니다.

### <a name="node-maintenance-and-access"></a>노드 유지 관리 및 액세스

작업자 노드는 공동 책임이며 고객이 소유합니다. 따라서 고객은 작업자 노드에 로그인하고 커널 업데이트 및 패키지 설치 또는 제거와 같이 잠재적으로 유해한 변경을 할 수 있습니다.

고객이 파괴적인 변경을 하거나 제어 평면 구성 요소가 오프라인 상태가 되거나 작동하지 않는 경우 AKS는 이 오류를 감지하고 작업자 노드를 이전 작업 상태로 자동으로 복원합니다.

고객이 작업자 노드에 로그인하고 변경할 수 있지만 변경하면 클러스터를 지원할 수 없으므로 이 작업을 수행하는 것이 좋습니다.

## <a name="network-ports-access-and-nsgs"></a>네트워크 포트, 액세스 및 NSG

관리되는 서비스로서 AKS에는 특정 네트워킹 및 연결 요구 사항이 있습니다. 이러한 요구 사항은 일반 IaaS 구성 요소에 대한 요구 사항보다 덜 유연합니다. AKS에서는 NSG 규칙 사용자 지정, 특정 포트 차단(예: 아웃바운드 포트 443을 차단하는 방화벽 규칙 사용) 및 URL을 허용 목록에 추가하여 클러스터를 지원할 수 없게 만들 수 있습니다.

> [!NOTE]
> 현재 AKS에서는 클러스터에서 송신 트래픽을 완전히 잠글 수 없습니다. 클러스터가 아웃바운드 트래픽에 사용할 수 있는 URL 및 포트 목록을 제어하려면 트래픽 [제한을 참조합니다.](limit-egress-traffic.md)

## <a name="unsupported-alpha-and-beta-kubernetes-features"></a>지원되지 않는 알파 및 베타 Kubernetes 기능

AKS는 업스트림 Kubernetes 프로젝트 내에서 안정적인 기능만 지원합니다. 달리 문서화되지 않는 한 AKS는 업스트림 Kubernetes 프로젝트에서 사용할 수 있는 알파 및 베타 기능을 지원하지 않습니다.

두 가지 시나리오에서 알파 또는 베타 기능은 일반적으로 사용할 수 있기 전에 롤아웃될 수 있습니다.

* 고객은 AKS 제품, 지원 또는 엔지니어링 팀과 만나 이러한 새로운 기능을 사용해 달라는 요청을 받았습니다.
* 이러한 기능은 [피처 플래그에 의해 활성화되었습니다.](https://github.com/Azure/AKS/blob/master/previews.md) 고객은 이러한 기능을 사용하도록 명시적으로 선택해야 합니다.

## <a name="preview-features-or-feature-flags"></a>피처 또는 피처 플래그 미리 보기

확장된 테스트 및 사용자 피드백이 필요한 기능 및 기능의 경우 Microsoft는 기능 플래그 뒤에 새로운 미리 보기 기능 또는 기능을 릴리스합니다. 이러한 기능을 시험판 또는 베타 기능으로 고려하십시오.

미리 보기 기능 또는 기능 플래그 기능은 프로덕션용이 아닙니다. API 및 동작, 버그 수정 및 기타 변경 사항의 지속적인 변경으로 인해 클러스터가 불안정하고 가동 중지 시간이 발생할 수 있습니다.

공개 미리 보기의 기능은 미리 보기중이며 프로덕션용이 아니며 업무 시간 동안에만 AKS 기술 지원 팀에서 지원하기 때문에 '최선의 노력' 지원을 받습니다. 자세한 내용은 다음을 참조하십시오.

* [Azure 지원 FAQ](https://azure.microsoft.com/support/faq/)

> [!NOTE]
> 미리 보기 기능은 Azure *구독* 수준에서 적용됩니다. 프로덕션 구독에 미리 보기 기능을 설치하지 마십시오. 프로덕션 구독에서 미리 보기 기능은 기본 API 동작을 변경하고 일반 작업에 영향을 줄 수 있습니다.

## <a name="upstream-bugs-and-issues"></a>업스트림 버그 및 문제

업스트림 Kubernetes 프로젝트의 개발 속도를 감안할 때 버그는 변함없이 발생합니다. 이러한 버그 중 일부는 AKS 시스템 내에서 패치하거나 해결할 수 없습니다. 대신 버그 수정에는 업스트림 프로젝트(예: Kubernetes, 노드 또는 작업자 운영 체제 및 커널)에 더 큰 패치가 필요합니다. Microsoft가 소유하고 있는 구성 요소(예: Azure 클라우드 공급자)의 경우 AKS 및 Azure 담당자는 커뮤니티의 업스트림에서 문제를 해결하기 위해 최선을 다하고 있습니다.

하나 이상의 업스트림 버그로 인해 기술 지원 문제가 근본 원인인 경우 AKS 지원 및 엔지니어링 팀은 다음을 수행합니다.

* 업스트림 버그를 식별하고 지원 세부 정보와 연결하여 이 문제가 클러스터 또는 워크로드에 영향을 미치는 이유를 설명합니다. 고객은 필요한 리포지토리에 대한 링크를 수신하여 문제를 보고 새 릴리스에서 수정 사항을 제공하는 시기를 확인할 수 있습니다.
* 잠재적인 해결 또는 완화 방법을 제공합니다. 문제를 완화할 수 있는 경우 [알려진 문제가](https://github.com/Azure/AKS/issues?q=is%3Aissue+is%3Aopen+label%3Aknown-issue) AKS 리포지토리에 제출됩니다. 알려진 문제 제출은 다음과 같습니다.
  * 업스트림 버그에 대한 링크를 포함한 문제입니다.
  * 업그레이드 또는 솔루션의 다른 지속성에 대한 해결 및 세부 정보입니다.
  * 업스트림 릴리스 케이던스를 기반으로 문제의 포함에 대한 대략적인 타임라인입니다.
