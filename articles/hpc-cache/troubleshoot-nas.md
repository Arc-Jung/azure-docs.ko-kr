# <a name="troubleshoot-nas-configuration-and-nfs-storage-target-issues"></a>NAS 구성 및 NFS 스토리지 대상 문제 해결

이 문서에서는 Azure HPC 캐시가 NFS 저장소 시스템을 저장소 대상으로 추가하지 못하도록 할 수 있는 몇 가지 일반적인 구성 오류 및 기타 문제에 대한 솔루션을 제공합니다.

이 문서에서는 포트를 확인하는 방법과 NAS 시스템에 대한 루트 액세스를 활성화하는 방법에 대한 자세한 내용을 설명합니다. 또한 NFS 저장소 대상 만들기가 실패할 수 있는 덜 일반적인 문제에 대한 자세한 정보도 포함되어 있습니다.

> [!TIP]
> 이 가이드를 사용하기 전에 [NFS 저장소 대상에 대한 필수 구성 조건을](hpc-cache-prereqs.md#nfs-storage-requirements)읽어보십시오.

문제에 대한 해결 방안이 여기에 포함되어 있지 않은 경우 Microsoft 서비스 및 지원에서 문제를 조사하고 해결할 수 있도록 [지원 티켓을 열어](hpc-cache-support-ticket.md) 주십시오.

## <a name="check-port-settings"></a>포트 설정 확인

Azure HPC 캐시는 백 엔드 NAS 저장소 시스템의 여러 UDP/TCP 포트에 대한 읽기/쓰기 액세스가 필요합니다. NAS 시스템에서 이러한 포트에 액세스할 수 있는지, 저장소 시스템과 캐시 서브넷 사이의 방화벽을 통해 이러한 포트에 대한 트래픽이 허용되는지 확인합니다. 이 구성을 확인하려면 데이터 센터의 방화벽 및 네트워크 관리자와 협력해야 할 수 있습니다.

포트는 다른 공급업체의 저장소 시스템에 대해 다르므로 저장소 대상을 설정할 때 시스템의 요구 사항을 확인합니다.

일반적으로 캐시는 다음 포트에 액세스해야 합니다.

| 프로토콜 | 포트  | 서비스  |
|----------|-------|----------|
| TCP/UDP  | 111   | rpcbind  |
| TCP/UDP  | 2049  | NFS      |
| TCP/UDP  | 4045  | nlockmgr |
| TCP/UDP  | 4046  | 장착 된   |
| TCP/UDP  | 4047  | 상태   |

시스템에 필요한 특정 포트를 보려면 다음 ``rpcinfo`` 명령을 사용합니다. 아래 명령은 포트를 나열하고 관련 결과를 테이블에 서식을 지정합니다. 시스템의 IP 주소를 *<storage_IP>* 용어 대신 사용합니다.)

NFS 인프라가 설치된 Linux 클라이언트에서 이 명령을 내릴 수 있습니다. 클러스터 서브넷 내에서 클라이언트를 사용하는 경우 서브넷과 저장소 시스템 간의 연결을 확인하는 데도 도움이 될 수 있습니다.

```bash
rpcinfo -p <storage_IP> |egrep "100000\s+4\s+tcp|100005\s+3\s+tcp|100003\s+3\s+tcp|100024\s+1\s+tcp|100021\s+4\s+tcp"| awk '{print $4 "/" $3 " " $5}'|column -t
```

``rpcinfo`` 쿼리에서 반환되는 모든 포트가 Azure HPC 캐시의 서브넷에서 무제한 트래픽을 허용하는지 확인합니다.

NAS 자체뿐만 아니라 스토리지 시스템과 캐시 서브넷 사이의 방화벽모두에서 이러한 설정을 확인합니다.

## <a name="check-root-access"></a>루트 액세스 확인

Azure HPC 캐시는 저장소 대상을 만들려면 저장소 시스템의 내보내기에 액세스해야 합니다. 특히 내보내기를 사용자 ID 0으로 탑재합니다.

다른 스토리지 시스템은 다음과 같은 다양한 방법을 사용하여 이 액세스를 활성화합니다.

* Linux 서버는 ``no_root_squash`` 일반적으로 내보낸 경로에 추가합니다. ``/etc/exports``
* NetApp 및 EMC 시스템은 일반적으로 특정 IP 주소 또는 네트워크에 연결된 내보내기 규칙으로 액세스를 제어합니다.

내보내기 규칙을 사용하는 경우 캐시는 캐시 서브넷에서 여러 개의 서로 다른 IP 주소를 사용할 수 있습니다. 가능한 서브넷 IP 주소의 전체 범위에서 액세스를 허용합니다.

NAS 스토리지 공급업체와 협력하여 캐시에 대한 적절한 수준의 액세스를 활성화합니다.

### <a name="allow-root-access-on-directory-paths"></a>디렉터리 경로에 대한 루트 액세스 허용
<!-- linked in prereqs article -->

계층적 디렉터리를 내보내는 NAS 시스템의 경우 Azure HPC 캐시는 각 내보내기 수준에 대한 루트 액세스가 필요합니다.

예를 들어 시스템에 다음과 같은 세 가지 내보내기가 표시될 수 있습니다.

* ``/ifs``
* ``/ifs/accounting``
* ``/ifs/accounting/payroll``

내보내기는 ``/ifs/accounting/payroll`` 의 ``/ifs/accounting``자식이며 ``/ifs/accounting`` 그 자체는 ``/ifs``의 자식입니다.

내보내기를 ``payroll`` HPC 캐시 저장소 대상으로 추가하면 캐시가 ``/ifs/`` 실제로 급여 디렉터리에 마운트되고 액세스됩니다. 따라서 Azure HPC 캐시는 ``/ifs`` 내보내기에 ``/ifs/accounting/payroll`` 액세스하기 위해 루트 액세스가 필요합니다.

이 요구 사항은 캐시가 파일을 인덱싱하고 저장소 시스템에서 제공하는 파일 핸들을 사용하여 파일 충돌을 방지하는 방법과 관련이 있습니다.

계층적 내보내기가 있는 NAS 시스템은 파일을 다른 내보내기에서 검색하는 경우 동일한 파일에 대해 다른 파일 핸들을 제공할 수 있습니다. 예를 들어 클라이언트가 ``/ifs/accounting`` 파일을 ``payroll/2011.txt``마운트하고 액세스할 수 있습니다. 다른 클라이언트가 ``/ifs/accounting/payroll`` 파일을 ``2011.txt``마운트하고 액세스합니다. 저장소 시스템에서 파일 핸들을 할당하는 방법에 따라 이 두 클라이언트는 서로 다른 파일 ``<mount2>/payroll/2011.txt`` 핸들(하나는 용 및 다른 ``<mount3>/2011.txt``클라이언트)으로 동일한 파일을 받을 수 있습니다.

백 엔드 저장소 시스템은 파일 핸들에 대한 내부 별칭을 유지하지만 Azure HPC 캐시는 인덱스에서 동일한 항목을 참조하는 파일 핸들을 알 수 없습니다. 따라서 캐시가 동일한 파일에 대해 다른 쓰기를 캐시할 수 있으며 동일한 파일임을 알지 못하기 때문에 변경 내용을 잘못 적용할 수 있습니다.

여러 내보내기에 있는 파일에 대해 이 가능한 파일 충돌을 방지하기 위해 Azure HPC``/ifs`` 캐시는 경로(예에서)에서 사용 가능한 가장 얕은 내보내기를 자동으로 탑재하고 해당 내보내기에서 제공된 파일 핸들을 사용합니다. 여러 내보내기에서 동일한 기본 경로를 사용하는 경우 Azure HPC 캐시는 해당 경로에 대한 루트 액세스가 필요합니다.

## <a name="enable-export-listing"></a>내보내기 목록 사용
<!-- link in prereqs article -->

NAS는 Azure HPC 캐시가 내보내기를 쿼리할 때 내보내기를 나열해야 합니다.

대부분의 NFS 스토리지 시스템에서 Linux 클라이언트에서 다음 쿼리를 전송하여 이 쿼리를 테스트할 수 있습니다.``showmount -e <storage IP address>``

가능하면 캐시와 동일한 가상 네트워크에서 Linux 클라이언트를 사용합니다.

해당 명령에 내보내기가 나열되지 않으면 캐시가 저장소 시스템에 연결하는 데 문제가 있습니다. NAS 공급업체와 협력하여 내보내기 목록을 활성화합니다.

## <a name="adjust-vpn-packet-size-restrictions"></a>VPN 패킷 크기 제한 조정
<!-- link in prereqs article -->

캐시와 NAS 장치 사이에 VPN이 있는 경우 VPN은 전체 크기의 1500바이트 이더넷 패킷을 차단할 수 있습니다. NAS와 Azure HPC 캐시 인스턴스 간의 대규모 교환이 완료되지 않지만 작은 업데이트가 예상대로 작동하는 경우 이 문제가 발생할 수 있습니다.

VPN 구성의 세부 정보를 알지 못하면 시스템에 이 문제가 있는지 여부를 알 수 있는 간단한 방법은 없습니다. 다음은 이 문제를 확인하는 데 도움이 되는 몇 가지 방법입니다.

* VPN의 양쪽에 있는 패킷 스니퍼를 사용하여 어떤 패킷이 성공적으로 전송되는지 감지합니다.
* VPN에서 ping 명령을 허용하는 경우 전체 크기의 패킷을 보내는 테스트를 수행할 수 있습니다.

  이러한 옵션을 사용하여 VPN을 통해 NAS로 ping 명령을 실행합니다. (<*storage_IP>* 값 대신 스토리지 시스템의 IP 주소를 사용합니다.

   ```bash
   ping -M do -s 1472 -c 1 <storage_IP>
   ```

  다음은 명령의 옵션입니다.

  * ``-M do``- 조각하지 마십시오
  * ``-c 1``- 하나의 패킷만 보내기
  * ``-s 1472``- 페이로드 크기를 1472바이트로 설정합니다. 이더넷 오버헤드를 고려한 후 1500바이트 패킷의 최대 크기 페이로드입니다.

  성공적인 응답은 다음과 같습니다.

  ```bash
  PING 10.54.54.11 (10.54.54.11) 1472(1500) bytes of data.
  1480 bytes from 10.54.54.11: icmp_seq=1 ttl=64 time=2.06 ms
  ```

  1472바이트로 ping에 실패하면 원격 시스템이 최대 프레임 크기를 올바르게 감지하도록 VPN에서 MSS 클램핑을 구성해야 할 수 있습니다. 자세한 내용은 [VPN 게이트웨이 IPsec/IKE 매개 변수 설명서를](../vpn-gateway/vpn-gateway-about-vpn-devices.md#ipsec) 참조하십시오.

## <a name="check-for-acl-security-style"></a>ACL 보안 스타일 확인

일부 NAS 시스템은 ACL(액세스 제어 목록)과 기존 POSIX 또는 UNIX 보안을 결합하는 하이브리드 보안 스타일을 사용합니다.

시스템에서 약어 "ACL"을 포함하지 않고 보안 스타일을 UNIX 또는 POSIX로 보고하는 경우 이 문제는 영향을 주지 않습니다.

ACL을 사용하는 시스템의 경우 Azure HPC 캐시는 파일 액세스를 제어하기 위해 추가 사용자 별 값을 추적해야 합니다. 이 작업은 액세스 캐시를 사용하도록 설정하여 수행됩니다. 액세스 캐시를 켜는 사용자 대면 컨트롤은 없지만 지원 티켓을 열어 캐시 시스템의 영향을 받는 저장소 대상에 대해 사용하도록 요청할 수 있습니다.
