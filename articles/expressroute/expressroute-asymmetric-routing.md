# 여러 네트워크 경로를 포함한 비대칭 라우팅

이 문서에서는 원본과 대상 간에 여러 경로를 사용할 수 있을 때 전달 트래픽 및 반환 트래픽에서 다른 경로를 사용하는 방법을 설명합니다.

비대칭 라우팅을 이해하려면 두 가지 개념을 이해해야 합니다. 하나는 여러 네트워크 경로의 영향입니다. 다른 하나는 방화벽과 같은 상태를 유지하는 장치의 동작입니다. 이러한 장치는 상태 저장 장치라고 합니다. 이 두 가지 요인의 조합으로 트래픽이 상태 저장 장치를 통해 시작되지 않으면 해당 장치가 해당 트래픽을 삭제하는 시나리오를 만듭니다.

## 여러 네트워크 경로

엔터프라이즈 네트워크가 인터넷 서비스 공급자를 통해 인터넷에 대한 하나의 링크를 가진 경우 동일한 경로를 통해 인터넷 내외에서 모든 트래픽이 들어옵니다. 종종 회사는 네트워크 가동 시간을 개선하기 위해 중복 경로로 여러 회로를 구입합니다. 이러한 경우 다른 링크를 통해 반환 트래픽이 들어오는 반면 네트워크 외부에서 인터넷으로 이동하는 트래픽은 하나의 링크를 통해 이동합니다. 이러한 현상은 일반적으로 역방향 트래픽이 원래 흐름에서 다른 경로를 사용하는 비대칭 라우팅이라고 합니다.

![라우팅3](./media/expressroute-asymmetric-routing/AsymmetricRouting3.png)

이전의 설명은 인터넷에 대한 것이지만 다른 조합의 여러 경로에도 적용됩니다. 동일한 대상에 대한 인터넷 경로 및 개인 경로, 동일한 대상에 대한 여러 개인 경로를 예로 들 수 있습니다.

원본에서 대상으로 가는 과정에서 각 라우터는 대상에 도달하는 최상의 경로 계산을 기반으로 대상에 도달하는 최상의 경로를 계산합니다. 두 가지 주요 요소를 기반으로 가능한 경로를 결정합니다.

1.	외부 네트워크 간의 라우팅은 일반적으로 BGP라는 경계 게이트웨이 프로토콜이라는 라우팅 프로토콜을 기반으로 합니다. BGP는 인접 라우터에서 보급하고 대상에 최상의 경로를 결정하는 여러 단계를 통해 실행하며 해당 라우팅 테이블에 설치합니다.
2.	경로와 연결된 서브넷 마스크의 길이입니다. 다른 서브넷 마스크가 아닌 동일한 IP 주소에 대한 여러 보급이 수신되는 경우 더 긴 서브넷 마스크를 포함한 보급이 구체적인 경로로 간주되기 때문에 선호됩니다.

## 상태 저장 장치

라우터는 라우팅을 위해 패킷의 IP 헤더를 살펴봅니다. 그러나 패킷 내부를 훨씬 자세히 살펴보는 장치가 있습니다. 일반적으로 이러한 장치는 Layer4(TCP/UDP) 또는 Layer7(응용 프로그램 계층) 헤더까지도 살펴봅니다. 이러한 장치는 보안 장치 또는 대역폭 최적화 장치입니다. 방화벽은 상태 저장 장치의 일반적인 예입니다. 방화벽은 프로토콜, TCP/UDP 포트, URL 헤더 등 다양한 필드에 따라 해당 인터페이스를 통해 패킷을 허용하거나 거부합니다. 이 패킷 검사는 장치에 다양한 처리 부하를 배치합니다. 방화벽은 성능을 개선하기 위해 흐름의 첫 번째 패킷을 검사합니다. 패킷을 허용하는 경우 해당 상태 테이블에 흐름 정보를 보관합니다. 이 흐름에 관련된 모든 후속 패킷은 초기 의사 결정에 따라 허용됩니다. 따라서 기존 흐름의 일부인 패킷이 방화벽에 도달하고 방화벽에 이전 상태 정보가 포함되지 않는 경우 방화벽은 이 패킷을 삭제합니다.

## Express 경로를 포함한 비대칭 라우팅

Microsoft에 연결할 때 Express 경로를 통해 네트워크에 다음과 같은 변경 내용이 발생합니다.

1.	Microsoft에 대한 여러 링크가 있습니다. 한 가지 링크는 기존 인터넷 연결이고 다른 하나는 Express 경로를 통한 연결입니다. Microsoft에 대한 일부 트래픽은 인터넷을 통과할 수 있지만 Express 경로를 통해 돌아옵니다. 혹은 그 반대입니다.
2.	Express 경로를 통해 구체적인 IP 주소를 수신합니다. 따라서 항상 Express 경로를 통해 제공되는 서비스의 경우 네트워크에서 Microsoft로 전달되는 트래픽은 Express 경로를 선호합니다.

위의 두 가지가 가져오는 영향을 이해하기 위해 몇 가지 시나리오를 살펴보겠습니다. 인터넷에 대한 회로가 오직 하나이며 인터넷을 통한 모든 Microsoft 서비스를 사용한다고 가정합니다. 네트워크에서 Microsoft로 전달되는 트래픽은 동일한 인터넷 링크를 탐색하고 방화벽을 통해 전달됩니다. 첫 번째 패킷을 확인하면 방화벽이 흐름을 기록하고 흐름이 상태 테이블에 존재하면 반환 패킷이 허용됩니다.

![라우팅1](./media/expressroute-asymmetric-routing/AsymmetricRouting1.png)


이제 Express 경로를 켜고 Express 경로를 통해 Microsoft에서 제공하는 서비스를 사용합니다. Microsoft의 다른 모든 서비스는 인터넷을 통해 사용됩니다. Express 경로에 연결하여 에지에 별도 방화벽을 배포합니다. Microsoft는 특정 서비스에 대한 Express 경로에 네트워크에 대한 구체적인 접두사를 보급합니다. 라우팅 인프라는 해당 접두사에 대한 기본 경로로 Express 경로를 선택합니다. Express 경로에 걸쳐 Microsoft에 대한 공용 IP 주소를 보급하지 않는 경우 Microsoft는 인터넷을 통해 공용 IP 주소와 통신합니다. 따라서 Microsoft의 역방향 트래픽이 인터넷을 사용하는 반면 네트워크에서 Microsoft로 전달되는 트래픽은 Express 경로를 사용합니다. 에지의 방화벽이 상태 테이블에서 찾을 수 없는 흐름에 대한 응답 패킷을 확인하는 경우 반환 트래픽을 삭제합니다.

Express 경로 및 인터넷에 동일한 NAT 풀을 사용하는 경우 네트워크의 개인 IP 주소에 클라이언트와 비슷한 문제가 표시됩니다. 이러한 서비스에 대한 IP 주소가 Express 경로를 통해 보급되지 않으면 Windows 업데이트와 같은 서비스에 대한 요청은 인터넷을 통해 이동합니다. 그러나 반환 트래픽은 Express 경로를 통해 다시 돌아옵니다. Microsoft가 인터넷 및 Express 경로에서 동일한 서브넷 마스크를 가진 IP 주소를 수신하는 경우 인터넷에 대한 Express 경로를 선호합니다. 네트워크 에지에서 Express 경로에 연결된 방화벽이나 기타 상태 저장 장치에 흐름에 대한 이전 정보가 없는 경우 해당 흐름에 속하는 패킷을 삭제합니다.

## 비대칭 라우팅에 대한 솔루션

비대칭 라우팅 문제를 해결하는 두 가지 방법이 있습니다. 하나는 라우팅을 통한 방법이고 다른 하나는 SNAT(원본 기반 NAT)를 통한 방법입니다.

1. 라우팅

    - 공용 IP 주소를 적절한 WAN 링크에 보급하는지를 확인해야 합니다. 예를 들어, 인증 트래픽에 인터넷을 사용하고 메일 트래픽에 Express 경로를 사용하려는 경우입니다. 그러면 Express 경로에 대해 ADFS 공용 IP 주소를 보급하지 않아야 합니다. 마찬가지로, 온-프레미스 ADFS 서버는 Express 경로를 통해 수신된 IP 주소에 노출되지 않아야 합니다. Express 경로를 통해 수신된 경로는 구체적이므로 Express 경로를 Microsoft에 대한 인증 트래픽의 기본 경로로 만들어서 비대칭 라우팅을 발생시킵니다.

    - 인증을 위해 Express 경로를 사용하려는 경우 NAT 없이 Express 경로에 ADFS 공용 IP 주소를 보급해야 합니다. 이렇게 고객으로부터 Microsoft로 전달되는 반환 트래픽이 인터넷에 선호되기 때문에 Express 경로를 사용하는 반면 Microsoft에서 온-프레미스 ADFS 서버에 발생하는 트래픽은 Express 경로로 이동합니다.

2. 원본 기반 NAT

	비대칭 라우팅 문제를 해결하는 다른 방법은 SNAT(원본 NAT)를 통하는 것입니다. 예를 들어, 이 통신에 인터넷을 사용하려고 Express 경로에 걸쳐 온-프레미스 SMTP 서버의 공용 IP 주소를 보급하지 않은 경우입니다. Microsoft에서 온-프레미스 SMTP 서버로 만든 요청은 인터넷을 탐색합니다. 원본 NAT는 내부 IP 주소에 대한 들어오는 요청입니다. SMTP 서버의 역방향 트래픽은 Express 경로 대신 에지 방화벽(NAT를 수행하는 데 사용됨)으로 이동합니다. 이러한 방식으로 반환 트래픽은 인터넷을 통해 다시 돌아갑니다.


![라우팅2](./media/expressroute-asymmetric-routing/AsymmetricRouting2.png)

## 비대칭 라우팅 감지

경로 추적은 트래픽이 예상된 경로를 탐색하도록 하는 가장 좋은 방법입니다. 온-프레미스 SMTP 서버에서 Microsoft로 전달되는 트래픽이 인터넷 경로를 사용해야 하는 경우 SMTP 서버에서 Office 365로 경로를 추적합니다. 결과를 통해 트래픽이 네트워크를 Express 경로가 아닌 인터넷에 남겨두는지 유효성을 검사합니다.
