# <a name="how-to-troubleshoot-issues-with-the-log-analytics-agent-for-windows"></a>Windows용 Log Analytics 에이전트의 문제를 해결하는 방법 

이 문서에서는 Azure Monitor의 Windows용 로그 분석 에이전트에서 발생할 수 있는 오류를 해결하는 데 도움이 되며 이를 해결할 수 있는 해결 방법을 제안합니다.

이 문서의 단계를 수행해도 문제가 해결되지 않으면 다음 지원 채널을 사용할 수 있습니다.

* 프리미어 지원 혜택을 받는 고객은 [프리미어](https://premier.microsoft.com/)를 사용하여 지원 요청을 열 수 있습니다.
* Azure 지원 계약을 맺은 고객은 [Azure Portal](https://manage.windowsazure.com/?getsupport=true)에서 지원 요청을 열 수 있습니다.
* 로그 분석 피드백 페이지를 방문하여 제출된 [https://aka.ms/opinsightsfeedback](https://aka.ms/opinsightsfeedback) 아이디어와 버그를 검토하거나 새 아이디어를 제출합니다. 

## <a name="important-troubleshooting-sources"></a>중요한 문제 해결 소스

 Windows용 Log Analytics 에이전트와 관련된 문제 해결을 지원하기 위해 에이전트는 특히 *응용 프로그램 및 서비스\운영 관리자에서*Windows 이벤트 로그에 이벤트를 기록합니다.  

## <a name="connectivity-issues"></a>연결 문제

에이전트가 프록시 서버 또는 방화벽을 통해 통신하는 경우 원본 컴퓨터 및 Azure Monitor 서비스와의 통신을 방해하는 제한이 있을 수 있습니다. 잘못된 구성으로 인해 통신이 차단된 경우 에이전트를 설치하거나 에이전트 사후 설치를 구성하여 추가 작업 영역에 보고하도록 하는 동안 작업 영역에 대한 등록이 실패할 수 있습니다. 등록에 성공한 후 에이전트 통신이 실패할 수 있습니다. 이 섹션에서는 Windows 에이전트에서 이러한 유형의 문제를 해결하는 방법에 대해 설명합니다.

다음 표에 설명된 다음 포트 와 URL을 허용하도록 방화벽 또는 프록시가 구성되어 있는지 다시 한 번 확인합니다. 또한 에이전트와 Azure Monitor 간의 안전한 TLS 채널을 방지할 수 있으므로 웹 트래픽에 대해 HTTP 검사가 활성화되지 않은지 확인합니다.  

|에이전트 리소스|포트 |Direction |HTTPS 검사 무시|
|------|---------|--------|--------|   
|*.ods.opinsights.azure.com |포트 443 |아웃바운드|yes |  
|*.oms.opinsights.azure.com |포트 443 |아웃바운드|yes |  
|\*.blob.core.windows.net |포트 443 |아웃바운드|yes |  

Azure 정부에 필요한 방화벽 정보는 [Azure 정부 관리를](../../azure-government/documentation-government-services-monitoringandmanagement.md#azure-monitor-logs)참조하십시오. Azure 자동화 하이브리드 Runbook Worker를 사용하여 사용자 환경에서 Runbook 또는 관리 솔루션을 사용하기 위해 자동화 서비스에 연결하고 등록하려는 경우 [하이브리드 Runbook Worker에 대한 네트워크 구성에](../../automation/automation-hybrid-runbook-worker.md#network-planning)설명된 포트 번호 및 URL에 액세스할 수 있어야 합니다. 

에이전트가 Azure Monitor와 성공적으로 통신하고 있는지 확인할 수 있는 방법에는 여러 가지가 있습니다.

- 작업 영역에서 [Azure 로그 분석 에이전트 상태 평가를](../insights/solution-agenthealth.md) 사용하도록 설정합니다. 에이전트 상태 대시보드에서 **응답하지 않는 에이전트 수** 열을 확인하여 에이전트가 나열되어 있는지 빠르게 확인합니다.  

- 다음 쿼리를 실행하여 에이전트가 보고하도록 구성된 작업 영역에 하트비트를 보내고 있는지 확인합니다. 기기의 실제 이름으로 바꿉니다. `<ComputerName>`

    ```
    Heartbeat 
    | where Computer like "<ComputerName>"
    | summarize arg_max(TimeGenerated, * ) by Computer 
    ```

    컴퓨터가 서비스와 성공적으로 통신하는 경우 쿼리는 결과를 반환해야 합니다. 쿼리가 결과를 반환하지 않은 경우 먼저 에이전트가 올바른 작업 영역에 보고하도록 구성되었는지 확인합니다. 올바르게 구성된 경우 3단계로 진행하고 Windows 이벤트 로그를 검색하여 에이전트가 Azure Monitor와 통신하지 못하는 문제를 로깅하고 있는지 확인합니다.

- 연결 문제를 식별하는 또 다른 방법은 **TestCloudConnectivity** 도구를 실행하는 것입니다. 이 도구는 기본적으로 *폴더 %SystemRoot%\프로그램 파일\Microsoft 모니터링 에이전트의*에이전트와 함께 설치됩니다. 상승된 명령 프롬프트에서 폴더로 이동하여 도구를 실행합니다. 이 도구는 결과를 반환하고 테스트가 실패한 위치(예: 차단된 특정 포트/URL과 관련된 경우)를 강조 표시됩니다. 

    ![테스트클라우드커넥션 툴 실행 결과](./media/agent-windows-troubleshoot/output-testcloudconnection-tool-01.png)

- **이벤트 소스별** -  *운영 관리자* 이벤트 로그를*필터링하여 상태 서비스 모듈,* *상태 서비스*및 서비스 *커넥터를* 필터링하고 **이벤트 수준** *경고* 및 *오류별로* 필터링하여 다음 표에서 이벤트를 작성했는지 확인합니다. 이 경우 가능한 각 이벤트에 포함된 해결 단계를 검토합니다.

    |이벤트 ID |원본 |설명 |해결 방법 |
    |---------|-------|------------|-----------|
    |2133 & 2129 |상태 관리 서비스 |에이전트에서 서비스에 대한 연결이 실패했습니다. |이 오류는 에이전트가 직접 또는 방화벽/프록시 서버를 통해 Azure Monitor 서비스에 통신할 수 없는 경우에 발생할 수 있습니다. 에이전트 프록시 설정을 확인하거나 네트워크 방화벽/프록시에서 컴퓨터에서 서비스로의 TCP 트래픽을 허용하는지 확인합니다.|
    |2138 |상태 관리 서비스 모듈 |프록시에 인증 필요 |에이전트 프록시 설정을 구성하고 프록시 서버로 인증하는 데 필요한 사용자 이름/암호를 지정합니다. |
    |2129 |상태 관리 서비스 모듈 |연결 실패/TLS 협상 실패 |네트워크 어댑터 TCP/IP 설정 및 에이전트 프록시 설정을 확인합니다.|
    |2127 |상태 관리 서비스 모듈 |데이터 전송 실패수신 오류 코드 |하루 동안에만 주기적으로 발생하는 경우 무시할 수 있는 임의의 변칙일 수 있습니다. 모니터링하여 발생하는 빈도를 파악합니다. 하루 종일 자주 발생하는 경우 먼저 네트워크 구성 및 프록시 설정을 확인합니다. 설명에 HTTP 오류 코드(404)가 포함되어 있고 에이전트가 서비스에 데이터를 보내려고 시도하는 것은 처음인 경우 내부 404 오류 코드가 있는 500 오류가 포함됩니다. 404는 새 작업 영역의 저장소 영역이 아직 프로비전중임을 나타내는 찾을 수 없다는 것을 의미합니다. 다음 다시 시도시 데이터는 예상대로 작업 영역에 성공적으로 기록됩니다. HTTP 오류(403)는 권한 또는 자격 증명 문제를 나타낼 수 있습니다. 403 오류에 포함된 자세한 내용은 문제 해결에 도움이 됩니다.|
    |4000 |서비스 커넥터 |DNS 이름 확인에 실패했습니다. |서비스로 데이터를 보낼 때 사용한 인터넷 주소를 컴퓨터에서 확인할 수 없습니다. 이는 컴퓨터의 DNS 확인자 설정, 잘못된 프록시 설정 또는 공급자와의 임시 DNS 문제일 수 있습니다. 정기적으로 발생하는 경우 일시적인 네트워크 관련 문제로 인해 발생할 수 있습니다.|
    |4001 |서비스 커넥터 |서비스에 대한 연결이 실패했습니다. |이 오류는 에이전트가 직접 또는 방화벽/프록시 서버를 통해 Azure Monitor 서비스에 통신할 수 없는 경우에 발생할 수 있습니다. 에이전트 프록시 설정을 확인하거나 네트워크 방화벽/프록시에서 컴퓨터에서 서비스로의 TCP 트래픽을 허용하는지 확인합니다.|
    |4002 |서비스 커넥터 |서비스는 쿼리 응답으로 HTTP 상태 코드 403을 반환합니다. 서비스 관리자에게 서비스의 상태를 확인합니다. 쿼리는 나중에 다시 시도됩니다. |이 오류는 에이전트의 초기 등록 단계에서 작성되며 *https://\<workspaceID>.oms.opinsights.com/AgentService.svc/AgentTopologyRequest*와 유사한 URL이 표시됩니다. 오류 코드(403)는 금지됨을 의미하며 잘못 입력된 작업 영역 ID 또는 키로 인해 발생할 수 있거나 컴퓨터에서 데이터와 시간이 올바르지 않을 수 있습니다. 시간이 현재 시간에서 +/-15분인 경우 등록이 실패합니다. 이 작업을 해결하려면 Windows 컴퓨터의 날짜 및/또는 시간대를 업데이트합니다.|

## <a name="data-collection-issues"></a>데이터 수집 문제

에이전트를 설치하고 구성된 작업 영역 또는 작업 영역에 보고한 후 활성화된 기능및 대상 컴퓨터에 따라 구성 수신, 성능, 로그 또는 기타 데이터를 서비스에 수집 또는 전달하지 않을 수 있습니다. 다음과 같은 지 확인할 필요가 있습니다.

- 특정 데이터 형식입니까 아니면 작업 영역에서 사용할 수 없는 모든 데이터입니까?
- 솔루션에 의해 지정된 데이터 형식이거나 작업 영역 데이터 수집 구성의 일부로 지정됩니까?
- 영향을 받는 컴퓨터수는 몇 대입니까? 작업 영역에 보고하는 단일 또는 여러 컴퓨터입니까?
- 그것은 작동 하 고 그것은 하루 중 특정 시간에 중지 했 습니까, 또는 수집 되지 않은? 
- 사용 중인 로그 검색 쿼리가 구문적으로 정확합니까? 
- 에이전트가 Azure 모니터에서 구성을 받은 적이 있습니까?

문제 해결의 첫 번째 단계는 컴퓨터가 하트비트 이벤트를 보내는지 확인하는 것입니다.

```
Heartbeat 
    | where Computer like "<ComputerName>"
    | summarize arg_max(TimeGenerated, * ) by Computer
```

쿼리가 결과를 반환하는 경우 특정 데이터 형식이 수집되어 서비스로 전달되지 않는지 확인해야 합니다. 이는 에이전트가 서비스에서 업데이트된 구성을 받지 못하거나 에이전트가 정상적으로 작동하지 않는 다른 증상으로 인해 발생할 수 있습니다. 다음 단계를 수행하여 추가 문제를 해결합니다.

1. 컴퓨터에서 상승된 명령 프롬프트를 열고 `net stop healthservice && net start healthservice`을 입력하여 에이전트 서비스를 다시 시작합니다.
2. 운영 *관리자* 이벤트 로그를 열고 **이벤트 소스** *HealthService에서* **이벤트 아이디** *7023, 7024, 7025, 7028* 및 *1210을* 검색합니다.  이러한 이벤트는 에이전트가 Azure Monitor에서 구성을 성공적으로 수신하고 있으며 컴퓨터를 적극적으로 모니터링하고 있음을 나타냅니다. 이벤트 ID 1210에 대한 이벤트 설명은 에이전트의 모니터링 범위에 포함된 모든 솔루션 및 인사이트를 마지막 줄에 지정합니다.  

    ![이벤트 ID 1210 설명](./media/agent-windows-troubleshoot/event-id-1210-healthservice-01.png)

3. 몇 분 후 솔루션 또는 Insight에서 데이터를 보는지 여부에 따라 쿼리 결과 또는 시각화에 예상된 데이터가 표시되지 않는 경우 *Operations Manager* 이벤트 로그에서 **이벤트 소스** *HealthService* 및 *상태 서비스 모듈을* 검색하고 **이벤트 수준** *경고* 및 *오류별로* 필터링하여 다음 표에서 이벤트를 작성했는지 확인합니다.

    |이벤트 ID |원본 |설명 |해결 방법 |
    |---------|-------|------------|
    |8000 |HealthService |이 이벤트는 수집된 성능, 이벤트 또는 기타 데이터 형식과 관련된 워크플로가 작업 공간으로 수집하기 위해 서비스로 전달할 수 없는지 지정합니다. | Source HealthService의 이벤트 ID 2136은 이 이벤트와 함께 작성되며 프록시 및 인증 설정의 잘못된 구성, 네트워크 중단 또는 네트워크 방화벽/프록시로 인해 에이전트가 서비스와 통신할 수 없음을 나타낼 수 있습니다.| 
    |10102 및 10103 |상태 관리 서비스 모듈 |워크플로에서 데이터 원본을 확인할 수 없습니다. |지정된 성능 카운터 또는 인스턴스가 컴퓨터에 없거나 작업 영역 데이터 설정에 잘못 정의된 경우 이 작업이 발생할 수 있습니다. 사용자가 지정한 성능 [카운터인](data-sources-performance-counters.md#configuring-performance-counters)경우 지정된 정보가 올바른 형식을 따르고 대상 컴퓨터에 있는지 확인합니다. |
    |26002 |상태 관리 서비스 모듈 |워크플로에서 데이터 원본을 확인할 수 없습니다. |컴퓨터에 지정된 Windows 이벤트 로그가 없는 경우 이 이벤트가 발생할 수 있습니다. 이 오류는 컴퓨터가 이 이벤트 로그를 등록할 것으로 예상되지 않는 경우 무시될 수 있으며, 그렇지 않으면 사용자가 지정한 [이벤트 로그인](data-sources-windows-events.md#configuring-windows-event-logs)경우 지정된 정보가 올바른지 확인합니다. |
