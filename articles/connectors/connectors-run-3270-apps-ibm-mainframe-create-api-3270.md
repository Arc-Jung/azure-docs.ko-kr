# <a name="integrate-3270-screen-driven-apps-on-ibm-mainframes-with-azure-by-using-azure-logic-apps-and-ibm-3270-connector"></a>Azure 논리 앱 및 IBM 3270 커넥터를 사용하여 IBM 메인프레임에서 3270개의 화면 기반 앱을 Azure와 통합

> [!NOTE]
> 이 커넥터는 [*공개 미리 보기입니다.*](https://azure.microsoft.com/support/legal/preview-supplemental-terms/) 

Azure Logic Apps 및 IBM 3270 커넥터를 사용하면 3270 에뮬레이터 화면을 탐색하여 일반적으로 구동하는 IBM 메인프레임 앱에 액세스하고 실행할 수 있습니다. 이렇게 하면 Azure Logic Apps를 통해 자동화된 워크플로우를 만들어 IBM 메인프레임 앱을 Azure, Microsoft 및 기타 앱, 서비스 및 시스템과 통합할 수 있습니다. 커넥터는 TN3270 프로토콜을 사용하여 IBM 메인프레임과 통신하며 Azure 정부 및 Azure China 21Vianet을 제외한 모든 Azure 논리 앱 리전에서 사용할 수 있습니다. 로직 앱을 새로 접하는 경우 [Azure 논리 앱이란 무엇입니까?](../logic-apps/logic-apps-overview.md)

이 문서에서는 3270 커넥터를 사용하기 위한 이러한 측면에 대해 설명합니다. 

* Azure Logic Apps에서 IBM 3270 커넥터를 사용하는 이유와 커넥터가 3270 개의 화면 기반 앱을 실행하는 방법

* 3270 커넥터를 사용하기 위한 전제 조건 및 설정

* 논리 앱에 3270 커넥터 작업을 추가하는 단계

## <a name="why-use-this-connector"></a>이 커넥터를 사용하는 이유는 무엇입니까?

IBM 메인프레임의 앱에 액세스하려면 일반적으로 "녹색 화면"이라고 하는 3270 터미널 에뮬레이터를 사용합니다. 이 방법은 시간이 많이 테스트된 방법이지만 제한이 있습니다. HIS(호스트 통합 서버)는 이러한 앱으로 직접 작업하는 데 도움이 되지만 경우에 따라 화면과 비즈니스 논리를 분리하는 것이 불가능할 수 있습니다. 또는 호스트 응용 프로그램의 작동 방식에 대한 정보가 더 이상 없을 수도 있습니다.

이러한 시나리오를 확장하기 위해 Azure Logic Apps의 IBM 3270 커넥터는 특정 작업에 사용되는 호스트 화면인 "캡처"를 기록하고 메인프레임 앱을 통해 해당 작업에 대한 탐색 흐름을 정의하고 정의하는 데 사용하는 3270 디자인 도구와 함께 작동합니다. 해당 작업에 대한 입력 및 출력 매개 변수가 있는 메서드입니다. 디자인 도구는 논리 앱에서 해당 작업을 나타내는 작업을 호출할 때 해당 정보를 3270 커넥터에서 사용하는 메타데이터로 변환합니다.

디자인 도구에서 메타데이터 파일을 생성한 후 해당 파일을 Azure의 통합 계정에 추가합니다. 이렇게 하면 논리 앱에서 3270 커넥터 작업을 추가할 때 앱의 메타데이터에 액세스할 수 있습니다. 커넥터는 통합 계정에서 메타데이터 파일을 읽고, 3270 화면을 통해 탐색을 처리하고, 3270 커넥터 작업에 대한 매개 변수를 동적으로 제공합니다. 그런 다음 호스트 응용 프로그램에 데이터를 제공할 수 있으며 커넥터는 결과를 논리 앱에 반환합니다. 이렇게 하면 레거시 앱을 Azure, Microsoft 및 Azure Logic Apps가 지원하는 기타 앱, 서비스 및 시스템과 통합할 수 있습니다.

## <a name="prerequisites"></a>사전 요구 사항

* Azure 구독 Azure 구독이 없는 경우 [체험 Azure 계정에 등록](https://azure.microsoft.com/free/)합니다.

* [논리 앱을 만드는 방법에](../logic-apps/quickstart-create-first-logic-app-workflow.md) 대한 기본 지식

* 권장 사항: [통합 서비스 환경(ISE)](../logic-apps/connect-virtual-network-vnet-isolated-environment.md) 

  이 환경을 논리 앱을 만들고 실행하기 위한 위치로 선택할 수 있습니다. ISE는 논리 앱에서 Azure 가상 네트워크 내에서 보호되는 리소스에 대한 액세스를 제공합니다.

* 3270 화면 기반 앱을 자동화하고 실행하는 데 사용할 논리 앱

  IBM 3270 커넥터에는 트리거가 없으므로 다른 트리거를 사용하여 **되풀이** 트리거와 같은 논리 앱을 시작합니다. 그런 다음 3270 커넥터 작업을 추가할 수 있습니다. 시작하려면 [빈 논리 앱을 만듭니다.](../logic-apps/quickstart-create-first-logic-app-workflow.md) 
  ISE를 사용하는 경우 해당 ISE를 논리 앱의 위치로 선택합니다.

* [다운로드 및 3270 디자인 도구를 설치합니다.](https://aka.ms/3270-design-tool-download)
유일한 전제 조건은 [마이크로 소프트 .NET 프레임 워크 4.6.1입니다.](https://aka.ms/net-framework-download)

  이 도구를 사용하면 앱에서 추가하고 3270 커넥터 작업으로 실행하는 작업에 대한 화면, 탐색 경로, 메서드 및 매개 변수를 기록할 수 있습니다. 이 도구는 커넥터가 메인프레임 앱을 구동하는 데 필요한 메타데이터를 제공하는 HIDX(호스트 통합 디자이너 XML) 파일을 생성합니다.
  
  이 도구를 다운로드하고 설치한 후 호스트에 연결하려면 다음 단계를 따르십시오.

  1. 3270 설계 도구를 엽니다. **세션** 메뉴에서 **세션 호스트 세션을**선택합니다.
  
  1. TN3270 호스트 서버 정보를 제공합니다.

* 논리 앱이 해당 파일의 메타데이터 및 메서드 정의에 액세스할 수 있도록 HIDX 파일을 맵으로 저장하는 [통합 계정입니다.](../logic-apps/logic-apps-enterprise-integration-create-integration-account.md) 

  통합 계정이 사용 중이면 논리 앱에 연결되어 있는지 확인합니다. 또한 ISE를 사용하는 경우 통합 계정의 위치가 논리 앱에서 사용하는 ISE와 동일한지 확인합니다.

* 메인프레임 앱을 호스팅하는 TN3270 서버에 액세스

## <a name="create-metadata-overview"></a>메타데이터 개요 생성

3270 화면 기반 앱에서 화면 및 데이터 필드는 시나리오에 고유하므로 3270 커넥터에는 메타데이터로 제공할 수 있는 앱에 대한 이 정보가 필요합니다. 이 메타데이터는 논리 앱이 화면을 식별하고 인식하는 데 도움이 되는 정보를 설명하고, 화면 사이를 탐색하는 방법, 데이터를 입력할 위치 및 결과를 예상할 위치를 설명합니다. 이 메타데이터를 지정하고 생성하려면 자세한 내용은 다음과 같이 이러한 특정 *모드*또는 단계를 안내하는 3270 디자인 도구를 사용합니다.

* **캡처**: 이 모드에서는 메인프레임 앱(예: 은행 잔액)을 통해 특정 작업을 완료하는 데 필요한 화면을 기록합니다.

* **탐색**: 이 모드에서는 특정 작업에 대한 메인프레임 앱의 화면을 탐색하는 방법에 대한 계획 또는 경로를 지정합니다.

* **방법**: 이 모드에서는 화면 탐색 경로를 `GetBalance`설명하는 메서드(예: )를 정의합니다. 또한 메서드의 입력 및 출력 매개 변수가 되는 각 화면의 필드를 선택합니다.

### <a name="unsupported-elements"></a>지원되지 않는 요소

디자인 도구는 다음 요소를 지원하지 않습니다.

* 부분 IBM 기본 매핑 지원(BMS) 맵: BMS 맵을 가져오는 경우 디자인 도구는 부분 화면 정의를 무시합니다.
* 인/아웃 매개변수: 인/아웃 매개변수를 정의할 수 없습니다.
* 메뉴 처리: 미리 보기 중에 지원되지 않음
* 배열 처리: 미리 보기 중에 지원되지 않음

<a name="capture-screens"></a>

## <a name="capture-screens"></a>화면 캡처

이 모드에서는 각 3270 화면에 해당 화면을 고유하게 식별하는 항목을 표시합니다. 예를 들어 텍스트 줄 이나 특정 텍스트 및 비어 없는 필드와 같은 보다 복잡한 조건 집합을 지정할 수 있습니다. 호스트 서버에 대한 라이브 연결을 통해 이러한 화면을 기록하거나 IBM 기본 매핑 지원(BMS) 맵에서 이 정보를 가져올 수 있습니다. 라이브 연결은 호스트에 연결하기 위해 TN3270 에뮬레이터를 사용합니다. 각 커넥터 작업은 세션에 연결하는 것으로 시작하여 세션에서 연결을 끊는 것으로 끝나는 단일 작업에 매핑되어야 합니다.

1. 아직 설계 도구를 열지 않은 경우 3270 설계 도구를 엽니다. 도구 모음에서 캡처 모드를 입력하도록 **캡처를** 선택합니다.

1. 레코딩을 시작하려면 F5 키를 누르거나 **녹음/녹화** 메뉴에서 **녹음 시작을**선택합니다. 

1. **세션** 메뉴에서 **연결을**선택합니다.

1. **캡처** 창에서 앱의 첫 번째 화면에서 시작하여 녹화하는 특정 작업에 대해 앱을 단계별로 이동합니다.

1. 작업을 완료한 후 평소와 같이 앱에서 로그아웃합니다.

1. **세션** 메뉴에서 **연결을 선택합니다.**

1. 녹화를 중지하려면 Shift + F5 키를 누르거나 **녹음/녹화** 메뉴에서 **녹음 중지를**선택합니다.

   작업에 대한 화면을 캡처한 후 디자이너 도구에는 해당 화면을 나타내는 축소판 그림이 표시됩니다. 이러한 축소판에 대한 몇 가지 참고 사항:

   * 캡처한 화면에 "빈"이라는 화면이 표시됩니다.

     [CICS에](https://www.ibm.com/it-infrastructure/z/cics)처음 연결할 때 실행하려는 트랜잭션의 이름을 입력하려면 먼저 "지우기" 키를 보내야 합니다. "지우기" 키를 보내는 화면에는 화면 인식 편집기에서 추가할 수 있는 화면 제목과 같은 *인식 속성이*없습니다. 이 화면을 나타내기 위해 썸네일에는 "빈"이라는 화면이 포함되어 있습니다. 나중에 이 화면을 사용하여 트랜잭션 이름을 입력하는 화면을 표시할 수 있습니다.

   * 기본적으로 캡처된 화면의 이름은 화면의 첫 번째 단어를 사용합니다. 해당 이름이 이미 있는 경우 디자인 도구는 밑줄과 숫자(예: "WBGB" 및 "WBGB_1")로 이름을 추가합니다.

1. 캡처된 화면에 더 의미 있는 이름을 지정하려면 다음 단계를 따르세요.

   1. 호스트 **화면** 창에서 이름을 바꿀 화면을 선택합니다.

   1. 동일한 창에서 동일한 창의 아래쪽 근처에 화면 **이름** 속성을 찾습니다.

   1. 현재 화면 이름을 보다 설명적인 이름으로 변경합니다.

1. 이제 각 화면을 식별할 필드를 지정합니다.

   3270 데이터 스트림을 사용하면 화면에 기본 식별자가 없으므로 각 화면에서 고유한 텍스트를 선택해야 합니다. 복잡한 시나리오의 경우 고유한 텍스트와 특정 조건이 있는 필드와 같은 여러 조건을 지정할 수 있습니다.

인식 필드 선택을 마친 후 다음 모드로 이동합니다.

### <a name="conditions-for-identifying-repeated-screens"></a>반복 된 화면을 식별 하기 위한 조건

커넥터가 화면을 탐색하고 구분하려면 일반적으로 캡처된 화면 중에서 식별자로 사용할 수 있는 고유한 텍스트를 화면에 찾을 수 있습니다. 반복 된 화면의 경우 더 많은 식별 방법이 필요할 수 있습니다. 예를 들어 한 화면에서 유효한 값을 반환하는 반면 다른 화면에서는 오류 메시지를 반환하는 경우를 제외하고 는 두 개의 화면이 동일하게 보인다고 가정합니다.

디자인 도구에서 *화면 인식 편집기(화면 인식*편집기)를 사용하여 "계정 잔액 받기"와 같은 화면 제목과 같은 인식 특성을 추가할 수 있습니다. 분기된 경로가 있고 두 분기가 동일한 화면을 반환하지만 결과가 다른 경우 다른 인식 특성이 필요합니다. 런타임에 커넥터는 현재 분기 및 분기를 결정하기 위해 이러한 특성을 사용합니다. 사용할 수 있는 조건은 다음과 같습니다.

* 특정 값: 이 값은 지정된 위치에서 지정된 문자열과 일치합니다.
* 특정 값이 아님: 이 값은 지정된 위치에서 지정된 문자열과 일치하지 않습니다.
* 비어 있음: 이 필드는 비어 있습니다.
* 비어 있지 않음: 이 필드는 비어 있지 않습니다.

자세한 내용은 이 항목의 후반부 예제 [탐색 계획을](#example-plan) 참조하십시오.

<a name="define-navigation"></a>

## <a name="define-navigation-plans"></a>탐색 계획 정의

이 모드에서는 특정 작업에 대한 메인프레임 앱의 화면을 탐색하기 위한 흐름 또는 단계를 정의합니다. 예를 들어 앱이 한 경로에서 올바른 결과를 생성하는 위치에 앱이 취할 수 있는 경로가 두 개 이상 있고 다른 경로에서 오류가 발생할 수 있습니다. 각 화면에 대해 `CICSPROD <enter>`다음 화면으로 이동하는 데 필요한 키 입력을 지정합니다.

> [!TIP]
> 동일한 연결 및 연결 해제 화면을 사용하는 여러 작업을 자동화하는 경우 설계 도구는 특별한 연결 및 연결 해제 계획 유형을 제공합니다. 이러한 계획을 정의할 때 탐색 계획의 시작과 끝에 추가할 수 있습니다.

### <a name="guidelines-for-plan-definitions"></a>계획 정의에 대한 지침

* 연결부터 시작하여 연결을 끊는 것으로 끝나는 모든 화면을 포함합니다.

* 독립 실행형 계획을 만들거나 연결 및 연결 해제 계획을 사용하여 모든 트랜잭션에 공통된 일련의 화면을 다시 사용할 수 있습니다.

  * 연결 계획의 마지막 화면은 탐색 계획의 첫 번째 화면과 동일한 화면이어야 합니다.

  * 연결 해제 계획의 첫 번째 화면은 탐색 계획의 마지막 화면과 같아야 합니다.

* 캡처한 화면에는 반복되는 화면이 많이 포함될 수 있으므로 계획에서 반복되는 화면 중 하나의 인스턴스만 선택하고 사용합니다. 다음은 반복되는 화면의 몇 가지 예입니다.

  * 로그인 화면, 예를 들어, **MSG-10** 화면
  * CICS의 환영 화면
  * "지우기" 또는 **빈** 화면

   AID 키또는 AID 키와 고정 텍스트가 모두 있을 수 있습니다.

탐색 계획을 완료한 후 [다음 모드에서 메서드를 정의할](#define-method)수 있습니다.

## <a name="define-methods"></a>메서드 정의

이 모드에서는 탐색 계획과 연결된 메서드를 정의합니다. 각 메서드 매개 변수에 대해 문자열, 정수, 날짜 또는 시간 등과 같은 데이터 형식을 지정합니다. 작업이 완료되면 라이브 호스트에서 메서드를 테스트하고 메서드가 예상대로 작동하는지 확인할 수 있습니다. 그런 다음 메타데이터 파일 또는 HIDX(호스트 통합 디자이너 XML) 파일을 생성하여 IBM 3270 커넥터에 대한 작업을 만들고 실행하는 데 사용할 메서드 정의가 있습니다.

1. 3270 설계 도구의 도구 모음에서 메서드 모드를 입력하도록 **메서드를** 선택합니다. 

1. **탐색** 창에서 원하는 입력 필드가 있는 화면을 선택합니다.

1. 메서드의 첫 번째 입력 매개 변수를 추가하려면 다음 단계를 따르십시오.

   1. 3270 에뮬레이터 화면에서 **캡처** 창에서 필드 내부의 텍스트뿐만 아니라 첫 번째 입력으로 원하는 전체 필드를 선택합니다.

      > [!TIP]
      > 모든 필드를 표시하고 전체 필드를 선택했는지 확인하려면 **보기** 메뉴에서 **모든 필드를**선택합니다.

   1. 설계 도구의 도구 모음에서 **입력 필드를 선택합니다.** 

   입력 매개 변수를 더 추가하려면 각 매개 변수에 대해 이전 단계를 반복합니다.

1. 메서드의 첫 번째 출력 매개 변수를 추가하려면 다음 단계를 따르십시오.

   1. 3270 에뮬레이터 화면에서 **캡처** 창에서 필드 내부의 텍스트뿐만 아니라 첫 번째 출력으로 원하는 전체 필드를 선택합니다.

      > [!TIP]
      > 모든 필드를 표시하고 전체 필드를 선택했는지 확인하려면 **보기** 메뉴에서 **모든 필드를**선택합니다.

   1. 설계 도구의 도구 모음에서 **출력 필드를 선택합니다.**

   출력 매개 변수를 더 추가하려면 각 매개 변수에 대해 이전 단계를 반복합니다.

1. 메서드의 모든 매개 변수를 추가한 후 각 매개 변수에 대해 이러한 속성을 정의합니다.

   | 속성 이름 | 가능한 값 | 
   |---------------|-----------------|
   | **데이터 유형** | 바이트, 날짜 시간, 소수점, Int, 긴, 짧은, 문자열 |
   | **필드 채우기 기술** | 매개 변수는 필요한 경우 공백으로 채우는 이러한 채우기 유형을 지원합니다. <p><p>- **유형**: 필드에 순차적으로 문자를 입력합니다. <p>- **채우기**: 필드의 내용을 문자로 바꾸고 필요한 경우 공백으로 채웁니다. <p>- **EraseEofType**: 필드를 지웁니다, 그런 다음 필드에 순차적으로 문자를 입력합니다. |
   | **문자열 형식 지정** | 일부 매개 변수 데이터 형식은 형식 문자열을 사용하여 3270 커넥터에 화면의 텍스트를 .NET 데이터 형식으로 변환하는 방법을 알려줍니다. <p><p>- **DateTime**: DateTime 형식 문자열은 [.NET 사용자 지정 날짜 및 시간 형식 문자열을 따릅니다.](https://docs.microsoft.com/dotnet/standard/base-types/custom-date-and-time-format-strings) 예를 들어 날짜는 `06/30/2019` 형식 `MM/dd/yyyy`문자열을 사용합니다. <p>- **소수점**: 소수점 형식 문자열은 [COBOL 그림 절을](https://www.ibm.com/support/knowledgecenter/SS6SG3_5.2.0/com.ibm.cobol52.ent.doc/PGandLR/ref/rlddepic.html)사용합니다. 예를 들어 숫자는 `100.35` 형식 문자열을 `999V99`사용합니다. |
   |||

`..\Program Files\Microsoft Host Integration Server - 3270 Design Tool\SDK\WoodgroveBank.rap`

그러나 파일이 디자인 도구의 설치 폴더에 있는 동안 샘플 RAP 파일에 대한 변경 내용을 저장하거나 샘플 RAP 파일에서 HIDX 파일을 생성하려고 하면 "액세스 거부" 오류가 발생할 수 있습니다. 기본적으로 디자인 도구는 높은 권한 없이 프로그램 파일 폴더에 설치됩니다. 오류가 발생하면 다음 해결 방법 중 하나를 시도해 보십시오.

* 샘플 파일을 다른 위치로 복사합니다.
* 관리자로 디자인 도구를 실행합니다.

## <a name="generate-and-upload-hidx-file"></a>HIDX 파일 생성 및 업로드

준비가 되면 통합 계정에 업로드할 수 있도록 HIDX 파일을 생성합니다. 3270 디자인 도구는 RAP 파일을 저장한 새 하위 폴더에 HIDX 파일을 만듭니다.

1. 3270 설계 도구의 도구 모음에서 **코드 생성을**선택합니다.

1. RAP 파일이 포함된 폴더로 이동하여 HIDX 파일을 생성한 후 도구가 만든 하위 폴더를 엽니다. 도구가 HIDX 파일을 만들었으며 확인합니다.

1. [Azure 포털에](https://portal.azure.com)로그인하여 통합 계정을 찾습니다.

1. HIDX 파일을 맵을 [추가하기 위한 유사한 단계를 수행하여](../logic-apps/logic-apps-enterprise-integration-liquid-transform.md)통합 계정에 HIDX 파일을 추가하지만 맵 유형을 선택할 때 **HIDX를**선택합니다.

이 항목의 후반부에서는 논리 앱에 IBM 3270 작업을 처음으로 추가할 때 통합 계정 및 호스트 서버의 이름과 같은 연결 정보를 제공하여 논리 앱과 호스트 서버 간에 연결을 만들라는 메시지가 표시됩니다. . 연결을 만든 후 이전에 추가한 HIDX 파일, 실행할 메서드 및 사용할 매개 변수를 선택할 수 있습니다.

이러한 모든 단계를 완료하면 논리 앱에서 만든 작업을 사용하여 IBM 메인프레임에 연결하고, 앱의 화면을 구동하고, 데이터를 입력하고, 결과를 반환하는 등의 작업을 수행할 수 있습니다. 다른 앱, 서비스 및 시스템과 통합하기 위해 논리 앱에 다른 작업을 계속 추가할 수도 있습니다.
