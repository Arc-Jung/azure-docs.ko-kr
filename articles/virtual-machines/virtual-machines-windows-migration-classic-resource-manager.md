# 클래식에서 Azure Resource Manager로 IaaS 리소스의 플랫폼 지원 마이그레이션

Azure Resource Manager에서 가상 컴퓨터에 대한 지원을 발표한 이후 거의 1년이 지났습니다. 여기에서는 Azure Resource Manager의 개선 사항 및 추가 기능에 대해 자세히 설명합니다. 또한 가상 네트워크 사이트-사이트 게이트웨이를 사용하여 구독에서 두 가지 배포 모델의 리소스를 연결하고 사용하는 최적의 방법에 대해서도 설명합니다. 이 문서에서는 Microsoft가 어떻게 클래식에서 Resource Manager로 IaaS 리소스를 마이그레이션할 수 있도록 지원하는지 살펴봅니다.

## 마이그레이션의 목표

우리는 가상 컴퓨터의 새로운 환경과 API가 제공하는 성능과 역량에 매우 놀라워하고 있습니다. 이를 통해 클라우드를 더욱 다양한 방식으로 사용할 수 있을 것으로 기대되며 새로운 모델이 릴리스됨에 따라 리소스 그룹에 관련 서비스를 배포, 관리, 모니터링할 수 있습니다. Resource Manager는 템플릿을 사용하여 복잡한 응용 프로그램을 배포할 수 있도록 지원하며 VM 확장을 사용하여 가상 컴퓨터를 구성하고 액세스 관리와 태깅을 통합합니다. 또한 가용성 집합에 가상 컴퓨터에 대해 확장성 있는 병렬 배포를 포함합니다. 뿐만 아니라 새로운 모델에서는 계산, 네트워크, 저장소의 수명 주기를 독립적으로 관리할 수 있으며 가상 네트워크에 가상 컴퓨터를 적용하여 보안 구현을 기본적으로 중요시합니다.

기능 측면에서는 Azure Resource Manager의 계산, 네트워크 저장소에 대해 거의 모든 기능이 지원되며 몇 가지 예외 사항은 향후 몇 개월 이내에 완료하기 위해 최선을 다하고 있습니다. 또한 사용자 환경을 지속적으로 개선하고 환경을 매끄럽게 연결하기 위해 Azure 포털에 기능을 추가하고 있습니다.

새로운 Azure Resource Manager에서 이와 같은 새로운 기능을 제공하고 배포 기반이 확대됨에 따라 고객이 기존 클래식 배포를 마이그레이션할 수 있도록 지원하고자 합니다.

>[AZURE.NOTE] Microsoft는 이번 발표와 함께 마이그레이션 서비스의 공개 미리 보기를 시작합니다. 공개 미리 보기 중에는 Azure 구독에서 개발/테스트 워크로드만 마이그레이션하는 것이 좋습니다.

## 마이그레이션 후 자동화 및 도구 변경

클래식에서 Resource Manager 모델로 리소스를 마이그레이션할 때 변경해야 할 중요 사항 중 하나는 리소스를 마이그레이션한 후에도 기존 자동화 또는 도구를 업데이트하여 계속 작동하도록 하는 것입니다.

## 클래식에서 Resource Manager로 IaaS 리소스를 마이그레이션하는 것은 어떤 의미가 있습니까?

자세한 내용을 살펴보기 전에 IaaS 리소스에서 데이터 평면과 관리 평면 작업 간 차이를 간략히 설명하겠습니다. 이러한 차이에 따라 마이그레이션 지원 계획을 설명할 수 있으므로 이 차이를 이해하는 것이 중요합니다.

- 관리 평면 - 리소스 수정을 위해 관리 평면 또는 API로 들어오는 호출을 설명합니다. 예를 들어 VM 만들기, VM 다시 시작, 새로운 서브넷으로 가상 네트워크를 업데이트하는 작업 등이 있습니다. 이러한 작업은 모두 실행 중인 리소스를 관리하지만 인스턴스에 대한 연결에 직접적 영향을 미치지 않습니다.
- 데이터 평면(응용 프로그램) - 응용 프로그램 자체의 "런타임"을 설명하며 Azure API를 통하지 않는 인스턴스와의 상호 작용이 포함됩니다. 웹 사이트에 액세스하거나 실행 중인 SQL Server 또는 mongoDB 서버에서 데이터를 가져오는 경우 모두 데이터 평면 또는 응용 프로그램 상호 작용으로 간주됩니다. 저장소 계정에서 Blob을 복사하고 공용 IP 주소에 액세스하여 가상 컴퓨터에 RDP 또는 SSH 연결을 하는 작업도 데이터 평면에 해당합니다. 이러한 작업은 계산, 네트워킹, 저장소에서 응용 프로그램을 계속 실행하는 상태에서 수행합니다.

>[AZURE.NOTE] 일부 마이그레이션 시나리오(자세한 내용은 지원되지 않는 구성 참조)에서는 가상 컴퓨터를 중지, 할당 취소, 다시 시작하며, 이때 데이터 평면에 잠시 가동 중지 시간이 발생합니다.

## 마이그레이션의 지원 범위

공개 미리 보기 중에는 기본적으로 계산과 네트워크를 대상으로 두 가지 마이그레이션 범위를 제공합니다. 저장소 계정 마이그레이션 지원이 계획되어 있으며 곧 릴리스될 예정입니다. 하지만 원활한 마이그레이션을 위해 클래식 저장소 계정에 Resource Manager VM용 디스크를 포함할 수 있도록 했습니다. 자세한 내용은 아래를 참조하세요.

### 가상 컴퓨터 마이그레이션(가상 네트워크가 아님)

Resource Manager 배포 모델에서는 기본적으로 응용 프로그램 보안이 적용되어 있습니다. 따라서 모든 VM은 Resource manager 모델의 가상 네트워크에 있어야 합니다. 따라서 마이그레이션 중에 VM을 다시 시작합니다(중지, 할당 취소, 시작). 가상 네트워크와 관련하여 몇 가지 항목을 선택할 수 있습니다.
- 플랫폼에서 새 가상 네트워크를 만들도록 요청하고 가상 컴퓨터를 새 가상 네트워크로 마이그레이션하거나
- 가상 컴퓨터를 Resource Manager의 기존 가상 네트워크로 마이그레이션할 수 있습니다.

>[AZURE.NOTE] 이 마이그레이션 범위에서는 마이그레이션 중 특정 시간 동안 '관리 평면' 및 '데이터 평면' 작업이 허용되지 않을 수 있습니다.

### 가상 컴퓨터 마이그레이션(가상 네트워크에서)

이 범위에서는 대부분의 VM 구성에 대해 클래식과 Resource Manager 간 메타데이터만 마이그레이션합니다. 기본 가상 컴퓨터는 동일한 네트워크의 동일한 하드웨어에서 동일한 저장소로 실행됩니다. 따라서 클래식에서 Resource Manager로 메타데이터를 마이그레이션할 경우에는 마이그레이션 중 특정 시간 동안 '관리 평면' 작업이 허용되지 않을 수 있습니다. 하지만 데이터 평면은 계속 작동합니다. 즉, 마이그레이션 중 가상 컴퓨터(클래식) 상에서 실행되는 응용 프로그램에 가동 중지 시간이 발생하지 않습니다.

현재는 다음 구성이 지원되지 않습니다. 하지만 향후 이에 대한 지원을 추가할 경우 이 구성의 일부 VM에서 가동 중지 시간(중지, 할당 취소, 다시 시작)이 발생할 수 있습니다.

-	단일 클라우드 서비스에 둘 이상의 가용성 집합이 있는 경우
-	단일 클라우드 서비스에서 '하나 이상의 가용성 집합'과 '가용성 집합에 없는 VM'이 있는 경우

>[AZURE.NOTE] 이 마이그레이션 범위에서는 마이그레이션 중 특정 시간 동안 '관리 평면'이 허용되지 않을 수 있습니다. 위에서 설명한 특정 구성의 경우 '데이터 평면' 다운타임이 발생합니다.

### 저장소 계정 및 마이그레이션

저장소 계정 마이그레이션은 현재 공개 미리 보기에 지원되지 않습니다. 하지만 저장소 계정 마이그레이션에 대한 지원이 계획되어 있으며 곧 릴리스될 예정입니다. 마이그레이션 및 저장소 계정의 동작과 관련하여 두 가지 중요한 측면이 있습니다.

- 클래식 저장소 계정에 배포할 Resource Manager VM 사용 - Microsoft는 원활한 마이그레이션을 위해 Resource Manager VM을 클래식 저장소 계정에 배포하는 기능을 지원합니다. 이 기능을 사용하면 계산 및 네트워크 리소스를 저장소 계정과 상관없이 마이그레이션할 수 있습니다.
- 저장소 계정 마이그레이션 모범 사례 - 저장소 계정 마이그레이션이 지원될 경우 플랫폼은 원활한 환경을 구현하기 위해 계산 및 네트워크 리소스를 저장소 계정과 별도로 적용합니다.

## 지원되지 않는 기능 및 구성

현재는 특정 기능 및 구성 집합을 지원하지 않습니다. 이에 대한 지원을 추가하기 위해 노력하고 있지만 다음 섹션에서 권장 사항과 관련 계획을 확인할 수 있습니다.

### 지원되지 않는 기능

다음 기능 목록은 공개 미리 보기에 지원되지 않습니다. 이러한 설정을 제거하고 VM을 마이그레이션한 다음 Resource Manager 배포 모델에서 다시 활성화할 수 있습니다(선택 사항). 아래 기능은 대부분 지원될 예정이며 지원되는 즉시 릴리스됩니다.

리소스 공급자 | 기능
---------- | ------------
계산 | 부트 진단
계산 | 연결되지 않은 가상 컴퓨터 디스크
계산 | 가상 컴퓨터 이미지
네트워크 | 연결되지 않고 예약된 IP[가상 컴퓨터에 연결되지 않은 경우] VM에 연결되어 있고 예약된 IP는 지원됩니다.
네트워크 | 연결되지 않은 네트워크 보안 그룹[가상 네트워크 또는 네트워크 인터페이스에 연결되지 않은 경우]. VNET에서 참조하는 NSG는 지원됩니다.
네트워크 | 끝점 ACL
네트워크 | 가상 네트워크 게이트웨이(사이트 간, Express 경로, 지점 및 사이트 간)
저장소 | 저장소 계정

### 지원되지 않는 구성

다음 구성 목록은 공개 미리 보기에 지원되지 않습니다. 아래에서 권장 사항을 확인할 수 있습니다. 아래 구성 중 일부는 지원될 예정이며 지원되는 즉시 릴리스됩니다.

부여 | 구성 | 권장 사항
---------- | ------------ | ------------
리소스 관리자 | 클래식 리소스에 대한 역할 기반 액세스 제어 | 리소스의 URI는 마이그레이션 후 수정되므로 마이그레이션 후 필요한 RBAC 정책 업데이트를 미리 계획하는 것이 좋습니다.
계산 | VM과 연결된 여러 서브넷 | 서브넷만 참조하도록 서브넷 구성을 업데이트해야 합니다.
계산 | 가상 네트워크에 속하지만 명시적 서브넷이 할당되지 않은 가상 컴퓨터 | VM을 삭제할 수 있습니다(선택 사항). 이 기능은 현재 지원이 계획되어 있습니다.
계산 | 경고, 자동 크기 조정 정책이 있는 가상 컴퓨터 | 현재는 마이그레이션이 통과하며 이러한 설정이 삭제됩니다. 따라서 마이그레이션 전에 환경을 평가하는 것이 좋습니다. 또는 마이그레이션이 완료된 다음 경고 설정을 다시 구성할 수 있습니다.
계산 | XML VM 확장[VS 디버거, WebDeploy, RemoteDebug] | 이 기능은 지원되지 않을 예정입니다. 가상 컴퓨터에서 이러한 확장을 제거하고 마이그레이션을 계속하는 것이 좋습니다.
계산 | 웹/작업자 역할이 포함된 클라우드 서비스 | 이 기능은 현재 지원되지 않으며 계획 중입니다.
네트워크 | 가상 컴퓨터와 웹/작업자 역할이 포함된 가상 네트워크 | 이 기능은 현재 지원되지 않으며 계획 중입니다.
네트워크 | 이름에 공백이 포함된 서브넷 | 이 기능은 지원이 계획되어 있습니다.
앱 서비스 | 앱 서비스 환경이 포함된 가상 네트워크 | 이 기능은 현재 지원되지 않으며 계획 중입니다.
HDInsight Services | HDInsight Services가 포함된 가상 네트워크 | 이 기능은 현재 지원되지 않으며 계획 중입니다.
Dynamics Lifecycle Services | Dynamics Lifecycle Services에서 관리하는 가상 컴퓨터가 포함된 가상 네트워크 | 이 기능은 현재 지원되지 않으며 계획 중입니다.

## 마이그레이션 환경

마이그레이션 환경을 시작하기 전에 몇 가지 단계를 따르는 것이 좋습니다.

1. 마이그레이션을 계획 중인 리소스가 지원되지 않는 기능 또는 구성을 활용하지 않도록 해야 합니다. 대부분의 경우에는 플랫폼이 이러한 문제를 감지하고 오류를 throw합니다.
2. 가상 네트워크에 없는 VM이 있는 경우에는 준비 작업 중에 중지 및 할당 취소됩니다. 공용 IP 주소를 유지하려는 경우에는 준비를 트리거하기 전에 IP 주소 예약을 참조하십시오. 하지만 VM이 가상 네트워크에 있는 경우에는 중지 및 할당 취소되지 않습니다.
3. 프로덕션 배포를 마이그레이션하기 전에 유사한 설정의 테스트 환경을 설정하고 해당 리소스를 마이그레이션하는 것이 좋습니다.
4. 업무 외 시간 동안 마이그레이션을 계획하여 마이그레이션 중 발생할 수 있는 예기치 않은 오류를 해결할 수 있도록 계획하십시오.
5. 준비 단계가 완료된 후 쉽게 검증할 수 있도록 PowerShell, CLI 명령 도는 REST API를 사용하여 가상 컴퓨터의 현재 구성을 다운로드합니다.
6. 마이그레이션을 시작하기 전에 Resource Manager 배포를 처리할 수 있도록 자동화/운영 스크립트를 업데이트합니다. 또한 리소스가 준비됨 상태일 때 GET 작업을 실행할 수 있습니다(선택 사항).
7. 클래식 IaaS 리소스에 구성된 RBAC 정책을 평가하고 마이그레이션이 완료되면 계획을 세웁니다.

Microsoft는 공개 미리 보기를 발표하면서 REST API, PowerShell, Azure CLI를 통한 마이그레이션 트리거 지원을 추가했습니다. 사용자가 클래식에서 ARM 마이그레이션으로의 마이그레이션을 시각화 및 안내하기 위한 Azure 포털의 마이그레이션 지원이 계획되어 있습니다.

![마이그레이션 워크플로를 보여주는 스크린 샷](./media/virtual-machines-windows-migration-classic-resource-manager/migration-workflow.png)

1.	준비
	* 준비는 마이그레이션 프로세스의 첫 번째 단계입니다. 이 단계의 목표는 클래식에서 Resource Manager 리소스로의 IaaS 리소스 전환을 시뮬레이션하고 사용자가 볼 수 있도록 나란히 표시하는 것입니다. 작업의 자세한 흐름은 아래에 설명되어 있습니다.
  * 마이그레이션을 준비하려는 가상 네트워크 또는 호스티드 서비스(VNET이 아닌 경우)를 선택합니다.
  *	가장 먼저 플랫폼이 언제나 마이그레이션 대상 리소스에 대해 백그라운드에서 데이터 분석을 수행하고 리소스 마이그레이션이 가능할 경우 성공/실패를 반환합니다.
	*	리소스를 마이그레이션할 수 없을 경우 마이그레이션이 지원되지 않는 이유를 나열합니다.
	* 리소스 마이그레이션이 가능할 경우에는 플랫폼이 가장 먼저 마이그레이션 리소스에 대한 관리 평면 작업을 잠급니다. 예: 마이그레이션 중인 VM에 데이터 디스크를 추가할 수 없습니다.
  *	그런 다음 플랫폼이 마이그레션하는 리소스에 대해 클래식에서 리소스 관리자로 메타데이터 마이그레이션을 시작합니다.  
  *	준비 작업이 완료되면 리소스를 클래식 및 Resource Manager에서 볼 수 있는 옵션이 표시됩니다. 클래식 배포 모델의 모든 클라우드 서비스에 대해 `cloud-service-name>-migrated` 패턴의 리소스 그룹 이름을 만듭니다.

2.	수동 또는 스크립트 검사
  * 이 단계에서는 앞에서 다운로드한 구성을 사용하여 마이그레이션이 정상적으로 진행되는지 확인할 수 있습니다(선택 사항). 또는 포털에 로그인하고 속성과 리소스에 대해 주요 점검을 수행하여 메타데이터 마이그레이션이 정상적으로 진행되는지 확인합니다.
	* 가상 네트워크를 마이그레이션하는 경우 가상 컴퓨터의 대부분의 구성이 다시 시작됩니다. 이러한 VM의 응용 프로그램에 대해 응용 프로그램이 아직 실행 중인지 확인할 수 있습니다.
	* 모니터링/자동화와 작업 스크립트를 테스트하여 VM이 정상적으로 작동하며 업데이트된 스크립트가 올바르게 작동하는지 확인합니다. 리소스가 준비됨 상태인 경우에는 GET 작업만 지원됩니다.
  * 마이그레이션을 언제까지 '커밋'해야 하는가에 대해 정해진 시간은 없습니다. 이 상태를 원하는 시간 동안 유지할 수 있습니다. 하지만 '중단' 또는 '커밋'할 때까지 이러한 리소스에 대한 관리 평면이 잠깁니다.
  * 문제가 발생할 경우 언제든지 마이그레이션을 '중단'하고 '클래식' 배포 모델로 돌아갈 수 있습니다. 그럴 경우 클래식 배포 모델에서 해당 VM에서 정상 작업을 재개할 수 있도록 리소스에 대한 관리 평면 작업이 시작됩니다.

3. 중단
  * 이 단계에서는 변경 사항을 클래식 배포 모델로 되돌리고 마이그레이션을 중단할 수 있습니다(선택 사항). '커밋' 작업을 트리거하면 이 작업을 실행할 수 없습니다. 	

4.	커밋
  * 유효성 검사를 마치면 마이그레이션을 '커밋'할 수 있으며 클래식에 리소스가 더 이상 표시되지 않지만 Resource Manager 배포 모델에서만 사용할 수 있습니다. 즉, 새 포털에서는 마이그레이션된 리소스만 관리할 수 있습니다.
	* 이 작업이 실패할 경우 이 작업을 몇 번 더 시도하는 것이 좋습니다. 계속 실패할 경우 지원 티켓을 만들거나 [여기](https://social.msdn.microsoft.com/Forums/azure/ko-KR/home?forum=WAVirtualMachinesforWindows)에서 ClassicIaaSMigration 태그를 사용하여 포럼 게시물을 작성할 수 있습니다.

>[AZURE.NOTE] 아래에 설명된 모든 작업은 idempotent입니다. 지원되지 않는 기능 또는 구성 오류 이외의 항목을 실행할 경우 준비, 중단 또는 커밋 작업을 다시 시도하는 것이 좋습니다. 그러면 플랫폼에서 작업을 다시 시도합니다.

## 질문과 대답

**이 마이그레이션 계획이 Azure 가상 컴퓨터에서 실행되는 기존 서비스 또는 응용 프로그램에 영향을 미치나요?**

아니요. 가상 컴퓨터(클래식)는 완전히 지원되는 GA 서비스이며 계속 이러한 리소스를 활용하여 Microsoft Azure Cloud의 공간을 확장할 수 있습니다.

**가까운 미래에 마이그레이션할 계획이 없는 경우 내 VM은 어떻게 됩니까?**

Microsoft는 기존 클래식 API와 리소스 모델을 중단할 계획이 없으며 Resource Manager deployment 모델에 지원되는 고급 기능을 사용하여 마이그레이션을 쉽게 수행할 수 있도록 지원하고자 합니다. [여기](virtual-machines-windows-compare-deployment-models.md)에서 Resource Manager의 IaaS로 구현된 몇 가지 개선 사항을 살펴볼 것을 권장합니다.

**이 마이그레이션 계획으로 기존 도구는 어떻게 되나요?**

마이그레이션 계획에서 가장 중요하게 수행해야 할 변경 사항 중 하나는 사용자 도구를 Resource Manager 배포 모델에 맞게 업데이트하는 것입니다.

**관리 평면 가동 중지 시간은 얼마나 되나요?**

마이그레이션하는 리소스의 수에 따라 달라집니다. 소규모 배포(몇 십 대의 VM)의 경우 종단 간 전체 마이그레이션은 1시간 미만이 소요됩니다. 하지만 대규모 배포는(수백 대의 VM) 몇 시간이 소요될 수 있습니다. 서비스가 공개 미리 보기에 제공되므로 개발 또는 테스트 구독에서 실행하여 영향을 평가해 보는 것이 좋습니다.

**Resource Manager에서 마이그레이션 리소스를 커밋한 다음 롤백할 수 있나요?**

리소스가 '준비됨' 상태인 경우에만 마이그레이션을 중단할 수 있습니다. 커밋 작업을 사용하여 리소스를 성공적으로 마이그레이션한 후에는 롤백이 지원되지 않습니다.

**커밋 작업이 실패한 경우 마이그레이션을 롤백할 수 있나요?**

커밋 작업이 실패한 경우 마이그레이션을 중단할 수 없습니다. 커밋 작업을 포함한 모든 마이그레이션 작업은 idempotent입니다. 따라서 짧은 기간 이후 작업을 다시 시도해보는 것이 좋습니다. 그래도 오류가 발생할 경우 지원 티켓을 만들거나 [여기](https://social.msdn.microsoft.com/Forums/azure/ko-KR/home?forum=WAVirtualMachinesforWindows)에서 ClassicIaaSMigration 태그로 포럼 게시물을 작성할 수 있습니다.

**Resource Manager에서 IaaS를 활용해야 할 경우 다른 Express 경로 회로를 구매해야 하나요?**

아니요. Microsoft는 최근 [클래식 및 Resource Manager에서 Express 경로 회로의 공존](../expressroute/expressroute-howto-coexist-resource-manager.md)을 구현했습니다. 이미 Express 경로 회로가 있는 경우 새로 구입하지 않아도 됩니다.

**지원되지 않는 시나리오를 언제 마이그레이션 목록에 추가할지에 대한 로드맵이 있나요?**

현재는 2016년 전반기에 첫 번째 시나리오 집합을 출시합니다. 첫 번째 릴리스 이후 지원되지 않는 기능 지원을 계속 추가할 예정입니다.

**나의 클래식 IaaS 리소스에 대해 역할 기반 액세스 제어 정책을 구성한 경우 어떻게 되나요?**

마이그레이션 중에는 리소스가 클래식에서 Resource Manager로 전환됩니다. 따라서 마이그레이션 후 필요한 RBAC 정책 업데이트를 미리 계획하는 것이 좋습니다.

**현재 Azure에서 Azure Site Recovery 또는 백업 서비스를 사용하면 어떻게 되나요?**

최근 Resource Manager에서 가상 컴퓨터의 Azure Site Recovery 및 백업을 수행하는 기능이 추가되었습니다. Microsoft는 Resource Manager로 VM을 마이그레이션하는 기능을 지원하기 위해 관련 팀과 적극적으로 협력하고 있습니다. 현재 해당 기능을 사용하고 있는 경우에는 마이그레이션을 수행하지 않는 것이 좋습니다.

**내 구독 또는 리소스에서 마이그레이션이 가능한지 확인할 수 있나요?**

현재 준비 작업은 마이그레이션을 준비 중인 리소스에 대한 유효성 검사를 의미하지 않습니다. 플랫폼 지원 마이그레이션 옵션에서 마이그레이션을 준비하는 첫 번째 단계는 리소스를 마이그레이션할 수 있는지 여부를 확인하는 것입니다. 유효성 검사가 실패할 경우 리소스에 대해 아무 작업도 실행되지 않습니다. 하지만 이 작업을 훨씬 간단히 수행할 수 있는 새로운 유효성 검사 작업을 릴리스할 계획입니다.

**마이그레이션을 위해 IaaS 리소스를 준비하는 동안 할당량 오류가 발생하면 어떻게 되나요?**

마이그레이션을 중단하는 것이 좋습니다. 그런 다음 VM을 마이그레이션하고 있는 지역에서 할당량 증가에 대한 지원 요청을 로깅합니다. 할당량 요청이 승인되면 마이그레이션 단계를 다시 실행합니다.

**지원 계약이 없는 경우 어떻게 문제를 보고하나요?**

[여기](https://social.msdn.microsoft.com/Forums/azure/ko-KR/home?forum=WAVirtualMachinesforWindows)에서 키워드 ClassicIaaSMigration을 사용하여 가상 컴퓨터 포럼에 마이그레이션에 대한 문제와 질문을 게시하십시오. 이 포럼에 모든 질문을 게시할 것을 권장하지만 지원 계약이 있는 경우에는 지원 티켓도 로깅할 수 있습니다.

**마이그레이션 중 플랫폼이 내 리소스 그룹에 대해 선택한 이름이 마음에 들지 않으면 어떻게 하나요?**

리소스 그룹 간 리소스 이동에 대한 지원을 곧 발표할 예정입니다. 이 기능이 지원되면 리소스를 사용자가 선택한 리소스 그룹으로 이동할 수 있습니다.

**마이그레이션 중 플랫폼이 선택한 리소스 이름이 마음에 들지 않으면 어떻게 하나요?**

클래식 배포 모델에서 명시적으로 이름을 입력한 모든 리소스는 마이그레이션 중 해당 이름이 보존됩니다. 새 리소스가 생성되는 경우도 있습니다. 예를 들어 모든 VM에 대해 네트워크 인터페이스가 생성됩니다. 현재는 마이그레이션 중 이와 같이 새로 생성된 리소스 이름을 제어하는 기능을 지원하지 않습니다. [여기](http://feedback.azure.com)에서 이 기능에 대해 투표해 주십시오.
