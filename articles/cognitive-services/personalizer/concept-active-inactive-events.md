# <a name="active-and-inactive-events"></a>활성 및 비활성 이벤트

**활성** 이벤트는 고객에게 결과를 표시하고 보상 점수를 결정할 것으로 알고 있는 랭크로 의한 모든 호출입니다. 기본 동작입니다.

**비활성** 이벤트는 비즈니스 논리로 인해 사용자가 권장 작업을 볼 수 있는지 확실하지 않은 순위에 대한 호출입니다. 이렇게 하면 이벤트를 폐기할 수 있으므로 개인 설정자가 기본 보상으로 학습되지 않습니다. 비활성 이벤트는 리워드 API를 호출해서는 안 됩니다.

학습 루프가 이벤트의 실제 유형을 알고 있는지 가 중요합니다. 비활성 이벤트에는 리워드 콜이 없습니다. 활성 이벤트에는 리워드 콜이 있어야 하지만 API 호출이 이루어지지 않으면 기본 보상 점수가 적용됩니다. 이벤트 상태가 사용자 경험에 영향을 줄 것을 알고 있는 즉시 비활성에서 활성으로 변경합니다.

## <a name="typical-active-events-scenario"></a>일반적인 활성 이벤트 시나리오

응용 프로그램이 Rank API를 호출하면 응용 프로그램이 **rewardActionId** 필드에 표시해야 하는 작업을 받게 됩니다.  그 순간부터, 개인화는 동일한 이벤트Id를 가진 보상 점수와 보상 호출을 기대합니다. 보상 점수는 향후 랭크 호출에 대한 모델을 교육하는 데 사용됩니다. eventId에 대해 보상 콜을 받지 못하면 기본 보상이 적용됩니다. [기본 보상은](how-to-settings.md#configure-rewards-for-the-feedback-loop) Azure 포털의 개인 설정 리소스에 설정됩니다.

## <a name="other-event-type-scenarios"></a>기타 이벤트 유형 시나리오

일부 시나리오에서는 결과가 사용자에게 사용되거나 표시될지 알기 전에 응용 프로그램이 Rank를 호출해야 할 수 있습니다. 예를 들어 프로모션 콘텐츠의 페이지 렌더링이 마케팅 캠페인에 의해 덮어쓰이는 경우 이러한 상황이 발생할 수 있습니다. 랭크 호출의 결과가 사용되지 않았고 사용자가 이를 본 적이 없는 경우 해당 보상 호출을 보내지 마십시오.

일반적으로 이러한 시나리오는 다음과 같은 경우에 발생합니다.

* 사용자가 볼 수도 또는 표시되지 않을 수도 있는 UI를 미리 렌더링하고 있습니다.
* 응용 프로그램은 실시간 컨텍스트가 거의 없는 랭크 호출이 이루어지고 응용 프로그램이 출력을 사용하거나 사용하지 않을 수 있는 예측 개인 설정을 수행하고 있습니다.

이러한 경우 개인 설정자를 사용하여 순위를 호출하고 이벤트를 _비활성_으로 요청합니다. 개인 설정은 이 이벤트에 대한 보상을 기대하지 않으며 기본 보상은 적용되지 않습니다.

비즈니스 논리의 나중에 응용 프로그램이 Rank 호출의 정보를 사용하는 경우 이벤트를 _활성화하기만_ 하면 됩니다. 이벤트가 활성화되는 즉시, 개인화는 이벤트 보상을 기대합니다. 리워드 API에 대한 명시적 호출이 이루어지지 않으면 개인 설정자가 기본 보상을 적용합니다.

## <a name="inactive-events"></a>비활성 이벤트

이벤트에 대한 교육을 사용하지 않으려면 `learningEnabled = False`순위를 사용하여 순위를 호출합니다.

비활성 이벤트의 경우 eventId에 대한 보상을 보내거나 해당 eventId에 `activate` 대한 API를 호출하면 학습이 암시적으로 활성화됩니다.
